### 0.3 当前真实开发状态判断

从当前 OpenBot Android 代码看，上位机开发已经不再是“从零开始”，而是进入了“已有原型链路，需要补齐闭环和验证路径”的阶段：

- `HumanCartSimulatorFragment` 已经接入检测、初始化确认 UI、目标匹配、状态显示与控制提示；
- `TargetMemory`、`TargetMatcher`、`FollowStateMachine`、`HumanCommandInterpreter` 等模块已经落地，不再只是文档设想；
- 当前 `TargetMatcher` 的真实实现仍以 `position + size + color + confidence` 为主，尚未接入真正的 ReID embedding；
- `ReIDFeatureExtractor` 目前是接口占位，说明 ReID 已纳入架构，但 Android 端部署还处于下一阶段；
- `vehicle.setControl()` 实车控制链路、`LOST → SEARCH → STOP` 的完整执行逻辑、参数持久化与更多实测调参仍未完成。

因此，本阶段最合理的口径不是“已经完成 ReID 跟随”，而是：

> 已完成面向购物车跟随的 Android 端结构改造和 Human Cart Simulator 原型，正在从“状态机与调试 UI 跑通”推进到“控制闭环 + 真实 ReID 增强 + 数据驱动调参”。

### 0.4 本次修订后的核心判断

原计划已经完成“确认式目标初始化 + 目标记忆 + 人肉底盘实时仿真”的主线。本次修订进一步将上位机问题拆成四个子问题：

```text
1. 跟谁：目标身份保持 / ReID / 遮挡重识别
2. 离多远：跟随距离估计 / 距离控制
3. 能不能走：障碍风险判断 / 安全停车
4. 不确定怎么办：保守状态机 / 降速 / 停车 / 重新确认
```

最终推荐方向：

> 构建“多线索安全跟随框架”：确认式目标初始化 + 局部 ReID 目标记忆 + 单目深度辅助距离 / 障碍判断 + 保守安全状态机。

### 0.5 `tools/` 中 ReID 研究资产的工程定位

目前 `tools/reid_pc_test` 已经形成一套有价值的 PC 端研究工作区，其中包含：

- 自编写的对比、gallery 模拟、误匹配分析脚本；
- `docs/` 中的调研结论与迭代记录；
- `images/`、`weights/`、`outputs/` 等实验数据；
- 直接 clone 的上游仓库 `deep-person-reid`。

这里需要明确一个工程边界：

1. `tools/reid_pc_test` 的主要定位是**PC 侧算法研究与数据验证工作区**，不是 App 运行时依赖。
2. `deep-person-reid` 当前是一个**嵌套的独立 Git 仓库**。在首版项目中，不建议默认再把它升级成主仓库的正式 submodule。
3. 更推荐的处理方式是：
   - 主仓库重点保留你们自己写的脚本、调研文档、少量必要样例；
   - 若团队希望所有成员直接在本仓库获取这些研究资产，可以移除 `deep-person-reid` 内部的 `.git`，将其转为普通目录后再统一纳入版本管理；
   - 即使转为普通目录，也应在文档中保留其上游来源、用途和本地改动边界；
   - `images/`、`outputs/`、`weights/` 等目录默认不应进入版本库，其中 `images/` 与 `outputs/` 可能含有私人照片或由私人照片导出的结果，必须通过 `.gitignore` 明确排除。

换句话说，现阶段更合理的方向是：

> 把 `tools/reid_pc_test` 当作“团队共享的研究工作区”，不必再维持嵌套仓库形态，但也不需要把它升级成新的 submodule。

## 1. 开发原则

### 1.1 先复用 OpenBot，再做场景化增强

首版不重写整套视觉与控制框架，而是复用 OpenBot 的：

- 手机摄像头与推理链路
- App 基础交互框架
- `Control(left, right)` 控制输出方式
- 串口 / 蓝牙到底盘控制板的现有控制路径

新增部分主要集中在：

- 目标初始化与用户确认
- 目标记忆与目标保持
- ReID 增强重识别
- 跟随距离估计
- 障碍风险判断
- 面向购物场景的安全状态机

---

### 1.2 ReID 作为首版增强项，但必须可降级运行

ReID 的作用是解决“这个人是不是用户确认过的目标”，而不是替代人物检测或状态机。

首版策略：

- 若 ReID 模块部署成功，则作为目标身份匹配的强特征之一；
- 若 ReID 模块尚未跑通，则系统退化为 `bbox + 运动连续性 + 尺寸 + 颜色特征 + 状态机`；
- ReID 不能作为唯一依据，不能因为单次 ReID 高分就直接切换目标；
- 当身份不确定时，应进入 `IDENTITY_UNCERTAIN` 或停车等待确认，而不是冒险跟随。

---

### 1.3 不依赖 ARCore 作为主路线

ARCore / ARCore Depth 依赖设备与 Google Play Services for AR 支持。在中国地区或无 GMS 设备上，部署稳定性不可控，因此不作为首版基础依赖。

推荐策略：

- ARCore Depth：仅作为设备支持时的增强选项；
- 首版主线：普通 RGB 摄像头 + OpenBot 检测 + 可选 TFLite 单目深度模型；
- 安全兜底：必要时增加 ToF / 超声波等低成本传感器。

---

### 1.4 手机视觉优先，但不承诺完整视觉避障

OpenBot 的核心思想是利用手机作为主要计算与感知平台，因此首版应优先尝试用手机完成：

- 目标身份判断；
- 目标距离估计；
- 前方障碍风险判断；
- 不确定时减速或停车。

但手机视觉不应被包装成可靠完整避障系统。首版目标应是：

```text
视觉发现风险 → 减速 / 停车
```

而不是：

```text
视觉建图 → 规划绕障 → 自动绕过障碍
```

---

### 1.5 无底盘实时仿真调试必须保留

在硬件未就绪前，上位机不应只输出日志，而应支持“人肉底盘”实时闭环：

```text
测试者 A 手持手机模拟购物车视角
测试者 B 作为目标用户
App 实时显示“前进 / 左转 / 右转 / 停止 / 原地搜索”
测试者 A 按提示移动
形成视觉—决策—人工执行—视觉反馈的近似闭环
```

该模式可以提前验证目标初始化、ReID、目标保持、距离控制、目标丢失处理和 UI 可用性。

## 2. 上位机总体架构

建议将上位机拆为以下模块：

```text
ShoppingCartFollow
├─ PersonDetectorAdapter
├─ TargetInitializer
├─ TargetMemory
├─ PersonTracker
├─ ReIDFeatureExtractor
│  ├─ ColorReIDFeatureExtractor
│  └─ TFLiteReIDFeatureExtractor（增强）
├─ PersonMatcher
├─ DistanceEstimator
│  ├─ BBoxScaleDistanceEstimator
│  ├─ MonocularDepthDistanceEstimator（增强）
│  └─ FusedDistanceEstimator
├─ ObstacleRiskEstimator
│  ├─ VisionRiskEstimator
│  ├─ MonocularDepthRiskEstimator（增强）
│  └─ ExternalSensorRiskEstimator（硬件到位后）
├─ FollowStateMachine
├─ SafetyFusion
├─ ControlGenerator
├─ HumanCommandInterpreter
├─ HumanCartSimulatorUI
└─ FollowDebugLogger
```

模块职责：

| 模块 | 作用 | 首版建议 |
| --- | --- | --- |
| `PersonDetectorAdapter` | 读取 OpenBot 人物检测结果并转换为统一数据结构 | 必做 |
| `TargetInitializer` | 候选目标采集、截图、确认前逻辑 | 必做 |
| `TargetMemory` | 记录目标外观、位置、动态状态、ReID 特征库 | 必做 |
| `PersonTracker` | 维护短时轨迹连续性 | 必做 |
| `ReIDFeatureExtractor` | 对候选人框提取身份 embedding | 推荐尝试，可降级 |
| `PersonMatcher` | 融合 ReID、位置、运动、颜色、尺寸、置信度，判断目标身份 | 必做 |
| `DistanceEstimator` | 估计目标远近，用于保持 0.8-1.2 m 跟随距离 | 必做 |
| `ObstacleRiskEstimator` | 判断画面前方是否存在近距离风险 | 推荐做视觉版，硬件到位后可融合传感器 |
| `FollowStateMachine` | 管理初始化、确认、重识别、跟随、丢失、搜索、停止 | 必做 |
| `SafetyFusion` | 融合身份、距离、障碍、通信、急停等风险，决定是否允许前进 | 必做 |
| `ControlGenerator` | 根据目标位置、距离和安全状态输出 `Control(left,right)` | 必做 |
| `HumanCommandInterpreter` | 将 `Control(left,right)` 翻译成人类可执行提示 | 必做 |
| `HumanCartSimulatorUI` | 无底盘模式实时显示目标框、动作提示、分数和日志 | 必做 |
| `FollowDebugLogger` | 记录状态、匹配分数、控制量、FPS 等 | 必做 |

## 3. 两阶段目标初始化方案

### 3.1 设计动机

如果用户站在车前完成初始化，而系统一检测到人就直接开始跟随，会有两个问题：

1. 用户不知道系统锁定的是不是自己；
2. 用户为了看手机回到车旁确认时，车辆若直接起步会有安全风险。

因此，目标初始化应拆成两层：

```text
系统候选锁定
  ↓
用户确认
  ↓
目标重新进入视野
  ↓
稳定重识别
  ↓
倒计时启动
```

---

### 3.2 状态流程

```text
IDLE
  ↓ 用户点击开始
CAPTURE_TARGET
  ↓ 采集候选目标
LOCKED_PENDING_CONFIRM
  ↓ 用户确认
CONFIRMED_ARMED
  ↓ 用户回到车前视觉范围
REACQUIRE_TARGET
  ↓ 连续稳定匹配
READY_TO_FOLLOW
  ↓ 倒计时结束
FOLLOW
```

异常分支：

- `CAPTURE_TARGET` 失败：继续采集或回到 `IDLE`
- 用户点击“重拍”：回到 `CAPTURE_TARGET`
- 用户点击“取消”：清空记忆，回到 `IDLE`
- `REACQUIRE_TARGET` 超时：保持停止，提示重新进入视野或重新采集

---

### 3.3 初始化阶段应保存的信息

建议保存：

- `confirmed_snapshot`
- `confirmed_bbox`
- `bbox_area`
- `bbox_aspect_ratio`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `seed_embedding` 或少量初始化 embedding
- `initial_confidence`
- `desired_height_ratio`
- `desired_area_ratio`
- `desired_depth_proxy`

其中 `desired_height_ratio`、`desired_area_ratio` 和 `desired_depth_proxy` 用于后续距离保持。

这里要特别强调：**初始化阶段的目标不是一次性建立完整 gallery**。初始化更重要的是：

- 完成“这是我要跟随的人”的确认；
- 记录最小可用的外观与位置基准；
- 为后续 gallery 动态增长留下起点。

也就是说，首版不应把“初始化时必须采满 5-8 张 confirmed gallery”写成硬规则。

## 4. 目标记忆设计

### 4.1 多粒度目标记忆

真实购物场景中，不能假设系统始终看到完整人体。用户可能蹲下、侧身、被货架或路人遮挡，甚至只露出上半身、背包或腿部。因此 `TargetMemory` 不应只保存一张完整人体截图，而应维护多粒度特征：

```text
TargetMemory
├─ confirmedSnapshot
├─ confirmedBbox
├─ globalFeature
├─ upperFeature
├─ lowerFeature
├─ colorFeature
├─ reidEmbeddingGallery
├─ galleryUpdatePolicy
├─ lastBbox
├─ lastCenter
├─ lastDepthProxy
├─ lastMotionDirection
├─ lastSeenTime
├─ trackingConfidence
└─ lostDuration
```

---

### 4.2 动态可见区域匹配

不同可见情况采用不同权重：

| 可见情况 | 推荐匹配重点 |
| --- | --- |
| 全身可见 | global ReID + bbox + 运动连续性 + 颜色 |
| 只见上半身 | upper feature + ReID crop + 位置预测 |
| 只见下半身 | lower feature + 运动连续性 + bbox 尺寸约束 |
| 被路人短时遮挡 | 不更新目标记忆，等待重识别 |
| 多人交叉 | 提高 ReID 权重，降低单纯位置权重 |
| ReID 不确定 | 降低置信度，进入 `IDENTITY_UNCERTAIN` |

---

### 4.3 记忆更新原则

目标记忆不能被快速污染：

- 只有高匹配分数且连续稳定时才更新外观特征；
- 遮挡期、多人交叉期不更新外观特征；
- 陌生人短暂更大、更近或更居中时，不立刻切换身份；
- ReID gallery 应允许后续动态扩展，但扩展必须保守，优先在“身份明确、连续稳定、视角有新增信息”时加入新样本；
- 初始化阶段形成的样本只应视为种子记忆，而不是最终完整特征库；
- 当候选人分数接近时，不赌身份，应停车或提示重新确认。

后续需要专门设计 `galleryUpdatePolicy`，至少要回答三个问题：

1. 什么条件下允许把当前帧加入 gallery；
2. 如何判断新样本是否真的带来了新的姿态 / 视角 / 可见区域信息；
3. 如何避免被路人、遮挡和错误匹配污染。

## 5. 融合式目标身份匹配：跟踪 + 颜色 + ReID

### 5.1 匹配特征组成

| 特征 | 含义 |
| --- | --- |
| `position_score` | 候选人是否接近上一帧目标位置 |
| `motion_score` | 是否符合目标运动趋势 |
| `size_score` | bbox 面积 / 宽高变化是否合理 |
| `color_score` | 上下身 HSV 直方图相似度 |
| `reid_score` | ReID embedding 余弦相似度 |
| `confidence_score` | OpenBot 检测置信度 |

---

### 5.2 融合评分

建议默认权重：

```text
match_score =
  0.30 * reid_score
+ 0.25 * position_score
+ 0.15 * motion_score
+ 0.10 * size_score
+ 0.10 * color_score
+ 0.10 * confidence_score
```

当 ReID 未接入时，自动重分配权重：

```text
match_score =
  0.35 * position_score
+ 0.20 * motion_score
+ 0.15 * size_score
+ 0.20 * color_score
+ 0.10 * confidence_score
```

---

### 5.3 ReID 运行策略

不建议每帧、每个候选都跑 ReID。推荐事件触发：

| 场景 | ReID 策略 |
| --- | --- |
| 初始化 / 确认 | 建立最小种子记忆，不要求一次性完成完整 gallery |
| 正常 FOLLOW | 低频运行，例如每 5-10 帧确认一次 |
| 长时间稳定跟随 | 在满足保守条件时，动态补充更有差异性的 gallery 样本 |
| 多人靠近 / 交叉 | 对候选人运行 ReID |
| 目标丢失后重现 | 对所有 person 候选运行 ReID |
| ReID 模型不可用 | 自动退化到颜色 + 运动 + bbox |

---

### 5.4 三档实现路线

| 路线 | 说明 | 优先级 |
| --- | --- | --- |
| A | `ColorReIDFeatureExtractor`：颜色直方图 + ReID 接口预留 | 必做 |
| B | `AndroidReIDFeatureExtractor`：预训练轻量 ReID 模型输出 embedding，优先尝试 ONNX Runtime Mobile，TFLite 作为备选 | 推荐尝试 |
| C | 自采超市低机位、背影、遮挡数据微调 ReID | 后续增强 |

### 5.5 当前 PC 端 ReID 调研结论

截至 2026-07-05，`tools/reid_pc_test` 中的 PC 端调研可为 Android 集成提供以下结论：

- ReID 作为辅助特征是有价值的，但绝不能单独决定“跟谁走”；
- 当前最优的临时组合是 `osnet_x0_25 + diverse gallery`；
- `reid_margin = best_score - second_score` 往往比单纯绝对分数更适合判断“是否足够确定”；
- gallery 的构成比单纯增加样本数量更重要，系统更需要持续获得差异更大的同一人样本，而不必强行在初始化阶段一次采满；
- 初始化阶段可以只有种子样本，后续在稳定跟随中再逐步扩展更有信息量的 gallery；
- 下一步不应继续主要依赖人工裁剪图，而应从 OpenBot Android 检测框直接导出 crop，形成更接近真实跟随视角的数据闭环。

因此，首版 ReID 集成建议不是“先追求手机上完整模型”，而是：

```text
Android 端先把真实检测框 crop 导出来
  ↓
PC 端继续完成 gallery / probe / margin 规则验证
  ↓
再把确定下来的轻量模型和后处理逻辑接回 Android
```

## 6. 跟随距离估计与控制

### 6.1 问题定义

当前“跟踪距离过大”的原因通常不是目标身份问题，而是距离控制问题。若只用检测框高度或面积作为远近代理，则会受到以下因素影响：

- 用户身高差异；
- 相机俯仰角；
- 目标只露出上半身；
- 顾客蹲下、侧身或坐下；
- 下半身被货架、购物筐或路人遮挡；
- 手机持握或安装角度变化。

因此距离估计不能只依赖单一特征，应采用多线索融合和置信度判断。

---

### 6.2 首版距离特征

首版可用的距离线索：

| 线索 | 优点 | 缺点 |
| --- | --- | --- |
| `bbox_height_ratio` | 实现简单，和现有检测框直接兼容 | 遮挡 / 蹲下 / 身高差异会影响 |
| `bbox_area_ratio` | 对目标远近敏感 | 人体姿态变化影响大 |
| `bbox_bottom_ratio` | 与地面位置相关，身高影响相对小 | 看不到脚时不可用 |
| `monocular_depth_proxy` | 能利用图像整体深度关系 | 需要模型、算力和标定 |
| `temporal_prediction` | 短时平滑，减少抖动 | 长时间遮挡会漂移 |

---

### 6.3 推荐距离估计策略

采用 `FusedDistanceEstimator`：

```text
target_distance_proxy =
  w1 * bbox_height_proxy
+ w2 * bbox_area_proxy
+ w3 * bbox_bottom_proxy
+ w4 * monocular_depth_proxy
+ w5 * temporal_prediction
```

各权重不固定，而应根据可见性动态调整：

| 情况 | 推荐权重变化 |
| --- | --- |
| 全身可见 | bbox height + bbox bottom 权重较高 |
| 只见上半身 | 降低 bbox height，增加 monocular depth 和 temporal |
| 脚不可见 | 不使用 bbox bottom |
| 深度模型不可用 | 退化为 bbox + temporal |
| 目标遮挡严重 | 不继续前进，进入 caution / lost |

---

### 6.4 距离标定

不建议一开始追求精确米制距离。更可行的是图像标定：

1. 用户站在期望跟随距离，例如 1.0 m；
2. 系统记录此时的 `bbox_height_ratio`、`bbox_area_ratio`、`bbox_bottom_ratio` 和可选 `depth_proxy`；
3. 后续跟随时努力保持这些图像比例；
4. 如果实际跟随距离过大，则增大期望 `height_ratio` 或降低前进停止阈值。

建议目标距离仍按原计划控制在：

```text
0.8 - 1.2 m
```

但系统内部优先使用“距离状态”：

```text
TOO_FAR
OK
TOO_CLOSE
UNKNOWN
```

---

### 6.5 控制策略

控制输出应基于目标身份、距离状态和安全状态共同决定：

```text
if identity_score low:
    forward = 0
elif obstacle_risk high:
    forward = 0
elif distance_state == TOO_FAR:
    forward = positive_low_speed
elif distance_state == OK:
    forward = 0
elif distance_state == TOO_CLOSE:
    forward = 0   # 首版不主动后退
else:
    forward = 0
```

转向控制仍可使用目标中心偏差：

```text
turn = K_turn * x_error
left = forward - turn
right = forward + turn
```

首版原则：

- 不主动高速追人；
- 不主动后退；
- 不确定就停；
- 搜索时只允许原地低速扫描。

---

## 7. 单目深度估计与视觉障碍风险

### 7.1 MiDaS / Depth Anything 的定位

MiDaS、Depth Anything 这类模型属于单目深度估计：输入普通 RGB 图像，输出相对深度图。

需要注意：

- 输出通常是相对深度或相对逆深度，不天然等于真实米数；
- 需要用卷尺或标定场景把深度值映射到风险阈值；
- 在手机端同时跑人物检测、ReID 和深度模型会有算力压力；
- 深度模型应作为辅助通道，而不是唯一安全依据。

---

### 7.2 使用方式

推荐低频运行深度模型：

```text
人物检测 / 跟踪：每帧
ReID：事件触发或低频
单目深度：每 3-5 帧，或在 FOLLOW / OBSTACLE 检查时运行
```

可以输出：

```text
target_depth_proxy
front_risk_depth
ground_zone_depth
depth_confidence
```

---

### 7.3 目标距离辅助

在目标 bbox 内取深度中值：

```text
target_depth_proxy = median(depth_map within target_bbox)
```

再与 bbox 尺度融合：

```text
distance_score =
  0.5 * bbox_distance_score
+ 0.3 * depth_distance_score
+ 0.2 * temporal_score
```

若目标被局部遮挡，应取可见区域的深度中值，避免被货架或路人污染。

---

### 7.4 障碍风险检测

定义前方危险区域：

```text
画面下方 40%
画面中间 50%
```

如果该区域出现近距离深度异常或非目标前景物体，则进入风险状态：

```text
OBSTACLE_CAUTION  → 减速
OBSTACLE_STOP     → 停车
```

首版只承诺：

```text
视觉风险检测 + 减速 / 停车
```

不承诺：

```text
完整绕障路径规划
```

---

### 7.5 手机视觉与外接传感器的关系

手机视觉优先用于：

- 目标识别；
- 距离趋势；
- 前方风险判断；
- 演示可视化。

但如果视觉效果不稳定，应允许增加外接传感器作为安全兜底：

| 传感器 | 用途 |
| --- | --- |
| ToF | 近距离障碍停车 |
| 超声波 | 前方距离阈值判断 |
| 碰撞开关 | 最后物理防线 |

外接传感器不取代手机主脑，只作为安全冗余。

---

## 8. 安全状态机升级

在原状态机基础上增加“谨慎状态”和“不确定状态”：

```text
IDLE
CAPTURE_TARGET
LOCKED_PENDING_CONFIRM
CONFIRMED_ARMED
REACQUIRE_TARGET
READY_TO_FOLLOW
FOLLOW_CONFIDENT
FOLLOW_CAUTION
IDENTITY_UNCERTAIN
LOST
SEARCH
OBSTACLE_CAUTION
OBSTACLE_STOP
STOP
EMERGENCY
```

### 8.1 状态说明

| 状态 | 说明 | 运动策略 |
| --- | --- | --- |
| `FOLLOW_CONFIDENT` | 身份明确、距离正常、安全风险低 | 允许低速跟随 |
| `FOLLOW_CAUTION` | 身份基本明确，但距离或深度不稳定 | 限速 / 减速 |
| `IDENTITY_UNCERTAIN` | 多人分数接近、ReID 不确定、遮挡严重 | 停止，不切换目标 |
| `LOST` | 目标暂时丢失 | 停止前进 |
| `SEARCH` | 短时搜索 | 只允许低速原地扫描 |
| `OBSTACLE_CAUTION` | 前方风险升高 | 减速 |
| `OBSTACLE_STOP` | 前方近距离风险高 | 停车 |
| `STOP` | 等待重新初始化或人工确认 | 停止 |
| `EMERGENCY` | 急停或严重异常 | 停止，人工复位 |

---

### 8.2 安全优先级

```text
EMERGENCY > OBSTACLE_STOP > IDENTITY_UNCERTAIN > LOST > FOLLOW_CAUTION > FOLLOW_CONFIDENT
```

只要上层状态不确定，`ControlGenerator` 就不允许输出前进速度。

---

## 9. Human Cart Simulator 升级

### 9.1 显示内容升级

在原有目标框、状态和控制量基础上，增加四类评分：

```text
identity_score
tracking_score
distance_state / distance_score
obstacle_risk
```

屏幕上应显示：

- 当前状态；
- 当前目标框；
- 普通行人框；
- ReID / match 分数；
- 跟随距离状态；
- 前方风险状态；
- 大号动作提示；
- FPS / 推理耗时；
- 是否启用 ReID；
- 是否启用单目深度；
- 是否处于安全降级模式。

---

### 9.2 动作提示

大号提示优先级：

| 状态 | 提示 |
| --- | --- |
| 身份未确认 | `目标未确认，请不要移动` |
| 身份不确定 | `目标不确定，请停止` |
| 目标太远且安全 | `向前慢走` |
| 目标偏左 | `向前并左转` 或 `左转` |
| 目标偏右 | `向前并右转` 或 `右转` |
| 目标太近 | `停止` |
| 障碍风险 | `前方风险，请停止` |
| 目标丢失 | `目标丢失，请停止` |
| 搜索 | `原地左转搜索` / `原地右转搜索` |

---

### 9.3 无底盘模式验收场景

| 场景 | 观察重点 |
| --- | --- |
| 单人远离 | 是否提示向前 |
| 单人靠近 | 是否提示停止 |
| 目标偏左 / 偏右 | 转向提示是否正确 |
| 路人穿过 | 是否保持目标，不切换 |
| 目标蹲下 | 是否降低置信度或进入 caution |
| 目标被遮挡 | 是否停止前进并进入 LOST |
| 目标重新出现 | 是否用 ReID / 颜色 / 运动重新识别 |
| 画面前方出现近物 | 是否进入 OBSTACLE_CAUTION / STOP |

## 10. 控制输出策略

### 10.1 OpenBot 兼容输出

首版仍保持 OpenBot 兼容控制方式：

- 上位机只输出 `Control(left, right)`；
- Android 端不处理四驱、麦轮和 AT8236 细节；
- 下位机负责左右轮到四电机的映射。

---

### 10.2 控制输入

`ControlGenerator` 不应只接收目标 bbox，而应接收综合感知结果：

```text
target_bbox
identity_score
distance_state
distance_score
obstacle_risk
follow_state
```

---

### 10.3 控制规则

```text
if emergency:
    Control(0, 0)

else if obstacle_stop:
    Control(0, 0)

else if identity_uncertain:
    Control(0, 0)

else if lost:
    Control(0, 0)

else if search:
    Control(-turn_low, turn_low) 或仅显示搜索提示

else if follow_confident / follow_caution:
    根据距离状态计算 forward
    根据目标中心偏差计算 turn
    限幅后输出 left/right
```

## 13. 风险与降级方案

| 风险 | 应对 | 降级 |
| --- | --- | --- |
| ReID 模型部署失败 | 保留接口，先用颜色 + 运动匹配 | 不影响首版运行 |
| 单目深度太慢 | 低频运行或关闭 | 退化为 bbox 距离控制 |
| ARCore 不可用 | 不作为主线 | 不影响设计 |
| bbox 距离估计不稳 | 加入标定、平滑和深度辅助 | 降速 / 停止 |
| 手机视觉障碍判断不可靠 | 只做风险停车，不做绕障 | 外接 ToF / 超声波 |
| 身份分数接近 | 进入 `IDENTITY_UNCERTAIN` | 停车并提示重新确认 |
| 目标被长时间遮挡 | 搜索超时 | STOP 等待人工干预 |
| `tools/` 中直接 clone 上游仓库 | 移除内部 `.git` 后转为普通目录，避免嵌套仓库干扰协作 | 后续补充来源说明并忽略大文件产物 |
| ReID 研究目录含私人照片 | 默认不提交 `images/` 与 `outputs/`，并在仓库规则中显式忽略 | 仅保留代码、文档和必要说明 |

## 14. 当前最建议的下一步

基于当前代码状态，下一步不应立刻沉迷于继续换模型，而应先按下面顺序推进：

1. **先把当前 OpenBot 购物车原型链路彻底验收清楚**  
   跑通确认、重拍、取消、重识别、倒计时、FOLLOW、LOST、SEARCH、STOP，并核对 `TargetMatcher` 当前究竟在哪些多人场景会误锁人。

2. **补齐控制闭环与丢失目标安全策略**  
   优先接通 `vehicle.setControl()`、实现 `LOST → SEARCH → STOP`，因为这决定了后续即便硬件一到位，也能直接做低速安全联调。

3. **把 ReID 研究从“PC 上看起来有希望”推进到“和 OpenBot 输入一致”**  
   在 Android 端导出真实检测框 crop，用当前 `tools/reid_pc_test` 的脚本验证 gallery 采样、`reid_score`、`reid_margin` 和多人混淆情况。

4. **新增距离估计与距离状态显示**  
   在 Human Cart Simulator 里实时显示：
   ```text
   bbox_height_ratio
   bbox_area_ratio
   bbox_bottom_ratio
   distance_state = TOO_FAR / OK / TOO_CLOSE / UNKNOWN
   ```
   先解决当前“跟踪距离过大”的问题。

5. **把 ReID 与单目深度保持为可插拔增强模块继续推进**  
   ReID 用于“是不是目标”，单目深度用于“距离和风险”，二者都不能绕过安全状态机；首版必须始终保留“无 ReID / 无深度也能降级运行”的路径。

---

## 15. 结论

上位机方向的最新核心创新点可以总结为：

> 面向超市场景的多线索安全跟随框架。系统通过确认式目标初始化确定跟随对象，通过颜色、时序跟踪和 ReID 保持目标身份，通过 bbox 标定和单目深度辅助控制跟随距离，通过视觉风险和可选 ToF / 超声波判断前方安全，并通过保守状态机保证在身份不确定、目标丢失或障碍风险升高时优先停车。

该方案相比“每帧跟随最大人框”有三点本质提升：

1. 从“检测到人”升级为“确认并保持指定目标身份”；
2. 从“目标框控制”升级为“身份、距离、风险多线索融合控制”；
3. 从“能跟随”升级为“不确定时不冒险跟随”。
