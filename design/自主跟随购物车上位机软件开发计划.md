# 自主跟随购物车上位机软件开发计划（行为决策与多线索安全跟随版）

更新日期：2026-07-08  
适用对象：上位机 / Android 端负责人  
当前定位：在 OpenBot 现有 Android 工程基础上，将人物检测、ReID、距离控制、局部可通行空间、状态机与控制输出统一到一个安全行为决策层中。

---

## 0. 当前判断与路线收口

### 0.1 文档定位

本文档不再承担完整进度流水账的职责。已完成的代码、采集器、PC 测试脚本和测试结果分别记录在：

- `dev/OpenBot/android/cartfollow-devlog.md`
- `tools/reid_pc_test/README.md`
- `tools/reid_pc_test/docs/ReID调研测试迭代记录与下一步计划.md`
- `tools/reid_pc_test/docs/person-crop-collector完成计划.md`

本文档只保留对后续上位机路线有直接影响的判断：系统下一阶段应从“验证 ReID 是否可用”转向“如何安全、主动、可解释地决定小车动作”。

### 0.2 核心阶段判断

ReID 的调研和离线验证已经足以支撑首版工程接入。它的角色应收敛为身份线索之一，而不是继续作为独立主线消耗主要开发时间。

上位机当前真正需要解决的是行为仲裁问题：

```text
1. 跟谁：目标身份保持 / ReID / 遮挡重识别
2. 离多远：跟随距离估计 / 距离控制
3. 能不能走：障碍风险判断 / 安全停车
4. 怎么动：FOLLOW / AVOID / LOCAL_SEARCH / BLOCKED_WAIT / hard STOP
```

推荐将所有线索统一成以下链路：

```text
Perception Evidence
  ↓
BehaviorDecision / SafetyFusion
  ↓
Safe Action
  ↓
Control(left, right) 或 Human Cart Simulator 提示
```

### 0.3 首版策略口径

首版仍然坚持安全优先，但安全优先不等于一丢目标就 hard `STOP`。后续状态机应明确区分：

| 概念 | 含义 |
| --- | --- |
| `motion_stop` | 线速度清零，禁止继续向前，但系统仍在观察、搜索或等待重捕获 |
| `LOCAL_SEARCH` | 低速原地扫描，根据最后目标位置、消失方向和场景风险选择方向 |
| `REACQUIRE` | 疑似目标重新出现，多帧融合 ReID、bbox 连续性、距离和风险后恢复 |
| hard `STOP` | 搜索失败、风险过高、急停、通信异常或需要人工重新确认 |

也就是说，智能性应体现在“停住以后还会安全地找人”，而不是“不停地冒险追人”。

### 0.4 近期与远期能力边界

近期默认路线仍是手机端本地闭环：

- 普通 RGB 摄像头；
- OpenBot person detector；
- ReID 低频或事件触发推理；
- bbox 距离标定与图像伺服；
- 左 / 中 / 右局部可通行空间估计；
- Human Cart Simulator 先验证行为策略，再低速接入真实底盘。

如果后续手机性能仍有余量，可以逐步打开增强能力：

- 更高频 ReID 或更大模型；
- 单目深度用于局部风险热力图；
- 光流 / TTC 近似用于动态接近风险；
- 更细的动作走廊评分，例如 `SLOW_FORWARD / LEFT_ARC / RIGHT_ARC / ROTATE_LEFT / ROTATE_RIGHT`。

如果手机性能不足，应优先降级频率，而不是牺牲安全：

- ReID 只在初始化、多人混淆、目标重现时运行；
- 深度模型关闭或低频运行；
- 局部避障退化为三方向启发式评分；
- 外接 ToF / 超声波作为近距离安全兜底。

云端服务器可以作为后续研究或演示增强，但不应进入首版实时安全闭环。可考虑的云端用途包括离线日志分析、参数推荐、复杂场景复盘、模型评估和非实时策略调优；不建议依赖云端完成急停、近距离避障或实时身份判决。

### 0.5 `tools/` 中 ReID 研究资产的工程定位

目前 `tools/reid_pc_test` 已经形成一套有价值的 PC 端研究工作区，其中包含：

- 自编写的对比、gallery 模拟、误匹配分析脚本；
- `docs/` 中的调研结论与迭代记录；
- `images/`、`weights/`、`outputs/` 等实验数据；
- 直接 clone 的上游仓库 `deep-person-reid`。

这里需要明确一个工程边界：

1. `tools/reid_pc_test` 的主要定位是**PC 侧算法研究与数据验证工作区**，不是 App 运行时依赖。
2. `deep-person-reid` 在首版项目中，不建议默认再把它升级成主仓库的正式 submodule。
3. 更推荐的处理方式是：
   - 主仓库重点保留你们自己写的脚本、调研文档、少量必要样例；
   - 所有成员直接在本仓库获取这些研究资产，因此移除 `deep-person-reid` 内部的 `.git`，将其转为普通目录后再统一纳入版本管理；
   - README文档中保留了其上游来源、用途和本地改动边界；
   - `images/`、`outputs/`、`weights/` 等目录默认不应进入版本库，其中 `images/` 与 `outputs/` 可能含有私人照片或由私人照片导出的结果，必须通过 `.gitignore` 明确排除。

## 1. 开发原则

### 1.1 先复用 OpenBot，再做场景化增强

首版不重写整套视觉与控制框架，而是复用 OpenBot 的：

- 手机摄像头与推理链路
- App 基础交互框架
- `Control(left, right)` 控制输出方式
- 串口 / 蓝牙到底盘控制板的现有控制路径

新增部分主要集中在：

- 目标初始化与用户确认
- 目标记忆与目标保持
- ReID 增强重识别
- 跟随距离估计
- 障碍风险判断
- 面向购物场景的安全状态机

---

### 1.2 ReID 作为首版增强项，但必须可降级运行

ReID 的作用是解决“这个人是不是用户确认过的目标”，而不是替代人物检测或状态机。

首版策略：

- 若 ReID 模块部署成功，则作为目标身份匹配的强特征之一；
- 若 ReID 模块尚未跑通，则系统退化为 `bbox + 运动连续性 + 尺寸 + 颜色特征 + 状态机`；
- ReID 不能作为唯一依据，不能因为单次 ReID 高分就直接切换目标；
- 当身份不确定时，应先进入 `motion_stop + IDENTITY_UNCERTAIN / LOCAL_SEARCH`，禁止继续前进，但允许原地低速搜索和多帧重捕获；只有搜索失败或风险过高时才进入 hard `STOP`。

---

### 1.3 不依赖 ARCore 作为主路线

ARCore / ARCore Depth 依赖设备与 Google Play Services for AR 支持。在中国地区或无 GMS 设备上，部署稳定性不可控，因此不作为首版基础依赖。

推荐策略：

- ARCore Depth：仅作为设备支持时的增强选项；
- 首版主线：普通 RGB 摄像头 + OpenBot 检测 + 可选 TFLite 单目深度模型；
- 安全兜底：必要时增加 ToF / 超声波等低成本传感器。

---

### 1.4 手机视觉优先，但不承诺完整视觉避障

OpenBot 的核心思想是利用手机作为主要计算与感知平台，因此首版应优先尝试用手机完成：

- 目标身份判断；
- 目标距离估计；
- 前方障碍风险判断；
- 不确定时减速或停车。

但手机视觉不应被包装成可靠完整避障系统。首版目标应是：

```text
视觉发现风险 → 减速 / 停车
```

而不是：

```text
视觉建图 → 规划绕障 → 自动绕过障碍
```

---

### 1.5 无底盘实时仿真调试必须保留

在硬件未就绪前，上位机不应只输出日志，而应支持“人肉底盘”实时闭环：

```text
测试者 A 手持手机模拟购物车视角
测试者 B 作为目标用户
App 实时显示“前进 / 左转 / 右转 / 停止 / 原地搜索”
测试者 A 按提示移动
形成视觉—决策—人工执行—视觉反馈的近似闭环
```

该模式可以提前验证目标初始化、ReID、目标保持、距离控制、目标丢失处理和 UI 可用性。

## 2. 上位机总体架构

建议将上位机从“感知模块并列输出”升级为“证据输入 + 行为决策 + 控制执行”的三层结构：

```text
ShoppingCartFollow
├─ PerceptionEvidence
│  ├─ PersonDetectorAdapter
│  ├─ TargetInitializer
│  ├─ TargetMemory
│  ├─ PersonTracker
│  ├─ ReIDFeatureExtractor
│  │  ├─ ColorReIDFeatureExtractor
│  │  └─ OnDeviceReIDFeatureExtractor
│  ├─ DistanceEstimator
│  │  ├─ ImageSetpointDistanceEstimator
│  │  └─ MonocularDepthDistanceEstimator（增强）
│  ├─ LocalTraversabilityEstimator
│  ├─ MotionRiskEstimator（增强）
│  └─ ExternalSensorRiskEstimator（硬件兜底）
├─ BehaviorDecision
│  ├─ PersonMatcher
│  ├─ FollowStateMachine
│  ├─ SafetyFusion
│  ├─ LocalAvoidancePolicy
│  └─ ActionArbitrator
├─ ActionExecution
│  ├─ ControlGenerator
│  ├─ HumanCommandInterpreter
│  └─ VehicleControlAdapter
└─ DebugAndData
   ├─ HumanCartSimulatorUI
   ├─ FollowDebugLogger
   └─ SequenceReplay / PC Regression
```

模块职责：

| 模块 | 作用 | 首版建议 |
| --- | --- | --- |
| `PersonDetectorAdapter` | 读取 OpenBot 人物检测结果并转换为统一数据结构 | 必做 |
| `TargetInitializer` | 候选目标采集、截图、确认前逻辑 | 必做 |
| `TargetMemory` | 记录目标外观、位置、动态状态、ReID 特征库 | 必做 |
| `PersonTracker` | 维护短时轨迹连续性 | 必做 |
| `ReIDFeatureExtractor` | 对候选人框提取身份 embedding，只作为身份线索之一 | 推荐接入，可低频运行 |
| `PersonMatcher` | 融合 ReID、位置、运动、颜色、尺寸、置信度，输出身份证据 | 必做 |
| `DistanceEstimator` | 估计目标远近，用于保持 0.8-1.2 m 跟随距离 | 必做 |
| `LocalTraversabilityEstimator` | 输出左 / 中 / 右可通行评分与置信度 | 必做视觉简版 |
| `MotionRiskEstimator` | 判断动态接近风险，例如路人横穿、目标突然停止 | 增强项 |
| `ExternalSensorRiskEstimator` | 接入 ToF / 超声波 / 碰撞开关等安全冗余 | 硬件到位后 |
| `FollowStateMachine` | 管理初始化、确认、重识别、跟随、丢失、搜索、停止 | 必做 |
| `SafetyFusion` | 融合身份、距离、障碍、通信、急停等风险，决定安全边界 | 必做 |
| `LocalAvoidancePolicy` | 根据目标方向和可通行空间选择局部避障候选动作 | 必做简版 |
| `ActionArbitrator` | 在跟随、搜索、等待、避障、急停之间做最终行为选择 | 必做 |
| `ControlGenerator` | 将安全动作映射为 `Control(left,right)` | 必做 |
| `VehicleControlAdapter` | 隔离 OpenBot 控制接口与底盘实际通信细节 | 实车阶段必做 |
| `HumanCommandInterpreter` | 将 `Control(left,right)` 翻译成人类可执行提示 | 必做 |
| `HumanCartSimulatorUI` | 无底盘模式实时显示目标框、动作提示、分数和日志 | 必做 |
| `FollowDebugLogger` | 记录证据、状态、动作、控制量、耗时与降级原因 | 必做 |

### 2.1 统一证据结构

后续各模块不要直接互相调用决策，而应先输出统一证据：

```text
IdentityEvidence:
  identity_score
  reid_score
  reid_margin
  tracking_score
  identity_confidence

DistanceEvidence:
  distance_state
  height_scale
  area_scale
  bottom_shift
  distance_confidence

TraversabilityEvidence:
  left_free_score
  center_free_score
  right_free_score
  left_risk
  center_risk
  right_risk
  traversability_confidence

SystemSafetyEvidence:
  emergency_stop
  communication_ok
  detector_ok
  inference_latency_ms
  thermal_or_fps_degraded
```

行为决策层只消费这些证据，不关心它们来自 ReID、bbox、深度、光流、外接传感器还是云端离线分析。

### 2.2 行为输出结构

第一版必须把“状态”和“动作”分开，避免状态机爆炸。

`FollowStateMachine` 管理任务阶段和身份阶段，例如初始化、确认、跟随、丢失、重捕获、停止；`ActionArbitrator` 根据当前状态和证据选择本帧动作，例如慢速跟随、停止、原地搜索、左弧线、右弧线或等待。

最终行为输出应比 `Control(left,right)` 更高一层：

```text
BehaviorAction:
  FOLLOW_SLOW
  FOLLOW_CAUTION
  LEFT_ARC
  RIGHT_ARC
  MOTION_STOP
  LOCAL_SEARCH_LEFT
  LOCAL_SEARCH_RIGHT
  BLOCKED_WAIT
  REACQUIRE_HOLD
  HARD_STOP
  EMERGENCY_STOP
```

同时建议输出完整决策结果：

```text
BehaviorDecisionResult:
  current_state
  selected_action
  action_reason
  safety_block_reason
  confidence
  degraded_mode
```

`ControlGenerator` 只负责把这些动作映射成 OpenBot 兼容控制量。这样做可以让 Human Cart Simulator、真实底盘和 PC replay 共用同一套决策语义。

## 3. 两阶段目标初始化方案

### 3.1 设计动机

如果用户站在车前完成初始化，而系统一检测到人就直接开始跟随，会有两个问题：

1. 用户不知道系统锁定的是不是自己；
2. 用户为了看手机回到车旁确认时，车辆若直接起步会有安全风险。

因此，目标初始化应拆成两层：

```text
系统候选锁定
  ↓
用户确认
  ↓
目标重新进入视野
  ↓
稳定重识别
  ↓
倒计时启动
```

---

### 3.2 状态流程

```text
IDLE
  ↓ 用户点击开始
CAPTURE_TARGET
  ↓ 采集候选目标
LOCKED_PENDING_CONFIRM
  ↓ 用户确认
CONFIRMED_ARMED
  ↓ 用户回到车前视觉范围
REACQUIRE_TARGET
  ↓ 连续稳定匹配
READY_TO_FOLLOW
  ↓ 倒计时结束
FOLLOW
```

异常分支：

- `CAPTURE_TARGET` 失败：继续采集或回到 `IDLE`
- 用户点击“重拍”：回到 `CAPTURE_TARGET`
- 用户点击“取消”：清空记忆，回到 `IDLE`
- `REACQUIRE_TARGET` 超时：保持停止，提示重新进入视野或重新采集

---

### 3.3 初始化阶段应保存的信息

建议保存：

- `confirmed_snapshot`
- `confirmed_bbox`
- `bbox_area`
- `bbox_aspect_ratio`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `seed_embedding` 或少量初始化 embedding
- `initial_confidence`
- `desired_height_ratio`
- `desired_area_ratio`
- `desired_depth_proxy`

其中 `desired_height_ratio`、`desired_area_ratio` 和 `desired_depth_proxy` 用于后续距离保持。

这里要特别强调：**初始化阶段的目标不是一次性建立完整 gallery**。初始化更重要的是：

- 完成“这是我要跟随的人”的确认；
- 记录最小可用的外观与位置基准；
- 为后续 gallery 动态增长留下起点。

也就是说，首版不应把“初始化时必须采满 5-8 张 confirmed gallery”写成硬规则。

## 4. 目标记忆设计

### 4.1 多粒度目标记忆

真实购物场景中，不能假设系统始终看到完整人体。用户可能蹲下、侧身、被货架或路人遮挡，甚至只露出上半身、背包或腿部。因此 `TargetMemory` 不应只保存一张完整人体截图，而应维护多粒度特征：

```text
TargetMemory
├─ confirmedSnapshot
├─ confirmedBbox
├─ globalFeature
├─ upperFeature
├─ lowerFeature
├─ colorFeature
├─ reidEmbeddingGallery
├─ galleryUpdatePolicy
├─ lastBbox
├─ lastCenter
├─ lastDepthProxy
├─ lastMotionDirection
├─ lastSeenTime
├─ trackingConfidence
└─ lostDuration
```

---

### 4.2 动态可见区域匹配

不同可见情况采用不同权重：

| 可见情况 | 推荐匹配重点 |
| --- | --- |
| 全身可见 | global ReID + bbox + 运动连续性 + 颜色 |
| 只见上半身 | upper feature 接口 + 位置预测 + 颜色 |
| 只见下半身 | lower feature 接口 + 运动连续性 + bbox 尺寸约束 |
| 被路人短时遮挡 | 不更新目标记忆，等待重识别 |
| 多人交叉 | 提高 ReID 权重，降低单纯位置权重 |
| ReID 不确定 | 降低置信度，进入 `IDENTITY_UNCERTAIN` |

这里的“上半身 / 下半身 / 局部 crop”是工程上的多粒度记忆接口，不意味着首版已经实现专门的 part-based ReID、人体解析或部件对齐网络。首版更现实的做法是：当检测框只覆盖局部人体时，降低身份置信度，并更多依赖位置连续性、运动趋势和状态机。

---

### 4.3 记忆更新原则

目标记忆不能被快速污染：

- 只有高匹配分数且连续稳定时才更新外观特征；
- 遮挡期、多人交叉期不更新外观特征；
- 陌生人短暂更大、更近或更居中时，不立刻切换身份；
- ReID gallery 应允许后续动态扩展，但扩展必须保守，优先在“身份明确、连续稳定、视角有新增信息”时加入新样本；
- 初始化阶段形成的样本只应视为种子记忆，而不是最终完整特征库；
- 当候选人分数接近时，不赌身份，应停车或提示重新确认。

后续需要专门设计 `galleryUpdatePolicy`，至少要回答三个问题：

1. 什么条件下允许把当前帧加入 gallery；
2. 如何判断新样本是否真的带来了新的姿态 / 视角 / 可见区域信息；
3. 如何避免被路人、遮挡和错误匹配污染。

---

## 5. 融合式目标身份匹配：跟踪 + 颜色 + ReID

### 5.1 匹配特征组成

| 特征 | 含义 |
| --- | --- |
| `position_score` | 候选人是否接近上一帧目标位置 |
| `motion_score` | 是否符合目标运动趋势 |
| `size_score` | bbox 面积 / 宽高变化是否合理 |
| `color_score` | 上下身 HSV 直方图相似度 |
| `reid_score` | 候选人与目标 gallery 的最高余弦相似度 |
| `reid_margin` | 当前最佳候选与第二候选的 ReID 分数差 |
| `reid_confidence` | 由 `reid_score`、`reid_margin`、候选排名和多帧稳定性共同得到的身份置信度 |
| `confidence_score` | OpenBot 检测置信度 |

为了避免概念混淆，建议明确区分：

| 名称 | 含义 | 是否可单独决策 |
| --- | --- | --- |
| `reid_score` | 候选人与目标 gallery 的最高余弦相似度 | 不可 |
| `reid_margin` | 当前最佳候选与第二候选的分数差 | 可作为置信度信号 |
| `match_score` | ReID、位置、运动、尺寸、颜色、检测置信度融合后的最终匹配分 | 可用于状态机，但仍需连续帧确认 |

本项目不采用：

```text
reid_score > threshold
→
直接确认目标
```

而采用：

```text
reid_margin + 位置连续性 + 多帧稳定性
```

的组合判断。

---

### 5.2 融合评分

这里需要特别强调：融合评分中的 ReID 项，不应理解为“原始余弦相似度直接线性加权”。

更准确的流程是：

```text
target_score = max cosine(candidate, confirmedGallery)
best_score = 当前画面中所有候选人的最高 target_score
second_score = 当前画面中第二高 target_score
reid_margin = best_score - second_score
```

真正进入融合的，应是经过 `margin`、候选排名和多帧稳定性处理后的：

```text
reid_confidence
```

而不是裸的 `reid_score`。

推荐更保守的两层策略：

```text
第一层：ReID gate

if reid_margin < MARGIN_WEAK:
    identity_state = UNCERTAIN

elif MARGIN_WEAK <= reid_margin < MARGIN_STRONG:
    ReID 只提供弱确认信号，仍需位置连续 + 多帧稳定

else:
    ReID 提供较强身份确认信号，但仍不能绕过位置、运动和尺寸约束
```

```text
第二层：融合评分
```

示例权重可调整为：

```text
match_score =
  0.25 * reid_confidence
+ 0.25 * position_score
+ 0.20 * motion_score
+ 0.10 * size_score
+ 0.10 * color_score
+ 0.10 * confidence_score
```

当 ReID 未接入或 `reid_margin` 过弱时，系统应自动退化为：

```text
match_score =
  0.35 * position_score
+ 0.20 * motion_score
+ 0.15 * size_score
+ 0.20 * color_score
+ 0.10 * confidence_score
```

---

因此，首版更推荐把 ReID 理解为：

> 身份置信度与身份门控信号，而不是身份判决器。

### 5.3 ReID 运行策略

不建议每帧、每个候选都跑 ReID。推荐事件触发：

| 场景 | ReID 策略 |
| --- | --- |
| 初始化 / 确认 | 建立最小种子记忆与 `confirmedGallery`，不要求一次性完成完整大容量 gallery |
| 正常 FOLLOW | 低频运行，例如每 5-10 帧确认一次 |
| 多人靠近 / 交叉 | 对候选人运行 ReID |
| 目标丢失后重现 | 对所有 person 候选运行 ReID |
| ReID 模型不可用 | 自动退化到颜色 + 运动 + bbox |

其中：

- `confirmedGallery + diverse selection + margin reject` 是首版优先实现目标；
- dynamic gallery 属于后续增强，不作为第一版 ReID 接入的必要条件；
- 若未来加入动态更新，也只能在 `FOLLOW_CONFIDENT`、位置连续、无多人交叉、`reid_margin` 足够大、连续多帧稳定时触发。

---

### 5.4 三档实现路线

| 路线 | 说明 | 优先级 |
| --- | --- | --- |
| A | `ColorReIDFeatureExtractor`：颜色直方图 + ReID 接口预留 | 必做 |
| B | `AndroidReIDFeatureExtractor`：预训练轻量 ReID 模型输出 embedding，优先尝试 ONNX Runtime Mobile，TFLite 作为备选 | 推荐尝试 |
| C | 自采超市低机位、背影、遮挡数据微调 ReID | 后续增强 |

### 5.5 当前 PC 端 ReID 调研结论

截至 2026-07-07，`tools/reid_pc_test` 已经完成从人工裁剪数据到 OpenBot 自动检测框 crop 数据的首轮验证。当前最有工程指导意义的是 `images_openbot_clean` 数据集：

```text
身份数：3
总图片数：209
采集来源：OpenBot Android Person Crop Collector
模型：osnet_x0_25 + osnet_x0_25_market1501.pth
gallery 策略：diverse selection
```

本轮结果给 Android 集成提供以下结论：

- OpenBot 自动检测框 crop 比早期人工裁剪图更接近未来 App 运行时输入，也更适合作为当前判断依据；
- `osnet_x0_25` 在当前数据上仍是首版主线候选，兼顾轻量和效果；
- `confirmedGallery` 建议首版优先采用 `diverse gallery-k=8`，当前明显优于 `gallery-k=5`；
- `reid_margin = best_score - second_score` 比单纯绝对相似度更适合作为身份置信证据；
- ReID 对“目标仍在画面中时从候选人里选谁”很有帮助，但不能单独判断“目标是否真的重新出现”；
- dynamic gallery 更新方向合理，但应后置，不作为第一版 ReID 接入前提。

关键实验结果如下：

| 测试 | g5 / 基线结果 | g8 / 推荐结果 | 结论 |
| --- | ---: | ---: | --- |
| Pairwise Top-1 最近邻身份正确率 | - | `0.990` | 真实 crop 上 identity embedding 有明显可用性 |
| Gallery-Probe 强制识别准确率 | `0.840` | `0.876` | `gallery-k=8` 优于 `gallery-k=5` |
| Target-present 强制选择目标准确率 | `0.767` | `0.843` | g8 更适合跟随场景候选排序 |
| Target-present `margin>=0.05` accepted accuracy | `0.879` | `0.957` | margin 可作为较强身份置信证据 |
| Target-absent `margin>=0.05` false accept rate | `0.457` | `0.457` | 单靠 margin 无法可靠拒绝目标缺席 |
| Target-absent `margin>=0.10` false accept rate | `0.194` | `0.184` | 提高阈值仍不能消除误接收 |

因此，阶段 3 的重点已经从“证明 ReID 是否有用”转为：

```text
把 osnet_x0_25 + diverse confirmedGallery(k=8)
  ↓
接入 TargetMemory / TargetMatcher / FollowStateMachine
  ↓
与位置连续性、bbox 尺寸、运动趋势、连续多帧稳定性融合
  ↓
在身份不确定或目标缺席风险高时进入 IDENTITY_UNCERTAIN / STOP
```

正式部署 Android ReID 前，下一步不应继续单纯追模型分数，而应先完成运行时接口设计：`ReIDFeatureExtractor`、`ReIDGallery`、`TargetMatcher`、`TargetMemory` 与状态机之间的数据结构和门控逻辑。

### 5.6 ReID 失败 / 不确定时的处理

若出现以下情况，系统不得仅凭 ReID 恢复 `FOLLOW`：

- `best_score` 和 `second_score` 接近，`reid_margin` 过小；
- 多个候选人与目标 gallery 相似度都较高；
- 目标刚从 `LOST` 状态恢复，但位置跳变过大；
- 当前候选与目标 gallery 相似，但运动轨迹与上一帧目标不连续；
- 画面中存在多人交叉、遮挡或检测框明显不稳定。

处理策略：

- 正常 `FOLLOW` 中：降级为 `FOLLOW_CAUTION`，降低速度或停止前进；
- `REACQUIRE_TARGET` 中：继续等待，不进入 `READY_TO_FOLLOW`；
- `LOST / SEARCH` 中：不把路人当作目标恢复跟随；
- 超时后进入 `STOP` 或要求用户重新确认。

## 6. 跟随距离估计与控制

### 6.1 问题定义

当前“跟踪距离过大”的原因通常不是目标身份问题，而是距离控制问题。此前计划中默认采用“多特征线性融合距离代理”：

```text
target_distance_proxy =
  w1 * bbox_height_proxy
+ w2 * bbox_area_proxy
+ w3 * bbox_bottom_proxy
+ w4 * monocular_depth_proxy
+ w5 * temporal_prediction
```

这条路线的问题不是单纯“参数还没调好”，而是它容易陷入：

- 缺乏稳定物理尺度来源；
- 可解释性弱，距离不对时很难定位到底是哪一路出了问题；
- 若想真正学权重，就需要大量带真实距离标注的数据；
- 对首版课程原型而言，调参与验证成本偏高。

因此，距离控制的主方案需要从“估计一个连续距离值”转向“保持初始化时的期望图像尺度”。

若只用检测框高度或面积作为远近代理，则会受到以下因素影响：

- 用户身高差异；
- 相机俯仰角；
- 目标只露出上半身；
- 顾客蹲下、侧身或坐下；
- 下半身被货架、购物筐或路人遮挡；
- 手机持握或安装角度变化。

因此距离估计不能只依赖单一特征，应采用多线索融合和置信度判断。

---

### 6.2 首版核心思想：Distance State + Image-based Visual Servoing

购物车真正需要解决的问题不是：

```text
用户现在离我 0.94 m 还是 1.03 m？
```

而是：

```text
用户是不是比期望状态更远了？
我现在应该前进、停止，还是因为不确定而等待？
```

因此系统内部更适合输出：

```text
TOO_FAR
OK
TOO_CLOSE
UNKNOWN
```

而不是把“米制距离”当作首版主目标。

这对应经典的 Image-based Visual Servoing（IBVS）思想：

> 控制机器人，使目标保持在图像中的期望位置和尺度，而不是先恢复完整三维世界坐标。

对购物车来说，更适合被控制的是：

- `bbox_height_ratio`
- `bbox_area_ratio`
- `bbox_bottom_ratio`
- 目标在画面中的相对位置

本质上，距离控制可以理解为：

> 保持目标的图像尺度接近初始化时记录下来的 setpoint。

### 6.3 推荐主方案：初始化距离标定 + 图像尺度保持

初始化阶段，在用户站在期望跟随距离时，系统记录：

```text
desired_bbox_height_ratio
desired_bbox_area_ratio
desired_bbox_bottom_ratio
(optional) desired_depth_proxy
```

这里记录的是：

```text
期望图像尺度
```

不是：

```text
真实米制距离
```

后续每帧计算类似：

```text
height_scale = current_height / desired_height
area_scale = sqrt(current_area / desired_area)
bottom_shift = current_bottom - desired_bottom
```

系统持续比较“当前图像状态”和“初始化期望状态”，判断目标是否偏远、偏近或状态不可信。

### 6.4 Distance State 规则

可先采用解释性更强的规则式状态判断：

```text
if height_scale < far_threshold:
    TOO_FAR
elif height_scale > close_threshold:
    TOO_CLOSE
elif bbox / pose / visibility unreliable:
    UNKNOWN
else:
    OK
```

其中：

- `bbox_height_ratio` 是首版主线；
- `bbox_area_ratio` 和 `bbox_bottom_ratio` 作为辅助一致性检查；
- 若姿态异常、遮挡严重、bbox 明显失真，则直接进入 `UNKNOWN`，优先停车。

---

### 6.5 软件结构建议

建议将原来偏“多权重融合”的距离结构，改写为更清晰的估计器族：

```text
DistanceEstimator
├── ImageSetpointDistanceEstimator
├── GroundPlaneDistanceEstimator
├── MonocularDepthDistanceEstimator
├── MarkerDistanceEstimator（调试用）
└── DistanceStateFusion
```

统一输出建议为：

```text
value
state
confidence
failure_reason
```

---

### 6.6 Ground Plane Distance 作为第二阶段增强

在首版图像伺服方案跑通后，建议预留：

```text
GroundPlaneDistanceEstimator
```

它利用：

- 相机高度；
- 手机安装角度；
- 相机内参；
- 地面平面假设；
- `bbox bottom` 或脚部位置；

将图像中的底部位置映射为近似地面距离。

它的优势是更接近物理距离，并且比“纯 bbox 高度”更不依赖目标身高；但前提是脚部或 bbox 底部必须足够可信，因此更适合作为第二阶段增强，而不是首版前置依赖。

### 6.7 单目深度在距离控制中的定位

单目深度模型不是首版距离主来源。它更适合：

- 在 `ImageSetpointDistanceEstimator` 不稳定时提供辅助信息；
- 帮助判断局部遮挡、近处突起和左右空间差异；
- 在后续 `DistanceStateFusion` 中提供“支持 / 反对”信号。

不建议把单目深度直接当成首版核心距离来源，因为：

- 算力压力大；
- 多数输出仍是 relative depth；
- 一旦不准，调试成本很高。

### 6.8 距离标定

不建议一开始追求精确米制距离。更可行的是图像标定：

1. 用户站在期望跟随距离，例如 1.0 m；
2. 系统记录此时的 `desired_bbox_height_ratio`、`desired_bbox_area_ratio`、`desired_bbox_bottom_ratio` 和可选 `desired_depth_proxy`；
3. 后续跟随时努力保持这些图像比例；
4. 如果实际跟随距离过大，则调整期望尺度或 `Distance State` 阈值，而不是强行训练复杂权重。

建议目标距离仍按原计划控制在：

```text
0.8 - 1.2 m
```

但系统内部优先使用“距离状态”：

```text
TOO_FAR
OK
TOO_CLOSE
UNKNOWN
```

---

### 6.9 控制策略

控制输出应基于目标身份、距离状态和安全状态共同决定：

```text
if identity_score low:
    forward = 0
elif obstacle_risk high:
    forward = 0
elif distance_state == TOO_FAR:
    forward = positive_low_speed
elif distance_state == OK:
    forward = 0
elif distance_state == TOO_CLOSE:
    forward = 0   # 首版不主动后退
else:
    forward = 0
```

转向控制仍可使用目标中心偏差：

```text
turn = K_turn * x_error
left = forward - turn
right = forward + turn
```

首版原则：

- 不主动高速追人；
- 不主动后退；
- 不确定就停；
- 搜索时只允许原地低速扫描。

## 7. 避障与主动寻找路线

障碍处理不应孤立成“看到障碍就停”的模块，而应与目标方向、身份置信度、距离状态共同决定行为。推荐路线是从保守可控的本地策略开始，按手机性能和实测效果逐步增强。

### 7.1 Level 0：安全停车底线

无论后续加入多少智能策略，底线始终是：

```text
身份不确定，不前进
目标丢失，不前进
中心走廊高风险，不前进
通信异常或急停，hard STOP
```

这个层级必须在不依赖 ReID、深度模型、云端服务器的情况下成立。

### 7.2 Level 1：手机端三方向可通行空间

首版最值得实现的避障能力是 `LEFT / CENTER / RIGHT` 三方向评分：

```text
left_free_score
center_free_score
right_free_score

left_risk
center_risk
right_risk
traversability_confidence
```

它可以先基于较轻量的线索实现：

- person detector 的目标框与非目标人框；
- 画面下半部分的区域占用；
- bbox bottom、面积变化、边缘接触；
- 目标方向与最后运动趋势；
- 最近若干帧的稳定性。

该层的目标不是完美避障，而是把“只会原地停”升级为“知道哪边更可能安全”。

### 7.3 Level 2：动作走廊与局部避障

在三方向评分稳定后，引入动作走廊：

| 动作 | 关注区域 |
| --- | --- |
| `SLOW_FORWARD` | 中间前方走廊 |
| `LEFT_ARC` | 左前方弧形走廊 |
| `RIGHT_ARC` | 右前方弧形走廊 |
| `ROTATE_LEFT` | 左侧近距离旋转区域 |
| `ROTATE_RIGHT` | 右侧近距离旋转区域 |

决策层不再问“画面里有没有障碍”，而是问：

```text
如果执行这个低速动作，未来短时间内是否可能进入高风险区域？
```

这能自然支持如下行为：

- 目标在左前方，中心受阻，左侧较空：低速左弧线；
- 目标在右前方，中心受阻，右侧较空：低速右弧线；
- 目标不可见且场景不确定：`BLOCKED_WAIT` 或 `LOCAL_SEARCH`；
- 非目标行人横穿中心：停止等待，不主动抢行。

### 7.4 Level 3：单目深度与动态风险增强

如果手机性能允许，再将单目深度和光流 / TTC 近似作为增强线索。

单目深度推荐用途：

```text
target_depth_proxy
left_depth_clearance
center_depth_clearance
right_depth_clearance
near_risk_map
depth_confidence
```

动态风险推荐用途：

```text
left_motion_risk
center_motion_risk
right_motion_risk
time_to_collision_proxy
```

注意：这些能力只提高风险判断质量，不直接绕过安全状态机。若深度、bbox、检测框之间互相矛盾，应降低整体置信度，而不是让某一个模型单独决定动作。

### 7.5 Level 4：云端或更高算力增强

云端服务器可以作为“看远一点”的后续方向，但必须明确边界：

- 可以用于离线日志分析、模型效果对比、参数搜索、复杂场景复盘；
- 可以用于演示模式下的高质量语义理解或场景标注；
- 不应作为急停、近距离避障、实时身份确认的唯一依据；
- 网络延迟、断连、隐私合规和图像上传成本都会影响工程可靠性。

如果未来确实接入云端，推荐架构是：

```text
手机端实时安全闭环
  +
云端非实时分析 / 策略建议 / 日志复盘
```

而不是：

```text
手机端等待云端决定是否停车
```

### 7.6 外接传感器的关系

外接 ToF、超声波、碰撞开关不取代手机主脑，只作为近距离安全冗余：

| 传感器 | 用途 |
| --- | --- |
| ToF | 近距离障碍停车 |
| 超声波 | 前方距离阈值判断 |
| 碰撞开关 | 最后物理防线 |

这些信号进入 `SystemSafetyEvidence`，并在优先级上高于普通视觉绕行建议。

---

## 8. 安全状态机升级

状态机的核心不再是简单的 `FOLLOW / LOST / STOP`。但第一版也不应把每一种局部动作都做成长期状态，否则状态转移会迅速复杂化。

第一版推荐拆成两层：

```text
FollowStateMachine：管理身份与任务阶段
ActionArbitrator：管理本帧动作选择
```

### 8.1 FollowStateMachine 核心状态

第一版状态层建议保留：

```text
IDLE
CAPTURE_TARGET
LOCKED_PENDING_CONFIRM
CONFIRMED_ARMED
REACQUIRE_TARGET
READY_TO_FOLLOW
FOLLOW_CONFIDENT
FOLLOW_CAUTION
IDENTITY_UNCERTAIN
LOST
REACQUIRE_TARGET
STOP
EMERGENCY
```

其中：

| 状态 | 说明 | 运动策略 |
| --- | --- | --- |
| `FOLLOW_CONFIDENT` | 身份明确、距离正常、安全风险低 | 允许低速跟随 |
| `FOLLOW_CAUTION` | 身份基本明确，但距离、空间或风险判断置信度下降 | 限速 / 减速 |
| `IDENTITY_UNCERTAIN` | 多人分数接近、ReID 不确定、遮挡严重 | `motion_stop`，不切换目标，不继续前进 |
| `LOST` | 目标暂时丢失 | 立即取消线速度，停止前进 |
| `REACQUIRE_TARGET` | 重新看到疑似目标 | 多帧 ReID + bbox + prediction 稳定后恢复 FOLLOW |
| `STOP` | 搜索失败、风险过高或安全异常 | hard STOP，等待重新初始化或人工确认 |
| `EMERGENCY` | 急停或严重异常 | 停止，人工复位 |

`TARGET_OCCLUDED` 可以先不作为独立长期状态，而是作为 `LOST / IDENTITY_UNCERTAIN / REACQUIRE_TARGET` 中的 `action_reason` 或 `safety_block_reason`。等真实场景中确实需要单独处理遮挡等待，再升级为状态。

---

### 8.2 BehaviorAction 动作层

动作层表达本帧实际要做什么：

| 动作 | 含义 |
| --- | --- |
| `FOLLOW_SLOW` | 低速前进跟随 |
| `FOLLOW_CAUTION` | 更低速谨慎跟随 |
| `MOTION_STOP` | 线速度清零，但不进入 hard STOP |
| `LOCAL_SEARCH_LEFT` | 原地低速左转搜索 |
| `LOCAL_SEARCH_RIGHT` | 原地低速右转搜索 |
| `LEFT_ARC` | 目标与左侧空间都支持时低速左弧线 |
| `RIGHT_ARC` | 目标与右侧空间都支持时低速右弧线 |
| `BLOCKED_WAIT` | 空间风险高或信息矛盾，停止等待 |
| `REACQUIRE_HOLD` | 疑似目标出现，暂停并多帧确认 |
| `HARD_STOP` | 搜索失败或需要人工重新确认 |
| `EMERGENCY_STOP` | 急停或严重安全异常 |

这样 `AVOID_LEFT / AVOID_RIGHT / BLOCKED_WAIT` 不必一开始都进入状态机，而是先作为动作层结果。状态机负责“现在处于什么任务阶段”，动作仲裁器负责“这一帧怎么动”。

### 8.3 安全优先级

第一版动作仲裁的硬优先级建议为：

```text
EMERGENCY_STOP
  > HARD_STOP
  > BLOCKED_WAIT
  > MOTION_STOP / REACQUIRE_HOLD
  > LOCAL_SEARCH_LEFT / LOCAL_SEARCH_RIGHT
  > LEFT_ARC / RIGHT_ARC
  > FOLLOW_CAUTION
  > FOLLOW_SLOW
```

只要身份、空间或系统安全证据不可靠，`ControlGenerator` 就不允许输出前进线速度。允许的主动性主要来自两个方向：

1. 线速度为 0 的原地搜索；
2. 身份明确、空间可通行且低速限幅下的局部弧线动作。

### 8.4 状态机与避障的统一

`LOCAL_SEARCH_LEFT / RIGHT`、`BLOCKED_WAIT`、`REACQUIRE_HOLD` 不应被看成 ReID 专属逻辑，也不应被看成避障专属逻辑。它们都是不确定状态下的安全行为。

典型例子：

| 场景 | 推荐状态 | 推荐动作 |
| --- | --- |
| 目标从画面左侧消失，周围空间安全 | `LOST` | `LOCAL_SEARCH_LEFT` |
| 目标被非目标行人挡住 | `LOST` 或 `IDENTITY_UNCERTAIN` | `BLOCKED_WAIT` |
| 中间有障碍，目标在左侧且左侧可通行 | `FOLLOW_CAUTION` | `LEFT_ARC` |
| 目标重新出现但身份分数一般 | `REACQUIRE_TARGET` | `REACQUIRE_HOLD` |
| 场景风险高、通信异常或搜索超时 | `STOP` / `EMERGENCY` | `HARD_STOP` / `EMERGENCY_STOP` |

这也是后续让小车“更积极”而不牺牲安全的关键。

## 9. Human Cart Simulator 升级

Human Cart Simulator 是下一阶段最重要的验证平台。它不只是“无底盘提示器”，而应成为行为决策层的可视化调试台。

### 9.1 显示内容

在原有目标框、状态和控制量基础上，显示证据、状态、动作和原因：

```text
identity_score
reid_score / reid_margin
tracking_score
distance_state / distance_score
left_free_score / center_free_score / right_free_score
left_risk / center_risk / right_risk
behavior_state
selected_action
action_reason
safety_block_reason
```

屏幕上应显示：

- 当前状态；
- 当前目标框；
- 普通行人框；
- ReID / match 分数与是否启用 ReID；
- 跟随距离状态；
- 左 / 中 / 右可通行评分；
- 大号动作提示；
- FPS / 推理耗时 / 是否降级；
- 当前动作为什么被选择；
- 哪条安全规则阻止了前进。

---

### 9.2 动作提示语义

大号提示优先级：

| 状态 | 提示 |
| --- | --- |
| 身份未确认 | `目标未确认，请不要移动` |
| 身份不确定 | `目标不确定，请停止` |
| 目标太远且安全 | `向前慢走` |
| 目标偏左且左侧可通行 | `向前并左转` 或 `左弧线` |
| 目标偏右且右侧可通行 | `向前并右转` 或 `右弧线` |
| 目标太近 | `停止` |
| 中间受阻 | `前方受阻，等待` 或 `低速绕行` |
| 目标丢失 | `目标丢失，停止前进` |
| 搜索 | `原地左转搜索` / `原地右转搜索` |
| 目标重现 | `疑似目标，等待确认` |
| hard STOP | `已安全停止，请人工确认` |

---

### 9.3 Human Cart Simulator 验收场景

| 场景 | 观察重点 |
| --- | --- |
| 单人远离 | 是否提示向前 |
| 单人靠近 | 是否提示停止 |
| 目标偏左 / 偏右 | 转向提示是否正确 |
| 路人穿过 | 是否保持目标，不切换 |
| 目标蹲下 | 是否降低置信度或进入 caution |
| 目标被遮挡 | 是否停止前进并进入 LOST |
| 目标重新出现 | 是否用 ReID / 颜色 / 运动重新识别 |
| 画面前方出现近物 | 是否进入 `BLOCKED_WAIT` 或低速绕行 |
| 目标从左侧离开 | 是否优先左侧 `LOCAL_SEARCH` |
| 目标绕过障碍 | 是否在可通行方向给出低速弧线动作 |
| 多人接近且分数接近 | 是否停止前进而不是切换目标 |

## 10. 控制输出策略

### 10.1 OpenBot 兼容输出

首版仍保持 OpenBot 兼容控制方式：

- 上位机只输出 `Control(left, right)`；
- Android 端不处理四驱、麦轮和 AT8236 细节；
- 下位机负责左右轮到四电机的映射。

---

### 10.2 控制输入

`ControlGenerator` 不应只接收目标 bbox，而应接收综合感知结果：

```text
BehaviorAction
TargetMotionHint
DistanceEvidence
TraversabilityEvidence
SystemSafetyEvidence
```

---

### 10.3 动作到控制的映射

建议先把高层动作映射为 OpenBot 兼容控制量：

```text
FOLLOW_SLOW:
    Control(+v_low, +v_low)

FOLLOW_CAUTION:
    Control(+v_very_low, +v_very_low)

LEFT_ARC:
    Control(+v_inner, +v_outer)

RIGHT_ARC:
    Control(+v_outer, +v_inner)

LOCAL_SEARCH_LEFT:
    Control(-turn_low, +turn_low)

LOCAL_SEARCH_RIGHT:
    Control(+turn_low, -turn_low)

MOTION_STOP / BLOCKED_WAIT / REACQUIRE_HOLD / HARD_STOP / EMERGENCY_STOP:
    Control(0, 0)
```

其中 `LEFT_ARC / RIGHT_ARC` 必须满足：

- 目标身份稳定；
- 对应方向可通行评分足够高；
- 中心走廊风险不要求继续前进；
- 动作速度处于 very low speed；
- 任何一帧出现高置信度近距离风险，立即回到 `MOTION_STOP` 或 `BLOCKED_WAIT`。

### 10.4 控制规则

```text
if emergency:
    Control(0, 0)

else if hard_stop:
    Control(0, 0)

else if identity_uncertain:
    Control(0, 0)

else if blocked_wait:
    Control(0, 0)

else if search:
    Control(±turn_low, ∓turn_low)

else if avoid_left / avoid_right:
    低速弧线，持续检查对应方向风险

else if follow_confident / follow_caution:
    根据距离状态计算 forward
    根据目标中心偏差计算 turn
    限幅后输出 left/right
```

控制层不负责决定是否“值得冒险”。这个判断必须在 `BehaviorDecision / SafetyFusion` 完成。

### 阶段 A：Human Cart Simulator 行为决策层

目标：

- 在 Android 端建立 `Evidence → BehaviorDecisionResult → BehaviorAction → HumanCommand` 的最小闭环；
- 第一轮不接真实 ReID、不接单目深度、不做复杂避障；
- 先用现有检测、目标匹配和启发式默认值填充证据；
- 在 Human Cart Simulator 中实时显示动作、原因、阻止前进的安全规则。

#### A1：先定义数据结构，不改变行为

先新增或整理这些结构：

```text
IdentityEvidence
DistanceEvidence
TraversabilityEvidence
SystemSafetyEvidence
BehaviorAction
BehaviorDecisionResult
```

最小字段可以先这样填：

```text
identity_confidence = 当前 TargetMatcher 分数或跟踪置信度
distance_state = UNKNOWN
center_free_score = 1.0
left_free_score = 1.0
right_free_score = 1.0
system_safety = ok
selected_action = 由旧 ControlGenerator 或旧提示逻辑推导
```

目标是先让日志和 UI 能看到统一结构，而不是立刻改变车辆行为。

#### A2：实现 ActionArbitrator 的硬优先级

先只实现最硬的规则：

```text
emergency              → EMERGENCY_STOP
hard_stop_required     → HARD_STOP
identity_uncertain     → MOTION_STOP
target_lost_left       → LOCAL_SEARCH_LEFT
target_lost_right      → LOCAL_SEARCH_RIGHT
target_lost_unknown    → MOTION_STOP
center_blocked         → BLOCKED_WAIT
else                   → FOLLOW_SLOW / FOLLOW_CAUTION
```

这一版不追求聪明，只追求链路清楚、优先级稳定、不会在不确定时继续前进。

#### A3：Human Cart Simulator 显示决策原因

每一帧至少显示：

```text
selected_action
action_reason
safety_block_reason
```

示例：

```text
selected_action = MOTION_STOP
action_reason = identity_margin_too_low

selected_action = LOCAL_SEARCH_LEFT
action_reason = target_lost_left_side

selected_action = FOLLOW_SLOW
action_reason = identity_ok_distance_far_center_clear
```

这是后续调参和排查“为什么小车不动 / 为什么小车想动”的关键。

#### A4：再接 DistanceEvidence 和 TraversabilityEvidence

最后再把距离状态和三方向可通行评分接进来：

- `DistanceEvidence` 先输出 `TOO_FAR / OK / TOO_CLOSE / UNKNOWN`；
- `TraversabilityEvidence` 先输出 `LEFT / CENTER / RIGHT` 的简版 free score；
- 只有当身份可信、距离需要前进、对应方向可通行时，才允许 `FOLLOW_SLOW / LEFT_ARC / RIGHT_ARC`。

验收重点：

- 目标丢失后不继续前进；
- 目标返回后不必直接 hard STOP，而是能进入 `REACQUIRE`；
- 多人混淆时不切换目标；
- 前方风险升高时停止或等待；
- 动作原因在 UI 上可解释。

---

### 阶段 B：Android 端 ReID 作为行为证据接入

目标：

- 接入轻量 ReID 推理，优先使用已经在 PC 端验证过的 `osnet_x0_25` 路线；
- ReID 低频或事件触发运行，不强行每帧推理；
- 将 `reid_score / reid_margin / gallery_stability` 写入 `IdentityEvidence`；
- 与 bbox 连续性、颜色、尺寸、检测置信度融合，而不是替代它们。

实现顺序：

1. 先保证阶段 A 的 `BehaviorAction + Evidence + UI` 已经可运行；
2. 再接 `ReIDFeatureExtractor`，先只打印推理耗时和分数；
3. 再把 `reid_score / reid_margin` 放入 `IdentityEvidence`；
4. 最后观察 `ActionArbitrator` 是否正确响应身份不确定、多人接近和重捕获场景。

首版 ReID 只建议在三个场景运行：

| 场景 | ReID 用途 |
| --- | --- |
| 初始化确认后 | 提取目标 gallery |
| 多人接近或候选分数接近 | 低频计算候选 ReID，帮助保持目标 |
| `REACQUIRE_TARGET` | 计算疑似目标 ReID，辅助多帧重捕获 |

不建议第一版每帧跑 ReID。每帧 ReID 容易压垮手机性能，也容易让系统误以为“分数就是决策”。

验收重点：

- ReID 接入后手机 FPS 和延迟仍可接受；
- 多人接近时身份分数能帮助保持目标；
- 目标缺席时不会因单帧高分误恢复；
- 目标重现时能在 `motion_stop` 边界内更积极重捕获。

---

### 阶段 C：目标轨迹与身份信念层改造

目标：

- 在 Android Human Cart Simulator 中新增轻量 `TargetTrackManager`，将连续 person bbox 关联为短时 track；
- 在 ReID 与状态机之间新增 `IdentityBeliefAccumulator`，对每个 track 累计目标身份信念；
- 将恢复跟随的条件从“单帧候选满足 ReID / bbox gate”升级为“稳定 track + 稳定 belief + 多帧证据”；
- 在 debug 面板显示 `trackId / lockedTrackId / targetBelief / trackAge / missedFrames / beliefReason`。

首版边界：

- 不做复杂多目标跟踪，不引入 DeepSORT / ByteTrack 等完整跟踪框架；
- 不做 dynamic gallery 更新，避免错误样本污染目标记忆；
- 不改变真实底盘控制输出，仍先在 Human Cart Simulator 中验证；
- ReID 继续作为身份强证据之一，但不能单独恢复 `FOLLOW`。

验收重点：

- 单人正常跟随时，锁定目标的 `trackId` 应稳定，`targetBelief` 应逐步升高；
- 目标离开后，系统进入 `motion_stop / LOCAL_SEARCH / IDENTITY_UNCERTAIN`，不继续前进；
- 干扰者进入时可以形成新 track，但不能凭一帧 ReID 高分抢走 locked target；
- 目标返回后，应先成为 suspected track，再经多帧 belief 稳定进入 `REACQUIRE_TARGET -> FOLLOW_CAUTION / FOLLOW_CONFIDENT`；
- debug 面板必须能解释为什么恢复、为什么停止、为什么不切换目标。

---

### 阶段 D：距离控制主线收敛

目标：

- 将距离控制收敛到“初始化标定 + 图像伺服”；
- 保存 `desired_bbox_height_ratio / desired_bbox_area_ratio / desired_bbox_bottom_ratio`；
- 输出 `TOO_FAR / OK / TOO_CLOSE / UNKNOWN`；
- 将距离证据交给行为决策层，而不是直接决定前进速度。

验收重点：

- 目标太近时优先停车；
- 目标太远但身份和空间安全时才允许慢速前进；
- 中心受阻时距离再远也不能盲目前进。

---

### 阶段 E：三方向可通行空间简版

目标：

- 先不追求复杂语义分割；
- 基于检测框、画面下半部区域、目标方向和历史稳定性输出 `LEFT / CENTER / RIGHT` free score；
- 在 Human Cart Simulator 中显示三方向评分、dominant risk source 和 suggested action。

验收重点：

- 非目标行人进入中心走廊时能降低 `center_free_score`；
- 目标在侧边且对应方向安全时能建议低速弧线；
- 信息矛盾时进入 `BLOCKED_WAIT`，不强行绕行。

---

### 阶段 F：性能评估与增强开关

目标：

- 记录 person detector、ReID、三方向评分、控制决策的耗时；
- 根据真实手机性能决定是否启用更高频 ReID、单目深度、光流 / TTC；
- 将所有增强能力做成可关闭开关，并保留降级路径。

验收重点：

- 关闭 ReID / 深度后系统仍能安全降级；
- 性能下降时优先降频，不牺牲安全状态机；
- UI 显示当前是否处于 degraded mode。

---

### 阶段 G：低速真实底盘联调

目标：

- 将 `Control(left,right)` 接到底盘
- 验证真实车速、转向半径、延迟和惯性
- 重新标定跟随距离与速度上限
- 先只开放 `FOLLOW_SLOW / MOTION_STOP / LOCAL_SEARCH / BLOCKED_WAIT`
- 再逐步开放 `LEFT_ARC / RIGHT_ARC`
- 接入 ToF / 超声波等安全冗余（若视觉不足）

验收重点：

- 任意不确定状态线速度清零；
- 目标丢失后不继续前进；
- 局部绕行动作足够慢，且可被任意高风险信号打断；
- 急停和通信异常优先级最高。

### 阶段 H：高级避障与云端增强评估

该阶段不是首版必做项，只在基础闭环稳定且手机性能或网络条件允许时启动。

可评估方向：

- 单目深度风险热力图；
- 光流 / TTC 动态接近风险；
- 更细粒度动作走廊；
- 云端离线日志复盘和参数推荐；
- 云端复杂场景标注或模型评估。

边界：

- 不让云端参与急停和近距离实时避障；
- 不把高级模型设为首版必需依赖；
- 所有增强必须能被关闭，并回退到本地安全闭环。

## 12. 首版验收口径

### 12.1 无底盘阶段验收

| 验收项 | 标准 |
| --- | --- |
| 目标初始化 | 完成采集、截图、确认、重识别、倒计时 |
| 目标身份保持 | 多人干扰时不轻易切换目标 |
| 目标重识别 | 短时遮挡后能重新匹配 |
| 距离控制 | 能显示并响应 `TOO_FAR / OK / TOO_CLOSE` |
| Human Cart Simulator | 能实时提示前进、转向、停止、搜索 |

## 13. 风险与降级方案

| 风险 | 应对 | 降级 |
| --- | --- | --- |
| ReID 模型部署失败 | 保留接口，先用颜色 + 运动匹配 | 不影响首版运行 |
| ReID 导致手机性能下降 | 事件触发或低频运行，必要时关闭 | 回退到 bbox + 颜色 + 运动连续性 |
| 单目深度太慢 | 低频运行或关闭 | 退化为 bbox 距离控制 |
| ARCore 不可用 | 不作为主线 | 不影响设计 |
| bbox 距离估计不稳 | 加入标定、平滑和深度辅助 | 降速 / 停止 |
| 手机视觉障碍判断不可靠 | 降级为 `BLOCKED_WAIT`，不做绕行 | 外接 ToF / 超声波 |
| 三方向可通行评分误判 | 绕行动作限速并持续复评 | 只保留停止/等待 |
| 云端链路延迟或断连 | 云端只做非实时分析 | 手机端本地安全闭环独立运行 |
| 图像上传带来隐私风险 | 默认不上传实时画面 | 只上传脱敏日志或离线样例 |
| 身份分数接近 | 进入 `IDENTITY_UNCERTAIN` | 停车并提示重新确认 |
| 目标被长时间遮挡 | 搜索超时 | STOP 等待人工干预 |
| `tools/` 中直接 clone 上游仓库 | 移除内部 `.git` 后转为普通目录，避免嵌套仓库干扰协作 | 后续补充来源说明并忽略大文件产物 |
| ReID 研究目录含私人照片 | 默认不提交 `images/` 与 `outputs/`，并在仓库规则中显式忽略 | 仅保留代码、文档和必要说明 |

## 14. 当前最建议的下一步

阶段 A 的行为层闭环和阶段 B 的 Android TFLite ReID 首版接入已经完成实机验证。当前不建议继续单纯调高 `bestScore / margin` 阈值，也不建议直接进入真实底盘控制。

下一步主线应是阶段 C：

```text
Detector persons
  ↓
TargetTrackManager
  ↓
ReIDCoordinator
  ↓
IdentityBeliefAccumulator
  ↓
FollowStateMachine
  ↓
ActionArbitrator
  ↓
HumanCommand
```

目标是解决两个实机暴露的问题：

| 问题 | 阶段 C 的解决方向 |
| --- | --- |
| 目标离开后偶尔跟错干扰者 | locked track 保护 + belief 累计 + 候选切换惩罚 |
| 目标返回后重捕获慢或无法确认 | suspected track + 多帧中等 ReID / bbox / prediction 证据恢复 |

阶段 C 的最小验收标准：

| 场景 | 预期 |
| --- | --- |
| 单人正常跟随 | `trackId` 稳定，`targetBelief` 上升，action 为 `FOLLOW_SLOW / FOLLOW_CAUTION` |
| 目标离开 | 取消线速度，进入 `LOCAL_SEARCH / IDENTITY_UNCERTAIN`，不 hard STOP 过早 |
| 目标离开后干扰者进入 | 干扰者可形成新 track，但不能快速获得恢复 `FOLLOW` 的 belief |
| 目标返回 | 先标记 suspected track，经连续稳定后恢复 `REACQUIRE_TARGET -> FOLLOW` |
| 目标在场且干扰者穿越 | locked target 不被一帧高分抢走，必要时进入 `FOLLOW_CAUTION / IDENTITY_UNCERTAIN` |

阶段 C 完成后，再继续距离控制收敛、三方向可通行空间、性能增强和真实底盘联调。

## 15. 结论

上位机方向的核心应从“能不能识别人”升级为：

> 能否把身份、距离、局部空间、系统安全和历史状态融合起来，安全但主动地决定购物车下一步动作。

首版最重要的不是追求复杂避障或云端智能，而是建立一个可靠的行为决策框架：

```text
证据可解释
状态可观察
动作可限速
不确定不前进
停止后仍能本地搜索和重捕获
风险升高立即 hard STOP
```

在这个框架下，ReID、单目深度、光流、外接传感器和云端分析都可以逐步成为增强证据；即使其中任一模块不可用，系统仍应保留本地、低速、安全、可演示的跟随闭环。

### 16.2 当前实机暴露的问题

实机反馈表明，当前系统已经能用 ReID 作为身份线索，但还不能保证“稳定跟住同一个目标人”：

1. 目标离开或多人靠近时，系统仍可能把干扰者识别为目标，并出现跟错人的风险。
2. 目标重新回到画面时，重捕获有时偏慢，甚至无法及时确认目标。
3. ReID 分数、margin 与 bbox 连续性都有效，但单帧证据仍不足以承担“切换目标 / 恢复跟随”的最终决定。

因此，下一阶段不应继续只调 `bestScore / margin` 阈值，而应把系统从“每帧选最像的人”升级为“持续维护目标轨迹和身份信念”。

### 16.3 下一步核心方向：TargetTrack + IdentityBelief

建议在 ReID 与状态机之间新增轻量目标轨迹与身份信念层：

```text
Detector persons
  -> TargetTrackManager
  -> IdentityBeliefAccumulator
  -> FollowStateMachine
  -> ActionArbitrator
  -> Human command
```

`TargetTrackManager` 的职责：

- 将连续多帧中位置、尺寸、运动方向相近的 person bbox 归并为短时 track。
- 给每个 track 分配临时 `trackId`。
- 记录最近 bbox、速度方向、丢失帧数、是否贴边、是否位于最后目标预测区域。

`IdentityBeliefAccumulator` 的职责：

- 对每个 track 维护 `target_belief`，而不是只看单帧 ReID 分数。
- ReID strong/mid、bbox 连续、预测区域命中、与最后目标位置一致时加分。
- 候选切换、bbox 大跳变、面积突变、ReID margin 低、多人混淆时扣分。
- 只有 belief 连续超过阈值，才允许进入 `REACQUIRE_TARGET` 并最终恢复 `FOLLOW`。

关键原则：

- 已锁定目标时，不因干扰者一帧 ReID 高分而切换目标。
- 目标丢失后，可以更积极地原地搜索和提高 ReID 频率，但线速度保持 0。
- 目标返回后，允许先标记为“疑似目标”，再通过多帧稳定证据恢复跟随。
- `FOLLOW`、`FOLLOW_CAUTION`、`IDENTITY_UNCERTAIN`、`REACQUIRE_TARGET` 应由累计信念驱动，而不是由单帧候选直接驱动。

### 16.4 后续优先级

当前建议的开发顺序调整为：

1. 在 Human Cart Simulator 中实现 `TargetTrackManager`，先只做 bbox IoU / center distance / area ratio 的短时关联。
2. 增加 `IdentityBelief`，把 ReID、bbox 连续性、prediction、候选切换次数累积为稳定分数。
3. 改造 `FollowStateMachine` 的恢复逻辑，使 `LOST / SEARCH / IDENTITY_UNCERTAIN` 只接受“稳定 track + 稳定 belief”的重捕获证据。
4. 在 debug 面板显示 `trackId / targetBelief / trackAge / missedFrames / beliefReason`。
5. 实机重复验证：目标离开、干扰者进入、目标返回、目标在场干扰者穿越、遮挡。

只有当上述链路稳定后，再考虑进一步压榨 ReID 频率、加入更复杂局部避障或云端离线分析。

### 16.6 阶段 C 当前暂停点：先做诊断采集，不继续盲目调策略

最新手机测试反馈显示，阶段 C 首版仍存在两个现象：

1. 目标离开后重新回到画面时，目标经常长期保持黄框，迟迟不能重新转为绿框。
2. 少数场景下，非目标人物仍可能变成绿框。

这说明当前问题已经不能只靠“再调高或调低某个阈值”解决。继续直接改 `targetBelief`、`reid_score` 或 relock 条件，容易在两个风险之间摇摆：

```text
放松恢复条件 -> 目标回来更快转绿，但干扰者更容易抢走目标；
收紧恢复条件 -> 干扰者更难抢走目标，但真实目标回来后长期黄框。
```

因此，阶段 C 的下一步应先转为**诊断采集与可复盘调试**，而不是继续叠加策略规则。

|---|---:|---:|---:|---:|---:|---:|---:|

阶段判断：

新数据中主要 blocker 为：

```text

belief_high_bbox_failed: 10
```

这意味着下一步策略修正应聚焦：

1. **Track association 与 locked target 保护。**  
   目标离开、返回、干扰者穿越时，不能让干扰者因为短时 ReID 高分或 bbox 接近就快速抢走 locked target。需要检查 `TargetTrackManager` 的 trackId 关联、missed frame 保留、locked track 优先级和 suspected track 迁移条件。

2. **分状态 bbox gate。**  
   `FOLLOW` 状态可以继续使用严格 bbox / prediction gate；但 `REACQUIRE_TARGET / IDENTITY_UNCERTAIN / SEARCH` 中，目标重新出现时天然会有更大的中心跳变和面积变化。如果这些状态仍使用同样严格的 gate，就会出现“belief 已经高，但黄框不转绿”。下一轮应为恢复状态设计更宽的候选 gate，再通过连续多帧稳定恢复安全性。

3. **Recoverable stop 与 hard STOP 区分。**  
   目标丢失后的 `motion_stop / SEARCH / IDENTITY_UNCERTAIN` 应继续保留重捕获机会；只有搜索超时、障碍风险、通信异常、急停或人工取消时才进入 hard `STOP`。这能避免“目标回来但系统已经不可恢复”的情况。

4. **诊断指标驱动调参。**  
   每次策略修改后继续采集 `cartfollow_diagnostics`，用 `diagnostic_return_comparison.csv` 对比 `recovered_rate / mean_ms_to_follow / candidate_switch_penalty / belief_high_bbox_failed / not_recovered_in_window`，而不是只凭肉眼看框颜色。

下一轮不建议优先做：

- 换更大的 ReID 模型；
- 继续只调 `bestScore / margin`；
- 一次性放宽所有 bbox gate；
- 直接接真实底盘验证跟随。

更稳妥的开发顺序是：

```text
Android track/bbox gate 策略小步修改
  -> Human Cart Simulator 复测
  -> 导出 cartfollow_diagnostics
  -> PC compare 分析
  -> 确认没有增加非目标转绿
  -> 再考虑接近真实底盘控制链路
```

阶段 C 后续的合格标准应改为：

| 场景 | 期望 |
|---|---|
| 目标返回 | 不长期卡黄框，能在 motion_stop 边界内进入 `REACQUIRE_TARGET / FOLLOW_CAUTION` |
| 干扰者进入 | 可形成新 track，但不能快速抢走 locked target |
| 目标在场且干扰者穿越 | 不因一帧高 ReID 分数切换目标 |
| 长时间无人 | 先 `SEARCH / IDENTITY_UNCERTAIN`，超时后 hard `STOP` |
| 策略修改后 | `candidate_switch_penalty` 与 `belief_high_bbox_failed` 应下降，非目标转绿不能增加 |

### 16.8 下一轮策略版本：track 粘性与分状态 bbox gate

本轮策略版本只处理 `candidate_switch_penalty` 和 `belief_high_bbox_failed` 两个 blocker，不扩大功能范围。默认继续使用 `osnet_x0_25 + TFLite + upright crop`，不更换模型，不启用 dynamic gallery，不全局放宽 bbox gate，不直接接真实底盘前进，也不把复杂避障提前为当前主线。

核心修改口径：

```text
loose admission
  -> suspectedTrack
  -> default confirm
  -> REACQUIRE_TARGET
  -> strict/default motion gate
  -> FOLLOW_CAUTION / FOLLOW
```

其中：

- `loose admission` 只表示“可以保留为疑似目标”，不得直接允许前进。
- `motion gate` 仍由 default / strict bbox gate、prediction、locked target 保护和多帧稳定证据共同决定。
- `belief 高但 bbox failed` 时，应解释为 `identity likely, motion not safe yet`，动作保持 `REACQUIRE_HOLD / MOTION_STOP`，不直接清空 suspected track。
- suspected track 一旦形成，需要最小滞回帧数；新候选只有在 belief 明显更高、连续稳定且旧 suspected 失效或过期时才能替换。
- locked track 在短时丢失后保留 ghost memory，用作恢复参考，但 ghost 本身不作为可运动目标。

进入真实底盘前的门槛同步调整为：

```text
非目标转绿不增加；
target_return recovered_rate 稳定不低于 0.85；
candidate_switch_penalty 明显下降；
belief_high_bbox_failed 明显下降；
MOTION_STOP / REACQUIRE_HOLD / BLOCKED_WAIT 行为稳定；
debug 能解释每次恢复、拒绝恢复和拒绝切换的原因。
```

复测仍使用 `cartfollow_diagnostics` 与 PC compare 脚本，不用肉眼观察绿框作为唯一判断依据。

### 16.9 2026-07-09 最新进展：relock、空间支持门控与日志开关已实现

截至 2026-07-09，本节 16.8 的下一轮策略已完成 Android 侧代码接入，当前状态从“待改策略”更新为“待安装新版 APK 后实测复盘”。本轮仍然不更换 ReID 模型、不启用 dynamic gallery、不全局放宽 bbox gate、不开放真实底盘前进，也不改 `ControlGenerator` 的真实底盘控制路径。

已完成的策略修正如下：

1. **恢复后 relock。**  
   目标离开后如果原目标以新的临时 `trackId` 返回，系统允许在安全恢复路径中把该 track 晋升为新的 `lockedTrackId`。晋升条件包括：非 locked track 连续通过 motion gate，并在 `FOLLOW_CAUTION / FOLLOW_SLOW` 等保守动作下稳定至少 2 帧；成功后同步调用 `IdentityBeliefAccumulator.lockTrack(newTrackId)`，清空旧 suspected，并在 debug reason 中输出 `relock_after_recovery`。

2. **非 locked track 空间支持门控。**  
   非 locked 候选不能只凭 ReID 高分进入 suspected。它必须同时满足 `bboxLoose / bboxDefault / prediction / nearLockedGhost` 中至少一个空间支持条件。若只有 ReID 高分但没有空间支持，则记录 `reid_interest_no_spatial_support` 或 `spatial_support_missing`，belief 不允许升到 caution / confirm 阈值，也不能触发 FOLLOW。

3. **identity belief 与 motion permission 继续分离。**  
   ReID 只作为身份线索；`FOLLOW / FOLLOW_CAUTION` 的运动许可仍由 default / strict bbox gate、prediction 或 locked track 保护决定。belief 高但 motion gate 失败时保持 `REACQUIRE_HOLD / MOTION_STOP`，而不是直接前进。

4. **candidate switch 计数降噪。**  
   `candidateSwitchCount` 只在真正选中、切换 suspected 或 selected track 时增长，避免“画面中观察到高分干扰者”本身就造成 candidate switch penalty 噪声。

5. **Human Cart Simulator 诊断日志开关。**  
   Simulator 底部控制区新增“记录日志”开关，默认关闭。关闭时不创建 `cartfollow_diagnostics` session，不写 `frame_log.csv / identity_log.csv / events.csv / session_info.json`，也不保存 crop、gallery 和 session_stop；开启后保持原诊断采集行为。事件按钮只在日志开启且 Confirm 后 session 激活时可用。

下一步测试口径更新为：

```text
1. 安装最新 robot debug APK。
2. 需要采集诊断时，先打开 Human Cart Simulator 的“记录日志”开关；平时调试可保持关闭。
3. 复测四段：
   - 目标离开后原目标返回；
   - 目标离开后干扰者进入；
   - 目标在场时干扰者穿越；
   - 短遮挡、蹲下或局部可见后恢复。
4. 导出新版 cartfollow_diagnostics。
5. 用 tools/reid_pc_test/analyze_cartfollow_diagnostics_v1.py --compare-roots old=...,new=... 复盘。
6. 重点检查 candidate_switch_penalty、belief_high_bbox_failed、recovered_rate、mean_ms_to_follow、非目标转绿次数、hard_stop_count 和 gallery_candidate_landscape_rate。
```

进入极低速真实底盘联调前，仍需满足：原目标返回能 relock，干扰者不能仅靠高 ReID 抬成 suspected，两个 blocker 明显下降，`recovered_rate` 不下降，非目标转绿与 hard stop 不增加，debug reason 能解释恢复、拒绝恢复和拒绝切换原因。

### 17.1 当前状态

ReID、track/belief、距离状态和行为仲裁继续复用 Simulator 的公共管线，真实模块在

下位机新增 `ESP32 WROOM-32E -> UART -> AT8236` 的 OpenBot BLE 固件，兼容
`c/f/h`，并增加 BLE/USB 控制权、通信与运动双超时、AT8236 遥测门控和
`!S,<seq>` 锁存急停。

### 17.2 首版开放范围

- 手动：按住移动、松手停车，前进/后退/原地左右转，协议输出不超过 28。
- 自动：需要每次长按 2 秒实验解锁，最大输出 32，搜索旋转最多 2 秒。
- 连接：本轮优先 BLE；USB 保持兼容但不作为真机测试前置条件。
- 日志：继续由“记录日志”开关控制，默认关闭。

### 17.3 安全声明

近场超声波/ToF 和物理急停尚未到位。当前自动模式只能用于空旷、低速、有人准备
物理断电的受控实验，不能宣称已经具备可靠避障，也不能进入货架、拥挤区域或无人
看护运行。

### 17.4 下一步验收
