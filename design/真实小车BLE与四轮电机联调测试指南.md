# 真实小车 BLE 与四轮电机联调测试指南

## 1. 测试目的

本次测试要回答三个问题：

1. 手机是否按正确顺序发送了新方向命令？
2. ESP32 是否按相同顺序收到并输出了命令？
3. 四个电机是否能在相同命令下可靠起步、停止和反转？

测试期间只使用手动模式。**不要开启自动跟随。**

## 2. 安全要求

- 第一轮测试必须让四个车轮完全悬空。
- 小车周围不要放手、衣服、线缆或其他容易卷入车轮的物品。
- 旁边必须有人可以立即切断电机电源。
- 先确认 `c0,0` 或 USB 停车命令有效，再测试非零速度。
- 出现异常响声、剧烈抖动、失控转动或电机过热时，立即断电。
- 四轮悬空测试全部通过前，不允许落地测试。

## 3. 当前软件版本

截至 2026-07-12：

- 正式 BLE 固件已加入 USB 诊断和安全换向保护。
- 四轮 USB 标定固件位于：
  `firmware/esp32_at8236_calibration/esp32_at8236_calibration.ino`
- 正式 BLE 固件位于：
  `firmware/esp32_at8236_openbot_ble/esp32_at8236_openbot_ble.ino`
- Android 上位机提交：`e1c5238`。
- Android Debug APK：
  `dev/OpenBot/android/robot/build/outputs/apk/debug/robot-debug.apk`

两个 ESP32 固件不能同时使用。完成四轮标定后，必须重新烧录正式 BLE 固件，才能连接手机。

### 当前已经完成

- 正式 BLE 固件可以记录 Android 命令、四轮 target/current 和 `$spd`。
- 独立标定固件可以逐个测试 M1-M4，并输出原始 AT8236 遥测。
- 正式 BLE 固件可以阻止目标值直接反转，进入 `REVERSAL_BLOCKED` 后保持四轮零速。
- Android 可以记录触摸、pointer、generation、BLE 队列和 GATT 写入结果。

### 当前还没有完成

- 尚未通过实测确认 `$MSPD` 四个字段与 M1-M4 的对应关系。
- 尚未得到四个电机正反转的最低可靠速度。
- 正式 BLE 固件尚未根据真实 `$MSPD` 自动判断四轮已经停稳。
- 当前换向保护使用人工松键并等待 1 秒的保守流程。
- 真机悬空和落地验收尚未完成，自动跟随仍不能开放。

## 4. 测试工具

准备以下工具：

- 装有新版 OpenBot APK 的 Android 手机；
- USB 数据线；
- Arduino IDE 串口监视器，波特率 `115200`；
- 能运行 `adb` 的电脑终端；
- 用于记录结果的表格或文本文件；
- 可立即断开电机电源的人员或开关。

串口监视器发送命令时选择“新行”或确保命令末尾带换行。

## 5. 第一阶段：四轮单独标定

### 5.1 烧录标定固件

烧录：

```text
firmware/esp32_at8236_calibration/esp32_at8236_calibration.ino
```

该固件不启动 WiFi 和 BLE，只能通过 USB 测试。

烧录后：

1. 打开串口监视器，设置为 `115200`。
2. 确认四个车轮完全悬空。
3. 确认串口中持续出现 AT8236 遥测，或发送 `!Q` 查看状态。

### 5.2 保存静止遥测

发送：

```text
!U,50
```

保存至少 5 秒日志。正常情况下会看到 `$MSPD`、`$MTEP` 或 `$MAll` 帧。

如果没有遥测，不要继续测试电机。先检查：

- ESP32 GPIO17 是否连接 AT8236 RX2；
- ESP32 GPIO16 是否连接 AT8236 TX2；
- 两块板是否共地；
- 串口波特率是否为 `115200`。

### 5.3 解锁测试

发送：

```text
!A,RAISED
```

发送过 `!S,1000` 后，后续 `!T` 的序号必须大于 `1000`。

看到下面的回复才能继续：

```text
!OK,A
```

解锁有效期为 60 秒。超时后需要重新发送 `!A,RAISED`。

### 5.4 单电机测试命令

命令格式：

```text
!T,序号,电机编号,速度,持续时间毫秒
```

示例：让 M1 以速度 3 正转 500 ms：

```text
!T,1,1,3,500
```

让 M1 以速度 3 反转 500 ms：

```text
!T,2,1,-3,500
```

注意：

- 序号必须一直递增，不能重复。
- 电机编号只能是 `1、2、3、4`。
- 速度范围为 `-40..40`，不能为 0。
- 持续时间为 `100..1500 ms`。
- 每次只允许一个电机转动。
- 每次测试结束后至少等待 1 秒。

### 5.5 推荐测试顺序

对 M1、M2、M3、M4 分别测试以下速度：

```text
正转：3、5、8、12、14、20
反转：-3、-5、-8、-12、-14、-20
```

每次建议持续 `500 ms`。例如 M1 的前几条命令：

```text
!T,1,1,3,500
!T,2,1,-3,500
!T,3,1,5,500
!T,4,1,-5,500
```

后续命令继续增加序号。

每次记录：

| 电机 | 方向 | 命令速度 | 是否每次都能起转 | 起转是否明显延迟 | 停止是否拖延 | `$MSPD` 对应字段 |
|---|---|---:|---|---|---|---|
| M1 | 正转 | 3 |  |  |  |  |
| M1 | 反转 | -3 |  |  |  |  |

最低可靠速度是“连续多次都能正常起转”的最小速度，不是偶尔转动一次的速度。

### 5.6 紧急停车

需要立即停止时发送递增序号：

```text
!S,1000
```

该命令会停止四轮并取消解锁。再次测试前需要重新发送：

```text
!A,RAISED
```

### 5.7 第一阶段通过条件

第一阶段完成后，应得到：

- M1-M4 的正转最低可靠速度；
- M1-M4 的反转最低可靠速度；
- `$MSPD` 四个字段分别对应哪个电机；
- `$MSPD` 正负号是否与真实转向一致；
- 哪个问题跟随电机，哪个问题固定在 AT8236 通道。

如果某个问题很明显，可以交换电机与 AT8236 通道后重复测试：

- 问题跟着电机移动：优先检查电机、编码器和减速箱。
- 问题固定在通道：优先检查 AT8236 通道、接线和参数。

## 6. 第二阶段：手机 BLE 命令链测试

### 6.1 烧回正式 BLE 固件

烧录：

```text
firmware/esp32_at8236_openbot_ble/esp32_at8236_openbot_ble.ino
```

烧录后重新启动 ESP32。确认手机能够连接：

```text
OpenBot: CART_AT8236
```

OpenBot 页面应显示 BLE 和固件已经就绪。

### 6.2 开启两侧日志

ESP32 USB 串口发送：

```text
!D,1
```

Android 页面打开“记录日志”开关。

电脑运行：

```powershell
adb logcat -c
adb logcat -s CartControl:I
```

ESP32 串口应能看到：

```text
motion_rx
motion_target
drive_output
```

Android 终端应能看到：

```text
touch_down
queue_enqueue
queue_dispatch
gatt_success
```

### 6.3 基本方向测试

保持车轮悬空，分别短按：

1. 前进；
2. 松手；
3. 后退；
4. 松手；
5. 原地左转；
6. 松手；
7. 原地右转；
8. 松手。

每次松手后都应看到：

```text
c0,0
```

如果松手后仍有非零命令，立即断电并停止测试。

### 6.4 按钮切换测试

依次测试：

1. 松开“前进”后按“左转”；
2. 按住“前进”，用第二根手指按“左转”；
3. 单指从“前进”滑到“左转”；
4. 新方向生效后，再松开旧手指；
5. 左转切右转；
6. 前进切后退。

Android 中一次换向应按顺序出现：

```text
touch
queue_transition
queue_dispatch STOP
gatt_success c0,0
queue_dispatch 新方向
gatt_success 新方向
```

ESP32 中应按相同顺序出现：

```text
motion_rx c0,0
motion_rx 新方向
```

如果 Android 顺序正确、ESP32 顺序错误，检查 BLE 链路。

如果 ESP32 顺序正确、四个电机动作错误，进入电机和 AT8236 排查，不再修改 Android 按钮逻辑。

## 7. 安全换向保护测试

正式固件不会允许车轮直接反转。

例如先按前进，再按后退时，第一次反向命令应触发：

```text
REVERSAL_BLOCKED
```

此时四轮应立即保持零速。固件不会在后台自动继续后退。

正确操作步骤：

1. 松开所有方向按钮；
2. 保持不按任何方向至少 1 秒；
3. 再重新按下目标方向。

USB 日志应依次看到：

```text
reversal_block
reversal_neutral
reversal_rearmed
```

如果等待期间又收到非零命令，会看到：

```text
reversal_reject
```

此时需要重新松开按钮并重新等待 1 秒。

可以发送以下命令查看状态：

```text
!Q
```

重点检查：

```text
reversal_blocked
reversal_neutral_age_ms
last_nonzero_target
```

### 安全换向通过条件

- 触发反向时四轮都先停车；
- 等待期间没有任何车轮提前反转；
- 中立满 1 秒前不能执行新方向；
- 中立满 1 秒后必须重新按键才能启动；
- BLE 断开、急停或遥测中断时仍保持停车。

## 8. 日志对齐方法

选择一次“前进切左转”事件，按时间顺序对齐：

```text
Android touch
-> Android queue_dispatch
-> Android gatt_success
-> ESP32 motion_rx
-> ESP32 motion_target
-> ESP32 drive_output
-> 实际车轮动作
```

正式 BLE 固件当前不会把原始 `$MSPD` 输出到 USB，因此本阶段只能对齐到 `$spd` 和
实际车轮动作。`$MSPD` 字段与电机的对应关系应在第一阶段使用标定固件确认，不要在
正式遥控时混用两个固件。

判断方法：

| 现象 | 优先检查 |
|---|---|
| Android 没有新方向日志 | Android 触摸路由 |
| Android 有 dispatch，但没有 success | BLE GATT 写入 |
| Android success，但 ESP32 没有 motion_rx | BLE 传输或固件接收 |
| ESP32 motion_rx 正确，但 target 错误 | ESP32 四轮映射 |
| target 和 `$spd` 正确，但实际轮子异常 | 对照第一阶段 `$MSPD`，检查 AT8236、编码器、电机和机械负载 |

## 9. 立即停止测试的情况

出现以下任意情况时立即停车并断开电机电源：

- 松手后仍持续收到或执行非零命令；
- BLE 断开后车轮不停；
- `REVERSAL_BLOCKED` 状态下仍有车轮反转；
- 四轮出现互相对抗的方向；
- AT8236 遥测中断但电机继续转动；
- 电机、驱动板或线缆明显发热；
- 出现异常噪声、卡死或剧烈抖动。

## 10. 测试完成后需要保存的材料

- Android `CartControl` 日志；
- ESP32 USB 完整日志；
- M1-M4 正反转最低可靠速度表；
- `$MSPD` 字段与电机对应关系；
- 问题是否随电机或 AT8236 通道移动；
- 每个失败步骤的现象和发生时间；
- 当前烧录的是标定固件还是正式 BLE 固件。

关闭 ESP32 诊断：

```text
!D,0
```

测试结束后关闭 Android“记录日志”开关，并断开电机电源。
