# 真实小车 BLE 与四轮电机联调测试指南

## 1. 测试目的

本次测试要回答三个问题：

1. 手机是否按正确顺序发送了新方向命令？
2. ESP32 是否按相同顺序收到并输出了命令？
3. 四个电机是否能在相同命令下可靠起步、停止和反转？

测试期间只使用手动模式。**不要开启自动跟随。**

## 2. 安全要求

- 第一轮测试必须让四个车轮完全悬空。
- 小车周围不要放手、衣服、线缆或其他容易卷入车轮的物品。
- 旁边必须有人可以立即切断电机电源。
- 先确认 `c0,0` 或 USB 停车命令有效，再测试非零速度。
- 出现异常响声、剧烈抖动、失控转动或电机过热时，立即断电。
- 四轮悬空测试全部通过前，不允许落地测试。

## 3. 当前软件版本

截至 2026-07-13：

- 正式 BLE 固件已加入 USB 诊断和安全换向保护。
- 四轮 USB 标定固件位于：
  `firmware/esp32_at8236_calibration/esp32_at8236_calibration.ino`
- 正式 BLE 固件位于：
  `firmware/esp32_at8236_openbot_ble/esp32_at8236_openbot_ble.ino`
- Android 上位机提交：`e1c5238`。
- Android Debug APK：
  `dev/OpenBot/android/robot/build/outputs/apk/debug/robot-debug.apk`

两个 ESP32 固件不能同时使用。完成四轮标定后，必须重新烧录正式 BLE 固件，才能连接手机。

### 当前已经完成

- 正式 BLE 固件可以记录 Android 命令、四轮 target/current 和 `$spd`。
- 独立标定固件可以逐个测试 M1-M4，并输出原始 AT8236 遥测。
- 正式 BLE 固件可以阻止目标值直接反转，进入 `REVERSAL_BLOCKED` 后保持四轮零速。
- Android 可以记录触摸、pointer、generation、BLE 队列和 GATT 写入结果。
- 车轮悬空标定已初步确认 `$MSPD` 字段顺序、速度符号关系和 M1/M2 交换测试结论。

### 当前还没有完成

- 尚未完成四轮在落地负载下的最低可靠速度确认。
- 正式 BLE 固件尚未根据真实 `$MSPD` 自动判断四轮已经停稳。
- 当前换向保护仍使用人工松键并等待 1 秒的保守流程；悬空实测显示停止尾速可能超过 1 秒，后续应升级为实测过零联锁或延长中立等待。
- 真机悬空和落地验收尚未完成，自动跟随仍不能开放。

## 4. 测试工具

准备以下工具：

- 装有新版 OpenBot APK 的 Android 手机；
- USB 数据线；
- Arduino IDE 串口监视器，波特率 `115200`；
- 能运行 `adb` 的电脑终端；
- 用于记录结果的表格或文本文件；
- 可立即断开电机电源的人员或开关。

串口监视器发送命令时选择“新行”或确保命令末尾带换行。

## 5. 第一阶段：四轮单独标定

### 5.1 烧录标定固件

烧录：

```text
firmware/esp32_at8236_calibration/esp32_at8236_calibration.ino
```

该固件不启动 WiFi 和 BLE，只能通过 USB 测试。

烧录后：

1. 打开串口监视器，设置为 `115200`。
2. 确认四个车轮完全悬空。
3. 确认串口中持续出现 AT8236 遥测，或发送 `!Q` 查看状态。

### 5.2 保存静止遥测

发送：

```text
!U,50
```

保存至少 5 秒日志。正常情况下会看到 `$MSPD`、`$MTEP` 或 `$MAll` 帧。

如果没有遥测，不要继续测试电机。先检查：

- ESP32 GPIO17 是否连接 AT8236 RX2；
- ESP32 GPIO16 是否连接 AT8236 TX2；
- 两块板是否共地；
- 串口波特率是否为 `115200`。

### 5.3 解锁测试

发送：

```text
!A,RAISED
```

发送过 `!S,1000` 后，后续 `!T` 的序号必须大于 `1000`。

看到下面的回复才能继续：

```text
!OK,A
```

解锁有效期为 60 秒。超时后需要重新发送 `!A,RAISED`。

### 5.4 单电机测试命令

命令格式：

```text
!T,序号,电机编号,速度,持续时间毫秒
```

示例：让 M1 以速度 3 正转 500 ms：

```text
!T,1,1,3,500
```

让 M1 以速度 3 反转 500 ms：

```text
!T,2,1,-3,500
```

注意：

- 序号必须一直递增，不能重复。
- 电机编号只能是 `1、2、3、4`。
- 速度范围为 `-40..40`，不能为 0。
- 持续时间为 `100..1500 ms`。
- 每次只允许一个电机转动。
- 每次测试结束后至少等待 1 秒。

### 5.5 推荐测试顺序

对 M1、M2、M3、M4 分别测试以下速度：

```text
正转：3、5、8、12、14、20
反转：-3、-5、-8、-12、-14、-20
```

每次建议持续 `500 ms`。例如 M1 的前几条命令：

```text
!T,1,1,3,500
!T,2,1,-3,500
!T,3,1,5,500
!T,4,1,-5,500
```

后续命令继续增加序号。

每次记录：

| 电机 | 方向 | 命令速度 | 是否每次都能起转 | 起转是否明显延迟 | 停止是否拖延 | `$MSPD` 对应字段 |
|---|---|---:|---|---|---|---|
| M1 | 正转 | 3 |  |  |  |  |
| M1 | 反转 | -3 |  |  |  |  |

最低可靠速度是“连续多次都能正常起转”的最小速度，不是偶尔转动一次的速度。

### 5.6 紧急停车

需要立即停止时发送递增序号：

```text
!S,1000
```

该命令会停止四轮并取消解锁。再次测试前需要重新发送：

```text
!A,RAISED
```

### 5.7 第一阶段通过条件

第一阶段完成后，应得到：

- M1-M4 的正转最低可靠速度；
- M1-M4 的反转最低可靠速度；
- `$MSPD` 四个字段分别对应哪个电机；
- `$MSPD` 正负号是否与真实转向一致；
- 哪个问题跟随电机，哪个问题固定在 AT8236 通道。

如果某个问题很明显，可以交换电机与 AT8236 通道后重复测试：

- 问题跟着电机移动：优先检查电机、编码器和减速箱。
- 问题固定在通道：优先检查 AT8236 通道、接线和参数。

### 5.8 2026-07-13 悬空标定结果摘要

本节记录 2026-07-13 使用 USB 标定固件完成的 M1-M4 单轮测试、M2 低速复测，以及 M1/M2 电机互换后的归因测试。该结论仅适用于车轮悬空状态，落地负载下仍需重新确认最低可靠速度。

#### 5.8.1 AT8236 遥测字段

实测确认 `$MSPD`、`$MTEP`、`$MAll` 的四个字段顺序与 AT8236 通道一致：

| 通道 | `$MSPD` 字段 | 备注 |
|---|---:|---|
| M1 | 第 1 字段 | 单轮测试时仅第 1 字段变化 |
| M2 | 第 2 字段 | 单轮测试时仅第 2 字段变化 |
| M3 | 第 3 字段 | 单轮测试时仅第 3 字段变化 |
| M4 | 第 4 字段 | 单轮测试时仅第 4 字段变化 |

单轮测试期间未观察到明显串通道：被测通道以外的 `$MSPD` 字段保持为 0。

#### 5.8.2 速度符号

在当前接线和 AT8236 参数下，`$MSPD` 速度符号与标定固件发送的 `$spd` 命令符号相反：

```text
!T,...,speed=正数  -> 对应 $MSPD 字段多为负数
!T,...,speed=负数  -> 对应 $MSPD 字段多为正数
```

后续如果实现基于 `$MSPD` 的实际速度过零联锁，不能直接把 `$MSPD` 正负号等同于 `$spd` 命令正负号，必须保留这一反号关系。

#### 5.8.3 低速起步与停止尾速

悬空测试表明，底盘问题主要不是 BLE 命令丢失或 Android 旧按键残留，而是四轮实际机械/闭环响应不同步：

- `speed=5` 附近属于低速边界，M2 曾出现一次 `speed=5` 无明显响应，复测后又能起转，说明低速可靠性不足。
- M2 复测中，`speed=5/8/12` 与 `speed=-5/-8` 均能起转，但一次 `speed=5` 曾出现实测速度符号异常，倾向于低速边界、停止残余或闭环抖动造成的偶发现象。
- M1/M2 电机互换后，M1 通道和 M2 通道在 `speed=5/8` 均能起转，没有稳定复现“问题跟随某个电机”或“问题固定在第 2 通道”的证据。
- M1/M2 互换测试中，停止后 `$MSPD` 非零尾速约持续 `1.3..1.7 s`；因此 1 秒中立等待可能不足以覆盖最慢停稳情况。
- 正式 BLE 固件悬空测试中，USB 单发 `c14,14` 时 `spd=14,-14,-14,14`，四轮 `$MSPD` 均有响应；但 M2 峰值约 `304.64`，低于 M1/M3/M4 的约 `418.88..476.00`。这说明“右侧完全不动”不是 ESP32 漏发命令，而是 M2 右后轮/通道响应偏弱叠加 500 ms 单发超时造成的观感差异。

M1/M2 互换后的关键数据如下：

| 测试命令 | 当前通道 | 峰值 `$MSPD` | 首次检测到运动 | 停止后最后非零 |
|---|---:|---:|---:|---:|
| `!T,301,1,5,800` | M1 | `-1142.40` | 约 128 ms | 约 1310 ms |
| `!T,302,1,8,800` | M1 | `-1237.60` | 约 547 ms | 约 1728 ms |
| `!T,303,2,5,800` | M2 | `-875.84` | 约 255 ms | 约 1435 ms |
| `!T,304,2,8,800` | M2 | `-1351.84` | 约 346 ms | 约 1527 ms |

#### 5.8.4 当前归因

当前证据支持以下判断：

- 不是 BLE 链路本身导致四轮不同步；问题在 USB 单轮标定固件下也能观察到。
- 不是 Android 按键仲裁的主因；标定过程没有经过 Android。
- 暂不能判定某一个电机或某一个 AT8236 通道已经损坏；M1/M2 互换后两边均可起转。
- 更可能的主因是低速短脉冲下的电机、编码器、减速箱、AT8236 闭环响应和停止惯性综合差异。
- 正式 BLE 固件已先对 M2 增加一阶 trim：`M2_TRIM_PERCENT=130`。该补偿把协议目标 `-14` 的实际 `$spd` 输出提高到约 `-18`，仍受四轮 `[-40,40]` 上限约束。该值是悬空首轮补偿，不是最终落地参数。
- 2026-07-13 后续 USB 复测还显示：右侧 M2/M4 在第一次 `c14,14` 的前 `500 ms` 起步偏慢，但连续多次发送后四轮 `$MSPD` 可逐渐接近。这更像低速起步死区/静摩擦问题，而不是 BLE 没收到新指令。程序可以用“停车到运动时短时提高最低输出”的方式缓解，但它仍是开环补偿，不等价于四轮速度闭环同步。
- 2026-07-13 落地转向测试补充显示：`c-5,5`、`c5,-5` 悬空可正常转，落地后无法顺畅起步。因此正式固件将起步助推拆为两类：直行/后退继续使用落地直行中点版四轮独立最低值；原地左/右转使用单独的低力度四轮对称助推，避免重引入左右偏置。
- 2026-07-13 落地直行测试又显示：沿用悬空右侧补偿后，小车明显左转；日志中右侧 M2/M4 实测速度明显高于左侧 M1/M3，且 M3 在部分片段接近 0。随后首版落地补偿 `M1=120%`、`M2=90%`、`M3=130%`、`M4=80%` 又导致小车明显右偏；中点版 `M1=110%`、`M2=105%`、`M3=115%`、`M4=95%` 仍有轻微右偏。因此正式固件参数已继续小步回调为 `M1=105%`、`M2=110%`、`M3=110%`、`M4=100%`，使 `c14,14` 输出更接近左右均衡。

#### 5.8.5 对正式 BLE 固件和测试的影响

在完成基于 `$MSPD` 的实际速度过零联锁前，正式 BLE 联调应遵守：

- 不使用 `5` 作为正式可控运动速度；悬空联调优先从 `12` 或 `14` 开始。
- 复测 `c14,14` 时重点观察 `drive_output` 中 `spd=`、`startup_assist=` 与 `mspd=`：
  - 起步助推触发期间应看到 `startup_assist=1`；
  - 首个软启动台阶可能类似 `current=6,-6,-6,6,spd=14,-15,-15,14`；
  - 达到目标但助推仍在窗口内时可能类似 `current=14,-14,-14,14,spd=14,-15,-15,14`；
  - 助推结束或连续 BLE 运动稳定后，也应接近 `spd=14,-15,-15,14`。
- 测试 `c-5,5`、`c5,-5` 等原地转向时，若四轮 `$MSPD` 已确认静止，应看到
  `startup_assist=1,startup_assist_mode=turn`。起步窗口为 `TURN_STARTUP_ASSIST_MS = 1600 ms`。
  刚起步前 `TURN_BREAKAWAY_KICK_MS = 400 ms` 应看到 `turn_kick=1`，四轮对称破静摩擦
  输出 `TURN_BREAKAWAY_KICK_OUTPUT = 40`，因此 `c-5,5` 预期 `$spd`
  约为 `-40,-40,40,40`，`c5,-5` 约为 `40,40,-40,-40`。kick 结束后助推阶段
  维持 `TURN_STARTUP_ASSIST_MIN_OUTPUT = 32`；起步窗口结束后，纯原地转向仍应看到
  `turn_floor=1`，并保持 `TURN_HOLD_MIN_OUTPUT = 24` 左右的最低输出，避免重新掉回
  `±5` 的落地静摩擦死区。
  不应出现 `spd=-5,-24,5,22` 或 `spd=5,24,-5,-22` 这类右侧单边助推输出。
- 起步助推和停车后重新起步都必须以 `$MSPD` 实测静止为前提。若 USB 单发 `c14,14` 超时后车轮仍有尾速，再次单发 `c14,14` 时应看到 `motion_start_blocked` 和 `!ERR,settling`，这是为了避免“软件 current 已清零、实际轮子仍在转”时继续叠加运动输出。
- 保留 `REVERSAL_BLOCKED`，禁止前进/后退、左转/右转等直接反向切换。
- 如果继续使用固定中立等待，建议下一版从 `1000 ms` 提高到 `1800..2000 ms` 后再悬空验证。
- 更推荐的正式方案是：收到反向目标后先四轮零速，进入等待状态；只有连续若干帧 `$MSPD` 四字段均低于零速阈值后，才允许从零执行新方向。
- 落地测试前必须先完成悬空状态下的“运动、松手、等待停稳、反向”验收。

## 6. 第二阶段：手机 BLE 命令链测试

### 6.1 烧回正式 BLE 固件

烧录：

```text
firmware/esp32_at8236_openbot_ble/esp32_at8236_openbot_ble.ino
```

烧录后重新启动 ESP32。确认手机能够连接：

```text
OpenBot: CART_AT8236
```

OpenBot 页面应显示 BLE 和固件已经就绪。

### 6.2 开启两侧日志

ESP32 USB 串口发送：

```text
!D,1
```

Android 页面打开“记录日志”开关。

电脑运行：

```powershell
adb logcat -c
adb logcat -s CartControl:I
```

ESP32 串口应能看到：

```text
motion_rx
motion_target
drive_output
```

开启新版 USB 诊断后，`drive_output` 中还应包含 `startup_assist=`、`startup_assist_mode=`、`mspd=` 和 `mspd_age_ms=`。其中 `spd=` 是 ESP32 发给 AT8236 的目标输出，`startup_assist=1,startup_assist_mode=straight` 表示直行/后退起步助推正在生效，`startup_assist=1,startup_assist_mode=turn` 表示原地转向低力度对称助推正在生效，`mspd=` 是 AT8236 最近一次上报的四轮实测速度。若 `spd=` 四轮有值但某一侧 `mspd=` 长时间接近 0，应优先检查该侧通道、电机、编码器、接线、机械负载或低速死区，而不是继续修改 Android 队列。

Android 终端应能看到：

```text
touch_down
queue_enqueue
queue_dispatch
gatt_success
```

### 6.3 基本方向测试

保持车轮悬空，分别短按：

1. 前进；
2. 松手；
3. 后退；
4. 松手；
5. 原地左转；
6. 松手；
7. 原地右转；
8. 松手。

每次松手后都应看到：

```text
c0,0
```

如果松手后仍有非零命令，立即断电并停止测试。

注意：通过 USB 串口监视器单发一次 `c14,14` 只能维持约 500 ms，随后正式固件会触发运动保鲜超时并停车。这适合验证解析、映射和急停，不适合判断慢启动电机是否能连续运行。需要判断四轮实际响应时，应使用 Android/BLE 的 100 ms 重发，或在日志中重点查看同一次 `drive_output` 的 `spd=` 与 `mspd=` 对齐关系。

落地直行补偿版固件的悬空/低速复测步骤：

1. USB 串口发送 `!D,1`；
2. 发送 `!Q`，确认 `state=READY_STOP`、`startup_assist=0`；
3. 单发 `c14,14`；
4. 在 `drive_output` 中确认起步窗口内出现 `startup_assist=1`，并且四轮 `$spd` 符合落地直行小步均衡补偿，例如 `spd=14,-15,-15,14`；
5. 等待 `communication_timeout` 后，不要立刻重复发送；先继续观察 `$MSPD` 直到四轮回到接近 0；
6. 若在尾速未归零时重复发送 `c14,14`，应看到 `motion_start_blocked` 和 `!ERR,settling`，并且不应进入新的 `MANUAL_ACTIVE` 运动输出；
7. 观察同一批日志的 `mspd=` 和肉眼轨迹：若仍明显左转，优先继续降低 M2/M4 或提高 M1/M3；若明显右转，则反向微调。不要在直行未收敛前进入落地连续换向。

落地原地转向起步复测步骤：

1. USB 串口发送 `!D,1`；
2. 发送 `!Q`，确认 `state=READY_STOP`、`startup_assist=0`；
3. 临时发送 `h1500`，避免 USB 单发在默认 `500 ms` 处过早停车；
4. 单发 `c-5,5`，确认日志出现 `startup_assist,mode=turn,duration_ms=1600,kick_ms=400,kick_output=40,turn_start_min=32,turn_hold_min=24`；
5. 在最早的 `drive_output` 中确认出现 `turn_floor=1,startup_assist=1,startup_assist_mode=turn,turn_kick=1`，且 `spd=` 约为 `-40,-40,40,40`；
6. 继续观察 kick 后的 `drive_output`，应看到 `turn_kick=0` 且 `spd=` 约为 `-32,-32,32,32`；
7. 等待 `communication_timeout` 和四轮 `$MSPD` 回到接近 0；
8. 单发 `c5,-5`，确认最早 `spd=` 约为 `40,40,-40,-40`；
9. 测完 USB 后发送 `h500` 恢复默认保鲜超时，再进入 Android/BLE 长按测试；
10. 若 USB 下 `±40` 仍然完全没有 `$MSPD` 响应或车体不动，应停止继续加软件输出，优先检查电池电压、电机供电限流、AT8236 电流保护、轮子落地阻力、麦轮接触方向和机械卡滞；若 USB 可转而 BLE 不转，再回到 Android/BLE 命令刷新链路排查。

### 6.4 按钮切换测试

依次测试：

1. 松开“前进”后按“左转”；
2. 按住“前进”，用第二根手指按“左转”；
3. 单指从“前进”滑到“左转”；
4. 新方向生效后，再松开旧手指；
5. 左转切右转；
6. 前进切后退。

Android 中一次换向应按顺序出现：

```text
touch
queue_transition
queue_dispatch STOP
gatt_success c0,0
queue_dispatch 新方向
gatt_success 新方向
```

ESP32 中应按相同顺序出现：

```text
motion_rx c0,0
motion_rx 新方向
```

如果 Android 顺序正确、ESP32 顺序错误，检查 BLE 链路。

如果 ESP32 顺序正确、四个电机动作错误，进入电机和 AT8236 排查，不再修改 Android 按钮逻辑。

## 7. 安全换向保护测试

正式固件不会允许车轮直接反转。

例如先按前进，再按后退时，第一次反向命令应触发：

```text
REVERSAL_BLOCKED
```

此时四轮应立即保持零速。固件不会在后台自动继续后退。

正确操作步骤：

1. 松开当前方向按钮，使 Android 发送 `c0,0`；
2. 固件立即向 AT8236 下发四轮零速，并等待新鲜 `$MSPD` 确认四轮速度都落在 `±80` 内；
3. 从确认静止开始连续等待 `1000 ms`；若轮子再次转动，计时重新开始；
4. 等待期间即使用户已经长按目标方向，BLE 重复帧也只会被拒绝而不会重新开始计时；出现 `reversal_rearmed` 后，下一条重复帧自动从零速启动目标方向。

USB 日志应依次看到：

```text
reversal_block
reversal_neutral
reversal_rearmed
```

如果等待期间又收到非零命令，会看到：

```text
reversal_reject,neutral_age_ms=...
```

这说明命令被安全拦截；只要 `$MSPD` 继续保持静止，`neutral_age_ms` 应持续增大，不能被 Android 长按的重复帧重置。

可以发送以下命令查看状态：

```text
!Q
```

重点检查：

```text
reversal_blocked
reversal_neutral_age_ms
last_nonzero_target
```

### 安全换向通过条件

- 触发反向时四轮都先停车；
- 等待期间没有任何车轮提前反转；
- 中立等待结束前不能执行新方向；
- 中立等待结束后必须重新按键才能启动；
- BLE 断开、急停或遥测中断时仍保持停车。

### 7.1 2026-07-13 BLE 长按换向验收结果

本次使用 `!D,1` 采集了前进、后退与原地转向的多轮 BLE 长按日志，结论如下：

- BLE 按住期间约每 `90..120 ms` 收到一条相同的 `c` 命令，属于 Android 的正常保鲜刷新，不是旧按键残留；
- 前进运行时四轮 `$MSPD` 能升至约 `1200..1400`，松开后先出现 `immediate_stop` 和 `REVERSAL_BLOCKED`，未观察到反向 `$spd` 提前下发；
- 停止尾速进入 `±80` 后，日志出现 `reversal_neutral,started=1`；随后 `neutral_age_ms` 连续增长至 `1000 ms`，出现 `reversal_rearmed`；
- 用户持续长按后退或转向期间，重臂前的帧保持 `reversal_reject`，重臂后的下一条 BLE 重复帧才进入 `MANUAL_ACTIVE` 并输出目标方向；
- 日志中未出现 `communication_timeout`、`motion_timeout` 或 `driver_timeout`，本轮换向保护链路通过悬空/低速 BLE 验收。

本轮修复的根因是：早期实现会在每一条被拒绝的反向 BLE 重复帧到达时清零中立计时，使 Android 长按永远无法离开 `REVERSAL_BLOCKED`。现已改为“拒绝命令但不打断已开始的实测静止计时”。

## 8. 日志对齐方法

选择一次“前进切左转”事件，按时间顺序对齐：

```text
Android touch
-> Android queue_dispatch
-> Android gatt_success
-> ESP32 motion_rx
-> ESP32 motion_target
-> ESP32 drive_output
-> 实际车轮动作
```

正式 BLE 固件当前不会把原始 `$MSPD` 输出到 USB，因此本阶段只能对齐到 `$spd` 和
实际车轮动作。`$MSPD` 字段与电机的对应关系应在第一阶段使用标定固件确认，不要在
正式遥控时混用两个固件。

判断方法：

| 现象 | 优先检查 |
|---|---|
| Android 没有新方向日志 | Android 触摸路由 |
| Android 有 dispatch，但没有 success | BLE GATT 写入 |
| Android success，但 ESP32 没有 motion_rx | BLE 传输或固件接收 |
| ESP32 motion_rx 正确，但 target 错误 | ESP32 四轮映射 |
| target 和 `$spd` 正确，但实际轮子异常 | 对照第一阶段 `$MSPD`，检查 AT8236、编码器、电机和机械负载 |

## 9. 立即停止测试的情况

出现以下任意情况时立即停车并断开电机电源：

- 松手后仍持续收到或执行非零命令；
- BLE 断开后车轮不停；
- `REVERSAL_BLOCKED` 状态下仍有车轮反转；
- 四轮出现互相对抗的方向；
- AT8236 遥测中断但电机继续转动；
- 电机、驱动板或线缆明显发热；
- 出现异常噪声、卡死或剧烈抖动。
