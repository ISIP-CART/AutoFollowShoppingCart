# 货架拐角处小车跟随目标转弯：讨论总结与后续工作计划

更新时间：2026-07-08  
适用对象：自主跟随购物车项目上位机 / Android 端后续开发  
当前定位：在不依赖 ARCore、不引入完整 SLAM / 路径规划的前提下，为“货架拐角处目标短时消失后如何继续安全跟随”提供一条可逐步落地的技术路线。最新推进方向是：先以独立 `Depth Traversability Simulator / Passage Risk Tester` 验证单目深度能否作为“短时候选动作的可通行性 / 近距离风险门控”，不把它提前设计成识别左通道、右通道或完整局部拓扑的模块，也不直接耦合进 Human Cart Simulator 或主跟随状态机。

---

## 1. 问题背景

当前系统已经具备了较完整的基础框架：

- 人物检测；
- 目标初始化与确认；
- ReID 辅助身份判断；
- TargetTrack / targetBelief 等轨迹与身份信念机制；
- 跟随距离控制；
- 行为决策层与安全状态机；
- Human Cart Simulator 调试闭环。

在空旷场地中，只要目标不要移动太快，基于“目标框居中 + 保持距离 + 轨迹/身份信念”的视觉跟随大概率可以工作。

但在超市货架环境中，会出现一个更困难的问题：

```text
目标用户走到货架拐角处并转弯
  ↓
目标短时从摄像头画面侧边消失
  ↓
小车如果原地转圈搜索，可能看不到目标
  ↓
小车如果直接朝当前目标框最后位置转向，可能切角撞货架
```

因此，货架拐角处的跟随问题不能简单理解为：

```text
目标丢失 → 停车 → 原地搜索
```

更合理的问题定义是：

> 当目标在高置信状态下从画面边缘消失时，小车能否根据目标近期运动趋势、消失方向和局部可通行空间，低速、短时、受限地沿目标可能经过的路径继续移动，并在更有可能的位置进行定向搜索与重捕获？

---

## 2. 关键思想：从“追当前人框”转向“追目标留下的短时轨迹”

普通视觉跟随的核心是：

```text
当前目标在哪里？
  ↓
让目标回到画面中心
```

这种方式可以称为视线跟随（line-of-sight following）。它在空旷场地中有效，但在拐角处有天然缺陷，因为目标可能短时不可见。

对于货架拐角，更合理的思想是：

```text
看到目标时：记录目标刚刚走过的轨迹
目标消失时：不要立即盲目搜索，而是沿可信轨迹低速探过去
重新看到目标时：通过 ReID + track + belief 重捕获
不确定时：停止，不冒进
```

也就是说，小车不应该只追：

```text
当前画面中的目标框
```

而应该追：

```text
目标用户最近几秒留下的短时路径线索
```

这可以理解为“目标动作的延时版本”，但更准确地说是：

> 小车不是复制人的动作，而是延迟追踪目标在地面上的候选路径；同时受小车宽度、转弯半径、速度限制和安全状态约束。

## 3. OpenBot PointGoalNavigation 的启发

OpenBot 自带的 `PointGoalNavigation` 功能不是传统地图导航，也不是完整 SLAM / 路径规划。根据源码分析，它的工作方式可以概括为：

```text
用户输入一个相对目标点，例如前方 1 m、左侧 0 m
  ↓
ARCore 估计当前手机 / 机器人位姿
  ↓
系统计算当前到目标点的距离 goalDistance 和相对朝向 deltaYaw
  ↓
将相机图像 + goalDistance + sin(deltaYaw) + cos(deltaYaw) 输入 navigation.tflite
  ↓
模型直接输出左右轮控制量
  ↓
Vehicle.setControl() 下发到底盘
```

这个功能对本项目的直接可用性较低，原因包括：

1. ARCore 依赖 Google Play Services for AR，在中国大陆实机环境中不可稳定依赖；
2. `navigation.tflite` 是端到端策略网络，训练分布未必适合超市货架环境；
3. 该功能没有显式障碍物检测、局部规划、身份保持、ReID、targetBelief 或安全仲裁；
4. 它的控制输出直接是左右轮命令，不适合绕过本项目已经建立的安全行为决策层。

但它提供了一个非常重要的启发：

> 目标不一定必须是“当前画面里的人框”，也可以是一个局部空间中的短时目标点。

对于购物车来说，这个“点”可以来自：

```text
目标用户刚刚走过的位置
目标从画面边缘消失前的最后可见位置
目标近期运动趋势推断出的短时局部目标
货架拐角处可能的通道开口
```

因此，PointGoalNavigation 的真正启发不是“直接使用 ARCore 导航”，而是：

```text
把跟随目标从当前可见人框扩展为局部目标点 / 短时轨迹点。
```

## 4. 明确放弃 ARCore，但保留 AR 思想

基于国内 Google 服务限制，可以明确：

```text
本项目不把 ARCore 作为工程依赖。
```

但放弃 ARCore 不等于放弃 AR / VIO / 局部位姿思想。ARCore 提供的本质能力包括：

```text
相对位姿估计
局部目标锚点
目标相对距离与朝向
```

本项目可以抽象出一个接口：

```text
PoseProvider
```

未来可以有多种实现：

```text
NoPoseProvider               当前默认，不提供世界位姿
WheelImuPoseProvider         轮速里程计 + 手机 IMU yaw
MarkerPoseProvider           AprilTag / ArUco / 二维码标记辅助定位
OpenSourceVIOPoseProvider    ORB-SLAM / VINS / OpenVINS 等后续研究
VendorARPoseProvider         厂商 AR SDK，可作为非首版增强
```

这样系统不依赖 ARCore，但架构上保留“局部位姿 / 局部目标”的扩展能力。

---

## 5. 不要追一个“飘渺的点”，而要维护一个“局部假设”

不用 ARCore 时，如果直接说“追随一个目标点”，这个点会显得很飘渺，因为系统并没有稳定世界坐标。

因此，建议不把它称为：

```text
PointGoal
```

而称为：

```text
LocalGoalHypothesis
```

或：

```text
TrailHypothesis
```

它不是一个长期存在、精确定位的世界点，而是一个短时行为假设：

```text
目标大概率从某个方向消失；
小车可以在低速、短时间、低风险条件下尝试向这个方向靠近；
如果很快不能重捕获目标，就放弃该假设。
```

它必须具备：

```text
confidence          置信度
expiresAt           过期时间
maxTravelDistance   最大允许移动距离
maxDurationMs       最大允许持续时间
riskGate            安全门控
cancelReason        取消原因
```

这使得“目标点”从一个不可靠的虚拟坐标，变成一个可控、可过期、可被证据支持或否定的局部搜索假设。

## 6. 目标从屏幕侧边消失时：事件触发式局部可通行性验证

用户提出的一个重要想法是：

> 如果目标轨迹越过屏幕两侧，例如向左消失，就开始消耗更多算力去检查小车接下来那一小段“低速试探动作”是否可走。

这个想法可以定义为：

```text
事件触发式局部可通行性验证
```

平时系统不运行高成本深度模型或复杂感知；只有出现“疑似货架拐角”的事件时，才提高感知强度。

这里需要特别区分两个问题：

```text
不应让深度图回答：
  左边是不是有一条完整通道？
  右边是不是货架拐角入口？
  应该从哪里进入通道？

应该让深度图回答：
  如果状态机想做一次低速左弧 / 右弧 / 短前进试探，当前画面里的近距离风险是否过高？
  小车前方或试探方向的运动包络是否明显被墙、柱、货架、门框或近处障碍占据？
  是否应该允许这次 cautious motion，还是直接 MOTION_STOP / BLOCKED_WAIT？
```

也就是说，深度图不是“拓扑理解器”，而是“短时候选动作的风险门控”。

### 6.1 触发条件

```text
targetBelief 高
lockedTrack 稳定
目标连续向左 / 右移动
目标接近画面边缘
目标 bbox 被遮挡或从边缘消失
最近几帧没有多人身份混淆
当前系统无急停、通信异常或高风险障碍
```

### 6.2 触发后的状态

进入：

```text
CORNER_HYPOTHESIS
```

或直接进入一个更具体的短时动作：

```text
TRAIL_FOLLOW_CAUTION
LEFT_ARC_CAUTION
RIGHT_ARC_CAUTION
```

系统此时不应立即正常前进，而是执行低速、短时、受限的探测式动作。

## 7. 这不是普通迷宫搜索，而是带先验概率的局部假设搜索

超市货架拐角可以类比迷宫，但不应采用普通 DFS / BFS。

普通搜索是：

```text
没有偏好地探索所有可能分支
```

而小车跟随顾客时有强先验：

```text
目标刚刚的运动方向
目标从哪一侧消失
消失前的速度与位置
货架环境的可通行空间
```

因此更合理的是维护一组局部假设：

```text
H_left_turn           目标从左侧拐弯
H_right_turn          目标从右侧拐弯
H_straight            目标继续向前
H_stop_or_occluded    目标停下或短时被遮挡
H_lost                目标已经离开可恢复范围
```

每个假设有概率或置信度：

```text
p_left_turn
p_right_turn
p_straight
p_stop_or_occluded
p_lost
```

目标从左侧消失时：

```text
p_left_turn 增大
p_right_turn 降低
p_straight 根据前方可通行性调整
```

如果未来加入深度图或局部通路检测：

```text
左侧候选弧形试探区域近距离风险低       → 允许验证 H_left_turn
左侧候选弧形试探区域被近距离墙/柱占据   → veto H_left_turn 的运动尝试
短前进候选区域近距离风险低             → 允许验证 H_straight
短前进候选区域被近距离障碍占据         → veto H_straight 的运动尝试
```

最终小车不是“搜索所有分支”，而是：

```text
先由目标运动趋势提出最可能的局部候选动作，再由安全证据决定是否允许短时验证。
```

## 8. 单目深度 / 可通行性风险门控在这里的定位

最新讨论后，单目深度不再只是远期想法，而可以作为下一步调研与测试主线启动。但它的工程定位必须收敛：

```text
先独立验证，不先接入主状态机；
先输出 PassageEvidence，不直接输出底盘控制；
先按钮触发，不先自动触发；
先低频 / 事件触发，不默认每帧实时运行。
```

单目深度最适合的用途不是完整避障，也不是识别“左侧通道 / 右侧通道 / T 字路口”这类拓扑结构，而是：

```text
判断小车接下来一个很短、很慢的候选动作是否明显不可走
判断候选运动包络内是否存在近距离墙、柱、货架、门框、纸箱或人腿等风险
判断是否应允许一次 cautious motion，还是应该 MOTION_STOP / BLOCKED_WAIT
```

也就是说，深度图应服务于：

```text
TraversabilityGate / PassageRiskGate
```

而不是直接输出控制。

可输出：

```text
candidate_action
motion_envelope_open_score
motion_envelope_near_risk
forward_clearance_score
side_clearance_score
floor_continuity_score
traversability_verdict
confidence
risk_reason
```

再交给：

```text
LocalTopologyHypothesis
TrailFollowingPolicy
ActionArbitrator
```

例如：

```text
目标从左侧消失
TrailFollowingPolicy 提出 LEFT_ARC_CAUTION 候选动作
候选动作运动包络 near_risk 低
前方和左弧扫掠区域未被近处障碍明显占据
identityState 稳定
  ↓
允许短时 LEFT_ARC_CAUTION
```

反之：

```text
目标从左侧消失
TrailFollowingPolicy 提出 LEFT_ARC_CAUTION 候选动作
候选动作运动包络被近距离墙/柱/货架占据
  ↓
MOTION_STOP / BLOCKED_WAIT / 原地定向搜索
```

### 8.1 独立 Depth Traversability Simulator

建议新增一个类似 Human Cart Simulator 的独立测试页，暂定名：

```text
Depth Traversability Simulator
Passage Risk Tester
Depth Passage Simulator
```

它要回答的不是“小车如何完整绕过货架”，也不是“识别左边/右边是不是通道”，而是一个更小的问题：

> 当状态机提出一个短时、低速、受限的候选动作时，手机视觉能否判断这个动作的局部运动包络是否明显不可走？

首版页面建议包含：

```text
摄像头预览
深度热力图 overlay
候选动作运动包络 overlay
候选动作 open_score / near_risk
前方近距离风险
侧向近距离风险
当前手动候选动作
verdict: ALLOW_CAUTION / VETO_STOP / UNCLEAR
推理耗时
日志开关
模拟 LEFT_ARC_CAUTION
模拟 RIGHT_ARC_CAUTION
模拟 SHORT_FORWARD_CAUTION
清除候选动作
```

按钮模拟的是未来由 `TargetTrailMemory / DisappearSideDetector` 自动给出的事件：

```text
目标从左侧边缘消失后，状态机提出左弧低速试探
目标从右侧边缘消失后，状态机提出右弧低速试探
目标短时遮挡但前方仍可能可走，状态机提出短前进试探
无候选动作 / 清除
```

因此这个页面的价值不是替代 Human Cart Simulator，而是在接入 Human Cart Simulator 前，先把深度图、候选动作运动包络、风险分数、verdict 和日志链路验证清楚。

### 8.2 PassageEvidence 输出

建议把深度检测输出收敛成统一的风险证据结构：

```text
PassageEvidence:
  trigger_reason
  candidate_action
  motion_envelope_open_score
  motion_envelope_near_risk
  forward_near_risk
  side_near_risk
  floor_continuity_score
  traversability_verdict
  confidence
  risk_reason
  depth_inference_ms
  total_pipeline_ms
```

其中 `candidate_action` 来自状态机或人工按钮，而不是深度模型自己决定。首版 verdict 只作为 UI / 日志输出，不直接下发控制：

```text
ALLOW_CAUTION
VETO_STOP
UNCLEAR
```

解释：

```text
ALLOW_CAUTION  表示可允许一次极低速、短时、受限试探；
VETO_STOP      表示候选动作运动包络内近距离风险过高，应停止或等待；
UNCLEAR        表示深度图证据不足，不能因为它而冒进。
```

### 8.3 深度图到“能不能走”的风险分数

深度模型输出不能直接等价于“可通行空间”，更不能直接等价于“应该往左/右进入通道”。首版评分应围绕小车的候选动作运动包络，而不是围绕场景拓扑。

```text
SHORT_FORWARD_CAUTION:
  画面下半部中心楔形 / 梯形区域

LEFT_ARC_CAUTION:
  画面下半部中心偏左的弧形扫掠近似区域

RIGHT_ARC_CAUTION:
  画面下半部中心偏右的弧形扫掠近似区域
```

重点仍看画面中下部，例如：

```text
y = 40% ~ 95% height
```

原因是中下部更接近小车前方地面、货架底部和实际通行区域；画面上半部的货架、墙面和远处人体对“接下来几十厘米能不能低速试探”意义较弱。

单目深度多数是相对深度，因此需要先做百分位归一化：

```text
d5  = depth 第 5 百分位
d95 = depth 第 95 百分位
depth_norm = clamp((depth - d5) / (d95 - d5), 0, 1)
```

然后通过可视化确认模型输出方向：

```text
depth_norm 越大是否越远？
如果方向相反，则取 1 - depth_norm
```

最终统一成：

```text
far_score 越大 = 候选运动包络内越远 / 越开放
near_risk 越大 = 候选运动包络内越近 / 越危险
```

首版 `motion_envelope_open_score` 可采用可解释的启发式公式，例如：

```text
motion_envelope_open_score =
  0.40 * far_median
+ 0.25 * far_p75
+ 0.20 * free_area_ratio
+ 0.15 * vertical_continuity
- 0.40 * near_risk
```

然后再给出保守 verdict：

```text
near_risk 高 或底部近距离风险高      → VETO_STOP
open_score 中高 且 near_risk 低      → ALLOW_CAUTION
分数矛盾、深度图不稳定、反光严重     → UNCLEAR
```

公式不需要一开始完美，但 UI 和日志必须能显示各项分数，方便判断系统是否把货架缝隙、玻璃、反光地面或人腿间空隙误判成“可以走”。

### 8.4 初步模型路线

当前更深入调研可以按以下优先级展开：

```text
第一优先：Depth Anything V2 Small
第二优先：MiDaS 作为 PC 端对照基线
后续关注：Video Depth Anything 的时间一致性
```

Depth Anything V2 Small 适合作为第一候选，是因为它的相对深度泛化能力较强，并且存在较小模型和社区移动端转换尝试。但它仍然应被当作：

```text
相对深度 / 开放空间线索
```

而不是：

```text
可靠米制距离
完整避障系统
局部路径规划器
```

Android 部署路线优先保持和现有 OpenBot Android 工程一致：

```text
优先：TFLite / LiteRT standalone
备选：ONNX Runtime Mobile
暂不优先：依赖 Google Play services 的运行时
```

原因是本项目手机环境不能默认依赖 Google Play 服务。即使后续使用 GPU delegate，也应优先调研 standalone 打包和 CPU 多线程 fallback。

## 9. 推荐的新模块设计

### 9.1 TargetTrailMemory

维护目标最近几秒的短时轨迹：

```text
TargetTrailMemory:
  target_center_x_history
  bbox_bottom_history
  bbox_height_history
  distance_state_history
  target_belief_history
  locked_track_id_history
  last_visible_bbox
  last_visible_center
  last_visible_direction
  disappear_side
  trail_confidence
  trail_age_ms
```

如果未来有轮速、IMU 或 VIO，可扩展为：

```text
cart_pose_history
target_position_in_cart_frame
target_position_in_world_frame
```

### 9.2 LocalTopologyHypothesis

维护局部拓扑假设：

```text
LocalTopologyHypothesis:
  p_left_turn
  p_right_turn
  p_straight
  p_stop_or_occluded
  p_lost
  dominant_hypothesis
  hypothesis_confidence
  evidence_reason
```

### 9.3 TraversabilityGate / PassageRiskGate

负责判断某个候选动作的局部运动包络是否可低速试探。

当前可先做启发式版本：

```text
HeuristicTraversabilityGate
```

未来可以扩展：

```text
DepthTraversabilityGate
MarkerTraversabilityGate
OdomTraversabilityGate
```

输出：

```text
candidate_action
motion_envelope_open_score
motion_envelope_near_risk
traversability_verdict
risk_reason
confidence
```

### 9.4 TrailFollowingPolicy

根据目标轨迹、局部假设和安全证据，输出短时动作建议：

```text
TrailFollowingPolicy:
  inputs:
    TargetTrailMemory
    LocalTopologyHypothesis
    IdentityEvidence
    DistanceEvidence
    TraversabilityEvidence
    SystemSafetyEvidence

  outputs:
    FOLLOW_TRAIL_SLOW
    LEFT_ARC_CAUTION
    RIGHT_ARC_CAUTION
    LOCAL_SEARCH_LEFT
    LOCAL_SEARCH_RIGHT
    MOTION_STOP
```

### 9.5 PoseProvider

预留接口，不作为当前依赖：

```text
PoseProvider:
  type = NONE / ODOM_IMU / MARKER / VIO / AR
  pose_confidence
  current_pose
  yaw
  delta_distance
```

当前实现：

```text
PoseProvider = NONE
```

未来可升级为：

```text
wheel odometry + phone yaw
AprilTag / ArUco marker
open-source VIO / SLAM
vendor AR SDK
```

### 9.6 DepthPassageSimulator / PassageEvidence

新增独立调研模块：

```text
DepthPassageSimulator:
  inputs:
    RGB frame
    manual_candidate_action = LEFT_ARC_CAUTION / RIGHT_ARC_CAUTION / SHORT_FORWARD_CAUTION / NONE
    logging_enabled

  optional future inputs:
    target_last_bbox
    target_disappear_side
    target_belief
    trail_confidence

  outputs:
    PassageEvidence
    depth_visualization
    motion_envelope_overlay
    traversability_verdict
```

它与 `TraversabilityGate` 的关系是：

```text
DepthPassageSimulator 是测试页面 / 调研工具
TraversabilityGate 是未来可接入主系统的风险门控接口
PassageEvidence 是两者之间应复用的数据结构
```

当独立模块效果稳定后，再考虑把 `PassageEvidence` 接入：

```text
TargetTrailMemory
LocalTopologyHypothesis
TrailFollowingPolicy
ActionArbitrator
Human Cart Simulator
```

接入后也应保持边界：

```text
TrailFollowingPolicy 提出候选动作；
TraversabilityGate 判断该候选动作是否可试探；
ActionArbitrator 根据所有安全证据决定是否真正允许运动。
```

## 10. 推荐新增行为动作

建议在 `BehaviorAction` 或局部动作层中预留：

```text
TRAIL_FOLLOW_CAUTION
LEFT_ARC_CAUTION
RIGHT_ARC_CAUTION
FOLLOW_TRAIL_SLOW
```

其中最重要的是：

```text
TRAIL_FOLLOW_CAUTION
```

含义：

```text
目标刚刚高置信可见；
目标从画面边缘消失；
系统允许小车以极低速、短时间、受限地沿目标历史方向移动；
一旦风险升高或超时，立即停止或进入定向搜索。
```

### 10.1 行为约束

```text
最大持续时间：1~2 秒
最大前进距离：0.3~0.8 m
最大速度：极低
最大连续触发次数：有限
必须 targetBelief 高
必须 lockedTrack 最近稳定
必须消失方向明确
必须局部风险低
不得在多人混淆时触发
不得在前方/消失侧风险高时触发
不得直接跳回 FOLLOW_CONFIDENT
```

## 11. 状态机建议扩展

当前状态机可以新增一条短时分支：

```text
FOLLOW_CONFIDENT
  ↓ 目标高置信从边缘消失
TRAIL_FOLLOW_CAUTION
  ↓ 达到时间/距离上限，或风险升高
LOCAL_SEARCH_LEFT / LOCAL_SEARCH_RIGHT
  ↓ 发现疑似目标
REACQUIRE_TARGET
  ↓ ReID + track + belief 确认
FOLLOW_CAUTION
  ↓ 进一步稳定
FOLLOW_CONFIDENT
```

禁止从：

```text
TRAIL_FOLLOW_CAUTION
```

直接进入：

```text
FOLLOW_CONFIDENT
```

必须经过：

```text
REACQUIRE_TARGET / FOLLOW_CAUTION
```

这样可以避免：

```text
目标消失后，小车凭方向先验直接恢复正常跟随
```

造成误跟随或撞障风险。

## 12. 分层实现路线

### Level 0：Disappearance-based turn cue

不接深度，不接 AR，不接里程计。

依据：

```text
目标高置信
lockedTrack 稳定
目标连续向画面边缘移动
目标从左/右边缘消失
```

动作：

```text
目标从左侧消失 → LEFT_ARC_CAUTION
目标从右侧消失 → RIGHT_ARC_CAUTION
短时执行后 → LOCAL_SEARCH_LEFT / RIGHT
```

这是近期最容易验证的版本。

---

### Level 1：TargetTrailMemory + LocalTopologyHypothesis

维护目标近期轨迹和局部假设。

不要求真实世界坐标，只基于：

```text
图像空间轨迹
目标消失方向
bbox 尺度变化
targetBelief
最近状态
```

输出：

```text
p_left_turn / p_right_turn / p_straight
```

用于指导定向搜索。

---

### Level 2：Event-triggered TraversabilityGate

仅在疑似拐角时启用更高成本感知。

最新路线建议把 Level 2 拆成两个步骤。

第一步是独立测试，不接主状态机：

```text
Depth Traversability Simulator
按钮模拟候选动作：LEFT_ARC_CAUTION / RIGHT_ARC_CAUTION / SHORT_FORWARD_CAUTION
摄像头预览 + 深度热力图 + 候选动作运动包络 overlay
日志记录 depth_vis + motion_envelope_open_score + near_risk + verdict
```

第二步才是系统接入：

```text
TargetTrailMemory / TrailFollowingPolicy 自动提出 candidate_action
TraversabilityGate 输出 PassageEvidence
ActionArbitrator 只把它作为安全证据之一
```

在深度模型稳定前，仍可保留启发式版本：

```text
候选动作方向是否有明显检测框风险
候选动作扫掠区域是否被大物体占据
候选动作扫掠区域是否存在近距离风险
```

未来可接入：

```text
单目深度
稀疏光流
TTC
```

用于增强 `motion_envelope_open_score / motion_envelope_near_risk / traversability_verdict`。

---

### Level 3：Odometry / IMU-assisted trail

底盘联调后，可利用：

```text
wheel odometry
phone gyro yaw
```

把短时目标轨迹从图像空间增强为车体坐标近似轨迹。

这比纯图像版本更可靠，也不依赖 Google 服务。

---

### Level 4：VIO / SLAM / vendor AR

作为后续研究，不作为首版依赖。

可调研：

```text
ORB-SLAM3
VINS-Mono / VINS-Fusion
OpenVINS
RTAB-Map
厂商 AR SDK
```

但集成成本较高，不建议作为当前主线。

---

## 13. 数据采集与测试建议

为了研究货架拐角跟随，需要专门采集 sequence 数据，而不是普通 ReID crop 数据。

建议采集场景：

```text
1. 目标直行后左转出画面
2. 目标直行后右转出画面
3. 目标在货架拐角处短时消失后重新出现
4. 目标消失后干扰者从相同方向出现
5. 目标消失后干扰者从相反方向出现
6. 目标慢走、正常走、稍快走
7. SHORT_FORWARD_CAUTION 可试探 vs 前方明显不可走
8. LEFT_ARC_CAUTION 可试探 vs 左弧扫掠区域被近处墙/柱占据
9. RIGHT_ARC_CAUTION 可试探 vs 右弧扫掠区域被近处墙/柱占据
10. 深度易误判场景：玻璃、反光地面、货架缝隙、人腿间空隙、低矮障碍
```

针对 Depth Passage Simulator，建议先做三阶段测试：

```text
阶段 1：静态场景肉眼验证
  空旷通道、正前方货架、左/右近处墙柱、T 字路口、货架转角、人站在前方、纸箱/椅子障碍、反光地面

阶段 2：按钮触发人肉模拟
  手动选择 LEFT_ARC_CAUTION / RIGHT_ARC_CAUTION / SHORT_FORWARD_CAUTION，观察 App 给出的 ALLOW_CAUTION / VETO_STOP / UNCLEAR 是否符合直觉

阶段 3：接入 TargetTrailMemory
  用真实 disappear_side 生成候选动作，再接入 Human Cart Simulator 验证模拟跟随真人
```

建议日志字段：

```text
timestamp
frame_id
state
selected_action
action_reason
lockedTrackId
suspectedTrackId
targetBelief
target_center_x
bbox_bottom
bbox_height_ratio
disappear_side
candidate_action
trail_confidence
local_hypothesis
motion_envelope_open_score
motion_envelope_near_risk
forward_near_risk
side_near_risk
floor_continuity_score
traversability_verdict
passage_confidence
depth_inference_ms
total_pipeline_ms
risk_reason
```

如果打开独立模块的日志开关，建议保存：

```text
session_info.json
frame_log.csv
rgb/
depth_vis/
optional_depth_raw/
```

`depth_raw` 可能较大，首版不必每帧保存，可以只保存触发前后 1~2 秒，或者只保存手动触发帧。

如果硬件阶段可用，再加入：

```text
wheel_odom_delta
yaw_delta
estimated_travel_distance
```

## 14. 验收标准建议

### 14.1 安全性

```text
目标消失后不继续正常 FOLLOW
TRAIL_FOLLOW_CAUTION 期间速度极低
超过时间/距离限制后必停或定向搜索
多人混淆时不触发 trail following
前方风险高时不触发 trail following
目标重现后必须经过 REACQUIRE 验证
```

### 14.2 有效性

```text
目标从左侧消失时，系统优先 LOCAL_SEARCH_LEFT
目标从右侧消失时，系统优先 LOCAL_SEARCH_RIGHT
在模拟货架拐角中，较原地转圈更容易重新看到目标
目标返回后能通过 ReID + track + belief 恢复 FOLLOW_CAUTION
深度模块能对候选动作给出保守 ALLOW_CAUTION / VETO_STOP / UNCLEAR，而不是强行解释空间拓扑
```

### 14.3 可解释性

Debug 面板或日志中应能解释：

```text
为什么认为目标从左/右消失
为什么触发 TRAIL_FOLLOW_CAUTION
为什么选择 LEFT_ARC_CAUTION / RIGHT_ARC_CAUTION
为什么深度证据允许或否决候选动作
为什么取消该假设
为什么没有恢复 FOLLOW
```

## 15. 当前近期建议

短期内仍不建议把以下能力直接接入主跟随状态机：

```text
ARCore
完整 SLAM
开源 VIO
稠密光流
端到端导航网络
```

但可以立即启动一个独立的深度通路调研分支：

```text
1. 先在 PC 或独立脚本上跑 Depth Anything V2 Small，对货架 / 通道 / 拐角图片生成深度图。
2. 人工检查深度图是否能识别候选动作包络内的近距离风险，而不是要求它识别左 / 中 / 右通路。
3. 写 traversability_score 原型，输入深度图 + candidate_action，输出 motion_envelope_open_score / near_risk / verdict。
4. 如果 PC 端能稳定 veto 明显不可走场景，再做 Android 独立 Depth Traversability Simulator 页面。
5. Android 页先支持手动选择 candidate_action，不接 Human Cart Simulator。
6. 打通日志：depth_vis + motion_envelope_overlay + near_risk + verdict。
7. 效果稳定后，再接入 TargetTrailMemory / ActionArbitrator / Human Cart Simulator。
```

主线架构仍推荐继续推进：

```text
1. TargetTrailMemory
2. DisappearSideDetector
3. LocalTopologyHypothesis
4. TrailFollowingPolicy
5. LEFT_ARC_CAUTION / RIGHT_ARC_CAUTION
6. 定向 LOCAL_SEARCH 与 REACQUIRE 衔接
7. 日志与 Human Cart Simulator 可视化
```

因此接下来可以并行两条线：

```text
主状态机线：先不依赖深度和 AR，继续完善目标轨迹、消失方向、定向搜索和安全门控；
深度调研线：独立验证 PassageEvidence 是否足够可靠，再决定是否接入主线。
```

主状态机第一版仍可以完全不依赖深度和 AR，只基于：

```text
目标高置信轨迹
目标消失方向
图像空间运动趋势
行为安全门控
```

深度图、IMU、里程计或视觉标记都应作为 `TraversabilityGate / PoseProvider / PassageEvidence` 的增强证据，而不是绕过安全层直接控制小车。

## 16. 最终阶段结论

货架拐角处的目标跟随，不应被设计成：

```text
看到人 → 追人
看不到人 → 原地转圈
```

而应设计成：

```text
看到人 → 记录短时目标轨迹
目标从边缘消失 → 生成局部转弯/通道假设
局部风险低 → 极低速、短时、受限地沿假设方向试探
到达时间/距离上限 → 定向 LOCAL_SEARCH
重新看到疑似目标 → ReID + track + belief 重捕获
不确定或风险高 → MOTION_STOP / BLOCKED_WAIT / HARD_STOP
```

这条路线不依赖 ARCore，也不要求完整 SLAM。它把 ARCore / PointGoalNavigation 中“局部目标点”的思想，弱化为一个可过期、可解释、受安全约束的：

```text
LocalGoalHypothesis
```

这是比“原地转圈搜寻”更适合超市货架环境的跟随策略，也是后续从普通视觉跟随 demo 走向真实购物车场景的关键一步。

最新补充是：在 `LocalGoalHypothesis` 之外，下一阶段应开始验证 `PassageEvidence / TraversabilityEvidence`。它不回答“目标是谁”，也不回答“小车应该直接怎么开”，更不负责理解完整空间拓扑。它只回答：

```text
状态机提出的候选动作是否明显不可走？
候选动作运动包络内哪里存在近距离风险？
是否应该 veto 这次试探？
是否值得允许一次低速、短时、受限的候选动作验证？
```

因此，下一步最合适的工程动作不是直接把深度模型塞进 Human Cart Simulator，也不是继续训练它“识别左通道 / 右通道”，而是先做独立 `Depth Traversability Simulator`，用人工候选动作触发、运动包络可视化和日志把“能不能低速试探”这个问题单独研究透。
