## 2. 项目目标的现实化定义

### 2.1 设计期望

从应用场景出发，我们希望完成的是一台能够在室内超市或模拟超市场景中跟随用户移动、协助承载少量物品、并在异常情况下优先保证安全的自主跟随购物车。理想状态下，它应具备以下特征：

- 面向室内购物辅助场景
- 能够对单个目标用户进行视觉识别与低速跟随
- 具备基础载物能力
- 在目标丢失、急停、失联、障碍等情况下进入安全状态

### 2.2 当前真实定义

结合当前项目阶段和现实条件，项目的当前真实目标应定义为：

> 验证“OpenBot + Android 手机主脑 + 下位机控制 + 移动底盘 + 模块化购物筐”这一技术路线在室内购物辅助场景中的可行性，完成一个可演示、可复盘、可继续扩展的缩比原型。

这意味着当前项目重点不是复现商用尺寸购物车，也不是在首版追求大载重、复杂导航和长期部署，而是优先完成课程可交付原型。

### 2.3 当前口径

当前统一采用以下口径：

- 本项目是“自主跟随购物车原型”，不是商用购物车成品
- 本项目是“缩比原型验证”，不是等比例超市购物车复刻
- 本项目优先验证技术路线和系统闭环，而不是追求最大载重和最复杂功能

## 3. 现实约束与收敛依据

### 3.1 预算约束

课程项目要求总体预算受控，且小组在首版原型上不宜投入过高成本。结合当前判断，核心平台更合理的目标是尽量控制在 3000 元左右，而不是向大型服务机器人平台靠拢。

这直接影响了硬件选型：

- 不优先考虑大型专用机器人底盘
- 优先选用成熟竞赛底盘和现成模块
- 优先复用 OpenBot 的手机主脑架构，避免额外计算平台

### 3.2 周期约束

项目为 4 周集中实践，时间非常紧。首版方案必须优先保证能完成“从感知到控制到演示”的闭环，而不是把时间消耗在高风险机械方案或低可得性硬件上。

因此，方案收敛的原则是：

- 优先成熟方案
- 优先快速采购到货
- 优先模块化集成
- 优先可展示和可复盘

### 3.3 采购约束

近期调研中最明显的现实问题是：适合直接承载大型购物筐、又具备较好资料支持和合理成本的大型机器人底盘并不容易采购。

调研中暴露出的典型问题包括：

- 市面大底盘数量少，偏竞赛或特定场景
- 价格高，且容易突破课程项目合理预算
- 商家资料完整度参差不齐
- 到货、适配和调试风险偏高

这些约束说明，本项目不适合把“大型底盘 + 大型购物车结构”作为首版默认路线。

### 3.4 控制与载重约束

调研还表明，底盘标称承载能力并不等于其在“带载低速跟随场景”下可以长期稳定、安全运行。承载、稳定性、控制精度和安全冗余并不是一回事。

早期设想更接近真实购物筐，因此曾倾向设置较高载重目标；但在采购和底盘能力调研后，当前首版已收敛为缩比原型验证，不再把真实购物筐级承重作为必达指标。

因此，首版不应把载重指标设得过高。当前更合理的表达是：

- 以“购物筐 + 1~3kg 物品”的稳定演示为首版目标
- 更高载重能力以后续实测结果为准

### 3.5 安全约束

本项目的首版是面向课程展示的移动机器人原型，安全优先级高于复杂功能优先级。系统必须优先保证：

- 急停可靠
- 失联停车可靠
- 目标丢失后先取消前进，再短时安全搜索，超时后停车
- 前方出现障碍时至少能够减速或停车

这意味着首版不应为了“更聪明的行为”牺牲控制可预测性。

## 4. 总体方案演化

本项目并非一开始就直接收敛到当前方案，而是经历了一个从理想目标到现实落地的方案演化过程。

### 4.1 方案 A：大型底盘直接承载购物筐

这是最接近“真实购物车”直觉的方案。其基本思路是选用较大尺寸底盘，直接在上方集成购物筐或载物平台。

| 维度 | 判断 |
| --- | --- |
| 采购可得性 | 偏低，大尺寸合适底盘不容易稳定获得 |
| 成本 | 偏高，容易接近或超过课程合理预算 |
| 机械实现难度 | 中等，结构直观，但对底盘尺寸和固定方式要求高 |
| 控制复杂度 | 中等到偏高，带载惯性和转向稳定性需要关注 |
| 风险 | 采购风险、到货风险、适配风险较高 |
| 课程周期匹配度 | 偏低 |
| 展示效果 | 强，外观最像“购物车” |

当前结论：

- 该方案在概念上合理
- 但在采购可得性、成本和周期上风险偏高
- 当前阶段不优先采用

### 4.2 方案 B：牵引式方案

该方案的思路是由一个较小的移动底盘作为“牵引头”，在后方牵引购物筐或拖车结构，以尝试兼顾小底盘可得性和较大载物容积。

| 维度 | 判断 |
| --- | --- |
| 采购可得性 | 中等，小底盘容易买到，但拖车整体方案需额外设计 |
| 成本 | 中等，底盘本体可控，但拖车结构会增加加工与调试成本 |
| 机械实现难度 | 偏高，牵引连接、转弯稳定性、摆动控制都更复杂 |
| 控制复杂度 | 高，牵引系统比直接承载更难控制 |
| 风险 | 结构与控制双重风险较高 |
| 课程周期匹配度 | 低 |
| 展示效果 | 有一定创新性，但不稳定性较大 |

调研中发现的关键问题是：

- 当前常见竞赛底盘更适合“承载”，并不适合“牵引”
- 商家说明和使用经验也倾向于不建议将其作为牵引底盘
- 牵引式在转弯、刹停和路径稳定性上明显更难收敛

当前结论：

- 这是一个有探索意义的备选方案
- 但在当前课程周期内实现风险过高
- 因采购适配与控制复杂度风险，调研后暂不采用

### 4.3 方案 C：竞赛底盘 + 模块化购物筐

该方案的思路是采用可获得性较高、资料较成熟的竞赛底盘作为移动平台，在其上集成可拆装的模块化购物筐，以缩比原型方式验证购物辅助场景。

| 维度 | 判断 |
| --- | --- |
| 采购可得性 | 高，竞赛底盘和配套模块较容易采购 |
| 成本 | 中等偏低，更符合课程项目预算 |
| 机械实现难度 | 中等，重点在固定方式和重心管理 |
| 控制复杂度 | 中等，可基于成熟底盘先做稳定闭环 |
| 风险 | 相对可控 |
| 课程周期匹配度 | 高 |
| 展示效果 | 虽非商用尺寸，但能较稳定展示核心能力 |

当前结论：

- 当前优先采用方案 C
- 它不是“最像真实购物车”的方案，但它是当前最可落地的方案
- 该方案可以在预算、周期、采购可得性和工程风险之间取得更平衡的结果

### 4.4 阶段性结论

方案 A 和方案 B 并不是错误方案，而是经过调研后放弃的合理备选。当前选择方案 C，核心原因不是“做不到别的”，而是：

- 方案 C 更容易采购
- 方案 C 更适合 4 周课程周期
- 方案 C 更容易形成稳定演示闭环
- 方案 C 更利于后续扩展与回填正式文档

## 5. 结构设计策略

### 5.1 当前目标

结构设计的核心目标不是做出一台大尺寸购物车，而是在移动底盘上完成一个稳定、可拆装、便于测试和维护的载物模块。

当前首选方向是：模块化购物筐。

### 5.2 结构思路对比

| 方案 | 描述 | 优点 | 缺点 | 当前判断 |
| --- | --- | --- | --- | --- |
| 直接固定载物平台 | 在底盘上直接固定平台或小筐 | 实现简单，装配直接 | 可维护性一般，更换不方便，结构灵活性差 | 可作为简化备选 |
| 牵引式购物筐/拖车 | 小底盘牵引后方购物筐或拖车 | 载物体积可扩展，形式有新意 | 转弯和控制复杂，牵引稳定性差 | 当前不优先 |
| 模块化购物筐 | 底盘上设计可拆卸购物筐模块 | 易装卸、易维护、便于后续替换模块 | 需要进一步验证固定可靠性 | 当前优先 |

### 5.3 模块化购物筐的当前优先实现

当前优先考虑的结构方向是：

`磁吸 + 机械限位 + 快拆`

当前理解如下：

- 机械限位负责提供基本定位和抗位移能力
- 磁吸固定负责提高装拆便利性
- 快拆设计用于提升运输、调试和维护效率

需要明确的是，这一方向目前仍属于首选方案，而不是已经验证完成的最终定论。后续仍需重点验证：

- 在加减速和转向时，磁吸固定是否足够可靠
- 模块安装后是否存在松动、偏移或共振
- 快拆设计是否会影响结构稳定性

### 5.4 当前判断

模块化购物筐方案之所以优先，不是因为它最接近真实超市购物车，而是因为它在当前工程阶段具备以下优势：

- 不依赖专用大型购物车底盘
- 更容易运输和装配
- 更利于快速试错
- 更利于后续扩展不同载物模块

## 6. 电子与控制实现策略

### 6.1 总体主线

当前统一采用以下实现主线：

`OpenBot Android 手机 + ESP32 + 电机驱动模块 + 四驱编码器底盘`

其中：

- Android 手机作为上位机主脑，负责视觉感知、AI 推理、目标跟随决策和交互
- ESP32 作为下位机，负责指令接收、底层控制与安全保护
- 电机驱动模块负责驱动编码器电机
- 四驱编码器底盘负责移动执行

### 6.2 为什么当前倾向 ESP32

OpenBot 原始思路中可以看到 Arduino Nano 一类的下位机路径，但结合当前情况，我们更倾向将 `ESP32` 作为当前优先选型，原因包括：

- 串口通信接入方便
- 集成度更高
- 扩展余量更好
- 更适合后续增加状态反馈、保护逻辑或附加传感器

因此，Arduino Nano 在当前文档中更适合被表述为：

- 曾评估方案
- OpenBot 参考方案的一部分
- 非当前优先选型

### 6.3 为什么采用“手机决策 + 下位机执行”

当前架构采用分层控制，而不是把智能逻辑全部塞到 MCU 中，原因如下：

- Android 手机更适合运行视觉感知和 AI 推理
- 下位机更适合执行电机控制、超时保护和急停逻辑
- 分层后模块边界更清晰，便于调试和替换
- 这与 OpenBot 的总体设计哲学一致

因此，当前不考虑以下方向作为首版主线：

- MCU 独立完成视觉智能控制
- 额外引入树莓派作为主脑
- 服务器或云端参与实时控制闭环

## 7. 每位成员负责方向的技术路线对比

本节直接服务于课程要求：每位成员都需要围绕自己负责的方向提出至少 2 个技术方案，并比较优缺点。

### 7.1 成员 A：视觉跟随 / 上位机

| 方案 | 描述 | 优点 | 缺点 | 当前推荐 |
| --- | --- | --- | --- | --- |
| 方案 A1：仅检测驱动跟随 | 仅使用人物检测结果进行目标定位和跟随控制 | 实现简单，便于尽快打通闭环 | 遮挡和多人干扰下稳定性一般 | 可作起步方案 |
| 方案 A2：检测 + 轻量跟踪 | 先检测目标，再结合轻量跟踪提升连续性 | 稳定性更好，更适合短时遮挡 | 实现复杂度略高 | 当前更推荐 |

当前判断：

- 首版应优先保证单目标稳定跟随
- 在课程周期允许的前提下，推荐采用“检测 + 轻量跟踪”的折中路线

### 7.2 成员 B：下位机 / 控制

| 方案 | 描述 | 优点 | 缺点 | 当前推荐 |
| --- | --- | --- | --- | --- |
| 方案 B1：Arduino Nano | 参考 OpenBot 常见思路实现基础下位机控制 | 成熟、简单、资料多 | 扩展余量有限，当前方案统一性较弱 | 作为参考方案保留 |
| 方案 B2：ESP32 | 使用 ESP32 负责串口、驱动和保护逻辑 | 集成度高，扩展更方便 | 需要统一开发环境与控制逻辑 | 当前优先 |

当前判断：

- 为减少后续反复切换，下位机主线应尽早统一到 ESP32

### 7.3 成员 C：结构 / 集成

| 方案 | 描述 | 优点 | 缺点 | 当前推荐 |
| --- | --- | --- | --- | --- |
| 方案 C1：固定式载物平台 | 平台或小筐直接固定到底盘 | 简单、快速、适合首轮装配 | 模块更换和维护不便 | 可作最小可行备选 |
| 方案 C2：牵引式结构 | 小底盘牵引购物筐或拖车 | 形式有新意，容积有想象空间 | 机械和控制风险高 | 当前不优先 |
| 方案 C3：模块化购物筐 | 在底盘上集成可拆卸购物筐模块 | 兼顾可维护性、扩展性和展示性 | 固定可靠性需验证 | 当前优先 |

当前判断：

- 成员 C 负责方向最合适的优先路线是模块化购物筐
- 牵引式更适合作为被比较、被放弃的方案，而非首版实施方案

## 8. 首版真实实现范围

### 8.1 首版要完成的内容

当前首版真实实现范围应明确为：

- 手动遥控链路打通
- 目标初始化
- 单目标低速视觉跟随
- 模块化购物筐集成
- 目标丢失后的安全搜寻与停车
- 急停
- 失联停车
- 基础障碍减速或停车

### 8.2 首版暂不追求的内容

以下内容不应被视为首版默认目标：

- 商用尺寸购物车复现
- 高载重长期运行
- 完整 SLAM
- 复杂路径规划
- 自动返航
- 多车协同

### 8.3 当前范围控制原则

当前范围控制的原则是：

- 优先闭环
- 优先安全
- 优先稳定展示
- 不为了“更完整的产品想象”牺牲首版落地性

## 9. 当前关键参数与待验证项

### 9.1 已确定事项

当前已基本确定的事项包括：

- 首版采用缩比原型路线
- 竞赛底盘为当前主线
- 模块化购物筐为当前优先结构方向
- 手机主脑 + 下位机执行为当前控制分层方案
- 目标丢失处理采用“先取消前进，再短时搜索，超时停车”的安全策略

### 9.2 待验证事项

当前仍需后续测试确认的事项包括：

- 实际可稳定演示的载重上限
- 磁吸固定是否足够可靠
- 底盘长时间运行的稳定性
- 障碍检测方案的灵敏度与误报表现

### 9.3 当前参数口径

当前建议采用如下口径：

- 首版演示载重目标：`购物筐 + 1~3kg 物品`
- 搜索策略：目标丢失后立即取消前进，仅允许短时原地低速搜索
- 安全原则：任何急停、失联、异常都应优先转入停车状态

需要强调：

- 更高载重不作为当前承诺指标
- 结构和控制参数需以后续实测结果为准

## 11. 当前阶段结论

本项目当前最重要的工程判断不是“做一台真正的大购物车”，而是：

> 在课程项目的预算、采购可得性、周期和安全约束下，先把“自主跟随购物车原型”这一技术路线稳定做出来。

因此，当前最合理的阶段性选择是：

- 用缩比原型替代商用尺寸目标
- 用竞赛底盘替代难以获取的大型底盘
- 用模块化购物筐替代高风险整体式或牵引式结构
- 用工程收敛替代概念堆叠

这不是简单妥协，而是课程项目中更成熟的工程决策过程。

## 12. 2026-07-09 上位机策略验证边界更新

### 12.1 当前工程事实

截至 2026-07-09，上位机开发已经从“ReID 能否跑通”进入“身份线索如何安全进入 track / bbox / state machine”的验证阶段。Android 端 Human Cart Simulator 已完成以下代码接入：

- `osnet_x0_25_market1501.tflite` 在手机端完成 TFLite ReID 推理；
- ReID crop 已修正为 upright 输入；
- `TargetTrackManager + IdentityBeliefAccumulator` 已接入 trackId、lockedTrackId、targetBelief、suspectedTrack；
- track / bbox gate 修正已补入 locked ghost memory、suspected track 滞回、loose/default/strict gate、恢复后 relock 和非 locked 空间支持门控；
- Human Cart Simulator 新增“记录日志”开关，默认关闭，只有需要复盘时才生成 `cartfollow_diagnostics`。

### 12.2 当前不做的事项

为了保持首版验证变量收敛，下一轮测试前仍不做以下扩展：

```text
不更换 ReID 模型；
不启用 dynamic gallery；
不全局放宽 bbox gate；
不开放真实底盘前进；
不优先做复杂避障；
不改 ControlGenerator 的真实底盘控制路径。
```

### 12.3 进入底盘联调前的门槛

在 Human Cart Simulator 复测前，真实底盘仍不接通前进控制。只有当新版 `cartfollow_diagnostics` 与旧数据 compare 后满足以下条件，才考虑进入极低速底盘联调：

- 目标返回后能完成 relock；
- 干扰者不能仅靠高 ReID 分数进入 suspected 或 FOLLOW；
- `candidate_switch_penalty` 和 `belief_high_bbox_failed` 明显下降；
- `recovered_rate` 不下降；
- 非目标转绿次数不增加；
- `hard_stop_count` 不增加；
- debug / CSV reason 能解释恢复、拒绝恢复和拒绝切换原因。
