## 1. 调研目的

## 2. 差速底盘选型

### 2.1 差速底盘基本原理

差速底盘通过左右两侧车轮的独立速度差实现前进、后退和转向。对于双轮差速：

- 两侧同速正转 → 直线前进。
- 两侧同速反转 → 直线后退。
- 左侧转速 > 右侧转速 → 向右转弯。
- 右侧转速 > 左侧转速 → 向左转弯。
- 两侧速度大小相等、方向相反 → 原地转向（零转弯半径，适合丢失目标后的原地搜索）。

本项目首版目标场景（室内超市/模拟场地）需要的能力包括：低速直线跟随、缓弯跟随、原地搜索、精准停车。差速底盘对首版而言结构足够简单、控制接口足够清晰。

### 2.2 OpenBot 原生支持的车体类型

OpenBot 固件（`openbot.ino`）内置了多种车体配置的硬件宏：

| 配置宏 | 描述 | 电机类型 | MCU | 本项目适配性 |
| --- | --- | --- | --- | --- |
| `DIY` | 自建 L298N 驱动版本 | TT 电机 | Arduino Nano | 参考框架，底盘不同 |
| `RTR_TT` | 成品套件（TT 电机） | TT 电机 | Arduino Nano | 负载有限，不适合购物车 |
| `RTR_520` | 成品套件（520 电机） | 520 编码器电机 | ESP32 | 扭矩更大，与本项目底盘相近 |
| `DIY_ESP32` | 自建 + ESP32 | 自定 | ESP32 | 最适合本项目起点 |
| `RTR_TT2` | TT 成品 + ESP32 | TT 电机 | ESP32 | 参考通信框架 |
| `MTV` | 多地形车辆 | 520 电机 | ESP32 | 价格高，室内过度 |
| `RC_CAR` | 改装 RC 卡车 | 有刷/无刷 | Arduino Nano | 改装工作量大 |
| `LITE` | 教育版简化版本 | TT 电机 | Arduino Nano | 太小，负载不足 |
| `PCB_V1/V2` | 定制 PCB 版本 | TT 电机 | Arduino Nano | 需定制 PCB |

关键发现：`RTR_520` 和 `DIY_ESP32` 配置均使用 ESP32 MCU、包含编码器传感器支持和更丰富的传感器外围，且 ESP32 版的固件使用 `ledcWrite`（ESP32 LEDC PWM）替代 `analogWrite`，PWM 分辨率和频率均更优。

**四驱适配注意事项**：

- 底盘自带编码器电机（每转信号数取决于编码器磁盘孔数或磁极对数），需要适配 OpenBot 固件中的 `TICKS_PER_REV` 参数。

## 3. 微控制器（MCU）选型

### 3.1 Arduino Nano（ATmega328P）

| 维度 | 内容 |
| --- | --- |
| 主控芯片 | ATmega328P |
| 工作电压 | 5V |
| 主频 | 16 MHz |
| Flash | 32 KB（2 KB 用于 Bootloader） |
| SRAM | 2 KB |
| PWM 通道 | 6 路（D3, D5, D6, D9, D10, D11） |
| 外部中断 | 仅 2 路（D2, D3），需 PinChangeInterrupt 库扩展 |
| USB 串口 | 内置 CH340 / FT232 芯片 |
| 蓝牙 | 不支持（需外接 BLE 模块） |
| 价格 | 约 15-40 元 |
| OpenBot 原生支持 | 是（DIY / PCB_V1 / PCB_V2 / RTR_TT 等配置） |

Arduino Nano 的优点在于 OpenBot 的默认参考平台，装机量大，资料成熟。缺点在 SRAM 仅 2KB、中断引脚受限、无原生蓝牙。对于需要同时处理编码器计数、超声波中断、心跳超时和状态上报的任务，资源较为紧张。

### 3.2 ESP32（ESP32-WROOM-32E）

| 维度 | 内容 |
| --- | --- |
| 主控芯片 | ESP32-D0WDQ6（双核 Xtensa LX6） |
| 工作电压 | 3.3V（开发板含 USB 转串口和稳压） |
| 主频 | 最高 240 MHz |
| Flash | 4-16 MB |
| SRAM | 520 KB |
| PWM 通道 | 16 路 LEDC（任意 GPIO 可映射） |
| 中断 | 所有 GPIO 支持外部中断 |
| USB 串口 | 内置 CH340 / CP2102 |
| 蓝牙 | 原生支持 BLE 4.2 + Classic |
| WiFi | 原生 2.4 GHz |
| ADC | 12 位（0-4095），参考电压 3.3V |
| 价格 | 约 20-50 元 |
| OpenBot 原生支持 | 是（RTR_TT2 / RTR_520 / MTV / DIY_ESP32 配置） |

ESP32 的优点在于性能富余、中断灵活、原生蓝牙、ADC 精度更高（12 位 vs Nano 的 10 位）、GPIO 充足。OpenBot 已经提供了完整的 ESP32 配置（`DIY_ESP32`），并且 BLE 通信已经集成在固件中（设备名 `OpenBot: <robot_type>`，使用自定义 UUID 的 BLE UART 服务）。

### 3.3 本项目选择：ESP32

| 对比维度 | Arduino Nano | ESP32 | 本项目选择 |
| --- | --- | --- | --- |
| 计算能力 | 低（单核 16MHz） | 高（双核 240MHz） | ESP32 |
| SRAM | 2 KB（紧张） | 520 KB（充裕） | ESP32 |
| 中断支持 | 2 路（需库扩展） | 全 GPIO 支持 | ESP32 |
| 蓝牙 | 需外接模块 | 原生 BLE | ESP32 |
| ADC 精度 | 10 位 | 12 位 | ESP32 |
| GPIO 数量 | 14（6 PWM） | 20+（16 LEDC） | ESP32 |
| 单价 | 15-40 元 | 20-50 元 | 均可 |
| OpenBot 支持 | 完整 | 完整（DIY_ESP32） | ESP32 |
| 本项目已采购 | 否 | 是（4 件相关套件） | ESP32 |

**结论**：根据当前采购记录，本项目已采购 4 件 ESP32 相关开发板套件（`TYPEC-USB-32E 主板 + 已焊排针` 2 件，`拓展板` 2 件），统一以 ESP32 作为下位机。Arduino Nano 在仓库文档和 OpenBot 参考资料中仍会被引用（作为 OpenBot 原始方案的一部分），但当前优先选型是 ESP32。

## 4. 电机驱动模块

### 4.1 常见电机驱动对比

| 驱动方案 | 通道数 | 最大电流 | 电压范围 | 是否支持编码器 | 价格 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| L298N | 2 路 | 2A/路（峰值 3A） | 5-35V | 否（纯 PWM 控制） | ~15 元 | OpenBot `DIY` 默认选型，发热较大 |
| TB6612FNG | 2 路 | 1.2A/路（峰值 3.2A） | 2.5-13.5V | 否 | ~30 元 | 替代 L298N，效率更高 |

| L293D | 4 路 | 0.6A/路 | 4.5-36V | 否 | ~12 元 | 电流偏小，不适合 520 电机 |

### 4.2 本项目已采购驱动板

本项目已采购的驱动模块为**亚博智能 AT8236 四路编码器电机驱动模块**（天猫 895192929302），其关键特点：

- **四路独立驱动**：可分别控制四个电机，也支持两两分组做差速控制。
- **集成编码器接口**：板上提供编码器 AB 相信号输入接口，简化接线。
- **2A/路驱动能力**：匹配 12V 520 编码器电机。
- **铜柱 + 数据线 + 电源线** 套餐：减少复杂接线和焊接需求，适合课程快速联调。

### 4.3 四驱差速的电机分组

本项目采用四驱底盘，但控制上按左右分组（差速模型）：

| 组 | 电机 | 控制方式 |
| --- | --- | --- |
| 左侧 | 左前 + 左后 | 同一 PWM 通道（同速、同向） |
| 右侧 | 右前 + 右后 | 同一 PWM 通道（同速、同向） |

在 ESP32 固件中，可将左前/左后的 PWM 引脚映射到同一个 LEDC 通道，右前/右后映射到另一个通道。AT8236 有四路输出，两组映射关系如下：

- ESP32 LEDC CH0 → AT8236 左前 + 左后输入
- ESP32 LEDC CH1 → AT8236 右前 + 右后输入

### 4.4 PWM 控制原理

OpenBot 使用的速度指令范围为 [-255, 255]：

- 正值（1~255）：前进方向 PWM 占空比（0~100%）。
- 负值（-255~-1）：后退方向 PWM 占空比（绝对值对应的百分比）。
- 0：停止。

在固件层面（`update_left_motors()` / `update_right_motors()`）：

```
ctrl > 0  →  PWM_A = ctrl, PWM_B = 0  （正转）
ctrl < 0  →  PWM_A = 0, PWM_B = -ctrl （反转）
ctrl = 0  →  coast_mode ? (PWM_A=0, PWM_B=0) : (PWM_A=255, PWM_B=255)
```

`coast_mode`（惯性滑行）vs 制动（短路制动）的区别：
- 惯性滑行：两路 PWM 均为 0，电机自由旋转。
- 制动：两路 PWM 均为 255（100% 占空比），电机绕组短路制动。

本项目在购物车场景中建议**默认使用制动模式**（`coast_mode = 0`），因为购物车带载后惯性更大，制动能更快停车，符合安全优先原则。

## 5. 通信方式

### 5.1 对比总表

| 维度 | USB Serial | BLE（ESP32 原生） |
| --- | --- | --- |
| 物理连接 | USB OTG 线（手机 → MCU） | 无线 2.4 GHz |
| 波特率/带宽 | 115200 bps（OpenBot 默认） | 理论 ~1 Mbps（实际 ~20-50 KB/s） |
| 延迟 | 低（~1-3 ms） | 中（~10-50 ms） |
| 抗干扰性 | 强（有线连接） | 中（无线信号可能受 WiFi/环境干扰） |
| 硬件要求 | 手机需支持 USB Host/OTG | 仅 ESP32 |
| 调试便利性 | 串口监视器可直接抓包 | 需要 BLE 调试工具 |
| 部署灵活性 | 低（有线约束） | 高（无线） |
| 可靠性 | 高 | 中（断连需重连） |
| OpenBot 支持 | 原生，所有 MCU | ESP32 专属 |

### 5.2 OpenBot 串口协议

波特率：115200，数据位 8，无校验，1 停止位。

协议的核心特性：

- 手机通过 USB Serial 发送单字符头 + 参数体的控制指令。
- 每条指令以换行符 `\n` 结束。
- MCU 定时上报传感器数据（电压 `v`、速度 `w`、超声波距离 `s`、碰撞 `b` 等）。
- MCU 可响应手机的特征查询指令 `f`，返回机器人类型和传感器支持情况。

详细指令集见 `02-openbot-platform.md` 第 4.4 节。

### 5.3 ESP32 BLE 通信

当启用 `HAS_BLUETOOTH 1`（仅 ESP32），OpenBot 固件自动创建一个 BLE UART 服务：

- 设备名：`OpenBot: <robot_type>`（如 `OpenBot: DIY_ESP32`）。
- 服务 UUID：`61653dc3-4021-4d1e-ba83-8b4eec61d613`。
- 数据通过 BLE Notification 双向传输，协议格式与 USB Serial 一致（相同的 `c`/`i`/`h`/`v` 等指令格式）。
- 固件同时通过 USB Serial 和 BLE 输出（`sendData()` 函数同时发送到两个通道），实现双模通信。

### 5.4 本项目首版选择

**首版优先 USB Serial，原因**：

1. USB Serial 延迟更低、可靠性更高，适合跟随控制闭环。
2. 调试阶段可通过电脑串口监视器直接验证通信和传感器数据。
3. 减少首版的风险变量，先跑通有线链路再做无线扩展。
4. 固件已经通过 `sendData()` 支持双模输出，后续可平滑切换到 BLE。

**先决条件确认**：

- 手机必须支持 USB Host / OTG（绝大部分 Android 手机支持）。
- 需要准备 USB OTG 转接线（Micro-USB OTG 或 Type-C OTG，取决于手机接口）。
- 需要确认 ESP32 的 USB 串口芯片（CH340 或 CP2102）在手机上是否需要额外驱动（通常 Android 内置 CDC-ACM 驱动，CH340 可能需要 OTG 权限确认）。

### 6.3 供电关键考虑

| 考虑点 | 说明 |
| --- | --- |
| 独立供电 | 电机驱动和 ESP32 应使用同一电池但通过稳压模块隔离，避免电机启动瞬间拉低 ESP32 电压导致复位。 |
| 电池容量 | 12V 9600mAh ≈ 115 Wh。假设总功率 ~30W（4 个电机 × 6W + 控制 2W），理论续航约 3.5 小时，满足单次演示 ≥20 分钟的要求。 |
| 电压监测 | OpenBot 固件通过 ADC 和分压电阻监测电池电压，当电压低于 `VOLTAGE_LOW` 时可通过手机 App 告警。本项目 ESP32 的 ADC 为 12 位（0-4095），参考电压 3.3V，分压系数需根据实际分压电路计算。 |
| 最低驱动电压 | `VOLTAGE_MIN = 6.0V`（低于此值电机可能无法驱动），应在演示前充满电。 |

### 6.4 手机供电

Android 手机使用自带电池，不从小车电池取电。这是 OpenBot 架构的优势之一——手机自带大容量电池和管理芯片，降低了电源系统复杂度。但需要注意：

- 手机运行摄像头 + AI 推理时功耗较高（约 3-8W），连续 20 分钟演示需要手机电量充足。
- 手机通过 USB OTG 连接 ESP32 时，部分手机会尝试给 MCU 供电（OTG 5V 输出），此时应确保 ESP32 的 USB 和外部 5V 不会同时供电造成冲突。

## 7. 安全机制

### 7.1 OpenBot 固件内置的安全功能

| 安全机制 | 实现方式 | 对本项目的适用性 |
| --- | --- | --- |
| **心跳超时停车** | 手机定期发送 `h<time_ms>` 设置心跳间隔，MCU 在 `loop()` 中检查 `(millis() - heartbeat_time) >= heartbeat_interval` 则 `ctrl_left = ctrl_right = 0` | **核心复用**，必须在手机端配合心跳发送或依赖周期性控制指令更新 `heartbeat_time` |
| **超声波距离停车** | `STOP_DISTANCE = 10cm`，当 `distance_estimate <= STOP_DISTANCE` 且正在前进时强制 `ctrl = 0` | **需要增强**——10cm 阈值太近，本项目应提高至 ≥30cm |
| **碰撞传感器** | 通过模拟碰撞检测电阻网络识别碰撞方向和位置，触发 `emergency_stop()` | 本项目采购的底盘不一定有碰撞传感器，可参考但不依赖 |
| **电压监测** | MCU 定时上报电池电压，低于 `VOLTAGE_LOW` 告警 | 复用，辅助安全 |
| **指示灯** | 支持左右指示灯闪烁 | 可选，用于状态指示 |
| **编码器速度反馈** | 监控实际转速 | 可用于闭环速度控制和打滑检测 |

### 7.2 本项目需要增强的安全功能

#### 7.2.1 急停按钮（物理硬线）

OpenBot 原生固件没有内置急停按钮的中断处理。本项目必须增加。

**推荐实现方案**：

```
急停按钮（常闭触点）
    │
    ├──→ 串联在电机驱动电源线上（硬线切断，最高优先级）
    │         [12V 锂电池] → [急停 NC 触点] → [AT8236 驱动板电源输入]
    │
    └──→ 连接到 ESP32 GPIO（带中断）
               └──→ 下降沿中断 → 固件中将 ctrl_left = ctrl_right = 0
                   └──→ 通过 `sendData()` 上报急停事件给手机
```

**双层设计**：

1. **硬线层**：急停按钮的常闭（NC）触点直接串联在电机驱动电源回路中。按下急停 → 物理切断电机电源 → 无论 MCU 状态如何，电机立即失电停止。
2. **软件层**：同时将急停信号接入 ESP32 GPIO，触发中断 → 固件记录急停状态 → 通过串口/BLE 上报给手机 → 手机更新状态机为 EMERGENCY。

响应时间：硬线切断为机械/电子瞬态（< 10ms），满足 ≤0.5s 的验收要求。

#### 7.2.2 通信超时保护增强

在 OpenBot 现有心跳机制基础上增强：

- **心跳间隔设置**：建议 500ms（`h500`），即 500ms 内未收到任何来自手机的消息则 MCU 自动停车。
- **手机端对应**：手机端的跟随控制循环中，即使在 LOST / SEARCH 状态下也应持续发送指令或心率包，避免无通信时的意外停车；但一旦手机自身判定应 STOP，则停止发送运动指令，MCU 侧的心跳超时会在 500ms 内完成备用停车。
- **双重保护**：手机端决策 + MCU 底层超时 → 即使手机程序崩溃，MCU 也会在 500ms 内自动停车。

#### 7.2.3 超声波障碍停车增强

OpenBot 自带 `STOP_DISTANCE = 10cm`，对本项目太近（购物车带载惯性大、安全余量不足）。

**本项目建议参数**：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| `STOP_DISTANCE` | 30 cm | 硬停车距离 |
| 减速距离 | 50-80 cm | 手机端根据超声波读数做减速控制 |
| `SONAR_MEDIAN` | 1（启用中值滤波） | 减少振动干扰导致的误触发 |
| `sonar_interval` | 200-500 ms | 跟随状态下更快的采样率 |

超声波传感器的已知问题：

- 对振动敏感（OpenBot 官方文档明确指出），需要加装硅胶减震垫。
- 对软质或吸声材料（布料、购物袋）可能检测不到。
- 声波锥形波束可能导致侧向误报。

**降级方案**：如果超声波在移动中读数噪声过大，首版可以仅依赖视觉（手机摄像头中的障碍物检测框面积）做减速决策，超声波仅作为独立停车备份。

#### 7.2.4 编码器速度闭环与打滑检测

底盘自带的编码器电机使速度闭环成为可能：

- MCU 通过中断累积编码器脉冲，定时计算 RPM 并通过 `w<left_rpm>,<right_rpm>` 上报手机。
- 手机端可比较指令速度与实际速度，检测打滑或堵转。
- 打滑/堵转时触发降速或停车。

首版可以不实现完整的 PID 速度闭环（增加调试时间），但编码器读数至少应用于：

- 验证电机控制方向是否正确（手动遥控测试时）。
- 打滑/堵转检测（目标速度 > 0 但实际速度接近 0 持续一段时间 → 停车告警）。

### 7.3 安全分层架构

```
┌────────────────────────────────────────────────────────┐
│ 第 1 层：手机端决策层（Software Safety）                   │
│  - 速度上限（0.3-0.8 m/s）                               │
│  - 目标丢失 → 立即取消线速度，进入 LOST → SEARCH            │
│  - 超声波/视觉检测到障碍物 → 减速或停车                      │
│  - 状态机管理（FOLLOW/LOST/SEARCH/OBSTACLE/EMERGENCY/STOP）│
│  - 心跳发送（周期 ≤ 500ms）                               │
└──────────────────────────┬─────────────────────────────┘
                           │ USB Serial (115200 bps)
┌──────────────────────────▼─────────────────────────────┐
│ 第 2 层：MCU 固件层（Firmware Safety）                    │
│  - 心跳超时保护（500ms 无消息 → 自动停车）                  │
│  - 超声波硬停车（距离 ≤ 30cm 时强制 ctrl = 0）             │
│  - 编码器打滑/堵转检测（可选）                             │
│  - 电压过低告警 + 限速                                     │
│  - 急停按钮 GPIO 中断上报                                  │
└──────────────────────────┬─────────────────────────────┘
                           │ PWM / GPIO
┌──────────────────────────▼─────────────────────────────┐
│ 第 3 层：硬件层（Hardware Safety）                        │
│  - 急停按钮 NC 触点串联电机电源（物理切断）                 │
│  - 电池保险丝 / 过流保护                                    │
│  - 机械限位 / 防撞结构                                     │
│  - 购物筐固定（机械限位 + 磁吸辅助 + 防滑）                  │
└────────────────────────────────────────────────────────┘
```

这三层互相独立、互为备份。任何一层触发安全条件都能使小车进入停车状态。

### 7.4 安全状态机衔接

MCU 和手机端的安全状态需要衔接：

| 触发的安全事件 | MCU 侧动作 | 上报手机 | 手机端状态转换 |
| --- | --- | --- | --- |
| 心跳超时 | `ctrl_left = ctrl_right = 0` | 无（已断连） | 通信超时 → EMERGENCY |
| 超声波硬停车 | `ctrl_left = ctrl_right = 0` | `s<distance>` | OBSTACLE |
| 急停按下（硬线） | 电机失电 | 无（断电前可尝试发一条消息） | EMERGENCY |
| 急停按下（GPIO） | `ctrl = 0`，上报 | 自定义消息 | EMERGENCY |
| 打滑/堵转 | 上报异常 | `w` 数据异常 | 手机端判断后 → STOP |

## 10. 已知风险与降级方案

| 风险 | 详细说明 | 降级方案 |
| --- | --- | --- |
| 四驱底盘 PWM 不匹配 | AT8236 是 4 路驱动，左右并联后可能需要更大电流，或接线复杂 | 先以两驱模式测试（仅用前两轮或后两轮），确认通信和控制链路后再扩展到四驱 |
| ESP32 3.3V 逻辑与 5V 驱动板不兼容 | AT8236 逻辑输入可能要求 5V，ESP32 GPIO 为 3.3V | 确认 AT8236 规格书的逻辑电平阈值；大多数 3.3V 信号能触发 5V 输入，必要时加电平转换模块 |
| 编码器 TICKS_PER_REV 偏差 | 当前采购参数已确认减速比 30、减速后编码器线数 330，但实际计数仍会受到接线和边沿采样方式影响 | 先以 330 作为初始值进行联调，通过串口观察 RPM 读数合理性，并手工转一圈轮子数脉冲做最终校准 |
| 超声波振动干扰 | 底盘行驶中超声波读数大幅波动 | 启用 `SONAR_MEDIAN`（3 点中值滤波）、加装硅胶减震垫、降低超声波的决策权重 |
| 手机 OTG 不兼容 | 部分手机的 USB Host 需要特殊权限或 OTG 芯片不识别 CH340 | 准备 2 根不同品牌的 OTG 线；测试手机的 USB Host 能力；如果 USB Serial 无法工作，启用 ESP32 BLE 作为备选 |
| ESP32 供电冲突 | 手机 OTG 5V 输出与外部 5V 稳压同时给 ESP32 供电 | ESP32 开发板上有二极管保护，但仍建议：USB 仅用于烧录/调试，运行时 ESP32 由降压模块供电，手机仅通过 USB 串口 DATA 线通信 |

## 11. 阶段性结论

1. **底盘**：已采购的 305 系列四驱编码器底盘在负载、成本、可得性和控制复杂度间取得良好平衡。参照 OpenBot `RTR_520` 配置做固件适配，四驱左右分组做差速控制。

2. **MCU**：ESP32 为当前优先选型，已采购 4 件相关开发板套件（2 件主板 + 2 件拓展板）。相比 Arduino Nano，ESP32 在中断、蓝牙、ADC 精度和扩展性上有明显优势。固件基于 `DIY_ESP32` 配置适配本项目的四驱底盘和传感器。

3. **电机驱动**：AT8236 四路编码器驱动板集成度高、编码器接口方便，匹配 520 编码器电机。四驱固件映射为左右两个 PWM 通道。

4. **通信**：首版优先 USB Serial（稳定、低延迟、调试方便），固件同时支持 BLE（后续可平滑切换）。

5. **安全**：三层安全架构——手机端决策（状态机 + 限速 + 目标丢失处理）、MCU 固件（心跳超时 + 超声波硬停车 + 编码器打滑检测）、硬件（急停 NC 触点串联电机电源）。急停按钮必须同时覆盖硬线切断和 GPIO 中断上报。

6. **电源**：12V 9600mAh 锂电池组统一供电，电机直驱、MCU 经降压模块供电。手机使用自身电池。满电续航预估 3.5h，远超 20 分钟演示要求。

7. **运动安全由手机端策略和 MCU 底层保护共同承担**：手机负责"智能"层面的安全判断（目标丢失、障碍判定、状态机切换），MCU 负责"硬"层面的安全执行（心跳超时、超声波停车、急停响应）。两者互为备份，任何一层失效都不会导致小车失控。

## 13. 后续建议

1. 硬件到位后，首选适配方案是在 OpenBot 固件中基于 `DIY_ESP32` 配置创建本项目的自定义硬件配置，先跑通最简的 PWM 电机驱动（不含编码器和超声波）。
2. 逐步增加编码器读取、超声波测距、心跳超时和急停中断功能。
3. 在固件侧完成双模通信（USB Serial + BLE），确保降级链路可用。
4. 在整个联调过程中，保留串口日志用于排查通信异常和传感器噪声问题。
5. 急停按钮务必先以硬线切断方式验证（优先级最高），再接入 GPIO 做软件检测。
