# 04 人物检测与跟随控制调研

## 1. 调研目的

本文用于回答以下问题：

1. 首版应选用何种轻量级人物检测模型，模型的精度和速度如何权衡。
2. OpenBot 内置的检测→跟随管线如何工作，哪些可以直接复用。
3. 目标锁定策略：在画面中锁定指定用户，并在跟随过程中保持锁定。
4. 跟随控制策略：如何从目标检测框的位置和大小计算线速度和角速度。
5. 目标丢失判定与多帧容错：连续多少帧未检测到目标应判定丢失。
6. 短时搜索（原地扫描）的实现方式和参数。
7. 多人干扰和遮挡场景下的风险与降级方案。

本文技术细节基于 OpenBot 的 Robot App 实际架构和预置模型列表展开。

## 2. 可用模型与选型

### 2.1 OpenBot 预置/可下载检测模型

OpenBot 的 Robot App 内置 Model Management 模块，可下载和切换多个 TFLite 目标检测模型。以下为关键候选模型的基准数据（来源：OpenBot Robot App README）：

| 模型 | 输入分辨率 | COCO mAP | CPU FPS (S20FE) | 适用场景 |
| --- | --- | --- | --- | --- |
| MobileNetV1-300 (预置) | 300×300 | 18% | 34 | 默认模型，速度最快，精度最低 |
| MobileNetV3-320 | 320×320 | 16% | 34 | 精度略差于 V1，不建议优先使用 |
| YoloV4-tiny-224 | 224×224 | 22% | 30 | 精度提升，速度尚可 |
| YoloV4-tiny-416 | 416×416 | 29% | 12 | 精度明显提升，帧率显著下降 |
| YoloV5s-320 | 320×320 | 28% | 13 (Mi9) | 精度/速度平衡好 |
| EfficientDet-L0-320 | 320×320 | 26% | 16 | 横屏模式下性能劣化，需注意 |

### 2.2 选型策略

对本项目的选型建议分两步走：

**第一步（优先）**：使用预置的 MobileNetV1-300 跑通人物检测和基础跟随闭环。理由：
- 无需额外下载模型，减少依赖风险。
- 帧率最高（~34 FPS），对跟随控制的实时性最友好。
- 在受控单人场景（简单背景、正常光照）下，18% mAP 对 "person" 类别的单类检测通常足够。

**第二步（条件提升）**：如果 MobileNetV1-300 在测试中出现频繁漏检或误检，切换到 YoloV5s-320：
- 28% mAP 相比 18% 有明显提升，漏检率降低。
- ~13 FPS 仍然满足跟随控制的最低帧率要求（≥10 FPS）。
- YoloV4-tiny-416（29% / 12 FPS）作为精度备选，EfficientDet 因横屏性能劣化不作为优先。

### 2.3 推理设备选择

OpenBot 支持 CPU / GPU / NNAPI 三种推理后端：
- GPU 推理通常比 CPU 快 2-4 倍，多数现代手机支持。
- NNAPI 利用专用 AI 加速芯片（部分手机），可能更快但兼容性不稳定。
- **建议**：首版优先使用 GPU 推理，如果手机不支持 GPU delegate 则降级为 CPU（4 线程）。

### 2.4 模型输入与预处理

- 所有模型接受 RGB 图像输入，预处理（归一化、resize）由 OpenBot 的 `Detector.java` 自动完成。
- 摄像头分辨率可在 App 中调节，降低分辨率可提高帧率但降低检测精度。建议首版使用 640×480 作为起点，根据实际帧率调整。

## 3. 目标检测 → 跟随控制的管线

### 3.1 OpenBot Object Tracking 的工作流程

OpenBot 的 Object Tracking 界面提供了人物检测和跟随的完整管线，其工作流程如下：

1. 手机摄像头持续采集画面。
2. 选定的 TFLite 检测模型输出画面中所有检测到的物体类别与边界框。
3. 用户选择 "person"（COCO 类别 ID = 0）作为跟踪目标类别。
4. 系统从检测结果中筛选所有 "person" 类别的边界框。
5. 根据一定的规则（如置信度最高或面积最大）选择一个目标框作为当前跟踪目标。
6. 根据目标框在画面中的位置和大小计算控制量：
   - 横向偏差（目标框中心偏离画面中线的像素距离）→ 转向量。
   - 目标框面积（或高度）→ 距离估计 → 线速度。
7. 通过 USB Serial 发送 `c<left>,<right>` 指令给 MCU。

### 3.2 OpenBot 原始管线的关键限制

OpenBot 原生 Object Tracking 存在以下对本项目关键的限制：

1. **无目标锁定**：它简单地跟踪画面中"person"类别置信度最高的检测框。当多人进入画面时，可能自动跳转到另一个人。
2. **无目标丢失检测**：没有明确的多帧丢失判定逻辑。画面中无人时，控制量可能维持上一帧的值或进入不确定状态。
3. **无速度上限约束**：Dynamic Speed 模式根据目标框大小自动调速，但没有明确的安全上限（本项目需要 0.3-0.8 m/s 限速）。

**因此，本项目需要在 OpenBot 管线基础上增加三层关键逻辑**：目标初始化锁定、目标丢失检测、安全速度限幅。

## 4. 目标锁定策略

### 4.1 目标初始化

目标初始化发生在用户切换到 FOLLOW 模式的时刻：

1. 用户站在小车前方约 1.0-2.0 m 处（初始化区域）。
2. 系统运行人物检测模型，收集画面中所有检测到的 "person" 边界框。
3. 选择画面中**面积最大**且**置信度高于阈值**（建议 ≥ 0.5）的人物作为锁定目标。
4. 若多人存在，可通过以下方式区分：
   - **位置优先**：选择画面正中央且面积最大的人（代表站在初始化区域的用户）。
   - **距离优先**：选择检测框面积最大的人（代表距离最近、站在初始化位置的人）。
   - **首版简化**：在受控演示场景中，初始化时只允许目标用户在画面中出现，通过操作流程（目标准备好后才切换 FOLLOW）避免干扰。

5. 记录目标的**初始特征**用于后续去歧义，如初始边界框尺寸范围、初始颜色直方图（可选，增加计算量）。首版可不做特征记录，仅依赖检测+简单跟踪维持锁定。

### 4.2 维持锁定：检测 + 简单位置跟踪

锁定目标后，每一帧需要判断哪个检测框属于已锁定的目标。可选方案：

| 方案 | 描述 | 复杂度 | 首版建议 |
| --- | --- | --- | --- |
| 中心最近匹配 | 选择距离上一帧目标框中心最近的 person 检测框 | 极低 | 起步方案 |
| IoU 匹配 | 选择与上一帧目标框 IoU 最大的 person 检测框 | 低 | 替代方案 |
| SORT (Simple Online and Realtime Tracking) | 卡尔曼滤波预测 + 匈牙利匹配（基于 IoU） | 中 | **推荐方案** |
| DeepSORT | SORT + 深度外观特征提取 | 高 | 首版不采用 |

**首版推荐使用 SORT**：
- SORT 的核心思想是用卡尔曼滤波器对每个跟踪目标的状态（位置、尺寸、速度）进行预测和更新，再通过匈牙利算法在当前帧检测框和已有跟踪目标之间做数据关联。
- 对于本项目场景（室内、低速、单人跟随），SORT 的跟踪稳定性明显优于简单的中心最近匹配，特别是在短暂遮挡（如转角、其他行人穿行）时。
- 同时 SORT 的计算量很小（核心是矩阵运算和匈牙利算法），在手机端运行不会显著增加延迟。
- OpenBot 的 Java/Android 环境中，SORT 的核心逻辑（卡尔曼滤波 + 匈牙利）可以轻量实现。

### 4.3 跟踪器特征

SORT 跟踪器为每个跟踪目标维护的状态为 `[u, v, s, r, u̇, v̇, ṡ]`：
- `u`：边界框中心 x 坐标。
- `v`：边界框中心 y 坐标。
- `s`：边界框面积。
- `r`：边界框宽高比。
- `u̇`, `v̇`, `ṡ`：对应项的变化速度。

匹配时使用检测框与预测框的 IoU 作为代价矩阵，匈牙利算法做最优分配。

### 4.4 Track ID 与去歧义

- 初始化时创建唯一的 `track_id` 绑定到锁定的目标用户。
- 后续跟随中，仅跟踪 `track_id` 匹配的边界框，不理会其他 person 的检测框。
- 如果 SORT 的匈牙利匹配返回 `track_id` 在当前帧未匹配（即目标丢失），触发目标丢失逻辑。
- 如果在 SEARCH 阶段重新匹配到相同 `track_id`（或新的检测框与丢失位置相近），则恢复 FOLLOW。

## 5. 跟随控制策略

### 5.1 控制量计算

跟随控制将目标检测框的信息映射为运动控制指令：

**线速度（前进速度）**基于目标框的高度（或面积）：
- 目标框高度可用于粗略估计目标距离：离得越近，框越高。
- 目标距离基准值设为约 1.5 m（对应目标框高度约占画面高度的 40-50%）。
- 当目标框过小（目标太远）→ 增加速度以缩小距离。
- 当目标框过大（目标太近）→ 减速或停止，保持安全距离。

**角速度（转向量）**基于目标框中心的横向偏差：
- 目标框中心 x 坐标与画面中心的偏移量（像素）映射为转向量。
- 目标偏左 → 小车左转。
- 目标偏右 → 小车右转。
- 偏移量越大，转向越急。

### 5.2 控制算法选择

| 方案 | 描述 | 优点 | 缺点 | 首版建议 |
| --- | --- | --- | --- | --- |
| 规则控制（分段式） | 将偏差划分为多个区间，每个区间对应固定输出 | 实现极简，无参数漂移 | 平滑性差，可能蛇形 | 作为基线 |
| 比例控制（P） | `ω = Kp × 横向偏差` + `v = Kd × (目标距离 - 参考距离)` | 平滑性好，参数少 | 可能有稳态误差 | **推荐** |
| PID | 增加积分和微分项 | 理论最优 | 参数调试复杂，4 周周期内可能来不及 | 可选扩展 |
| Dynamic Speed（OpenBot 原生） | 根据目标框面积自动调节线速度 | 已内置，无需额外开发 | 调参空间有限 | 可复用 |

**首版推荐使用 P 控制（比例控制）**：
- 参数少（仅需调节 Kp 和 Kd），适合快速调优。
- 平滑性优于规则控制。
- 可先以 Dynamic Speed 的逻辑验证基础跟随，再替换为 P 控制提高可调性。

### 5.3 控制参数初步建议

| 参数 | 含义 | 建议初始值 | 说明 |
| --- | --- | --- | --- |
| V_MAX | 最大线速度 | 0.5 m/s（初始），可上调至 0.8 m/s | 安全优先，低速起步 |
| V_MIN | 最小线速度 | 0.1 m/s | 避免过慢导致无法克服静摩擦 |
| Kp_steering | 转向比例系数 | 需要实际标定 | 目标框中心每偏移 100 px 对应的角速度变化 |
| TARGET_DIST | 目标跟随距离 | 1.5 m | 对应目标框占画面高度的约 40-50% |
| STOP_DIST | 最小安全距离 | 1.0 m | 目标框过大时停止前进 |
| ω_MAX | 最大角速度 | 0.5 rad/s | 转弯速度上限 |

参数的实际值取决于摄像头视角、手机安装高度和模型输入分辨率，需要在联调阶段实测标定。标定方法详见第 9 节。

### 5.4 速度与距离的非线性关系

由于单目摄像头无法直接测量深度，目标框高度与距离的关系近似反比：

`距离 ≈ k / 目标框高度`

其中 `k` 与摄像头焦距、安装高度、目标真实身高有关，需要实测标定。在控制中，判断逻辑为：

- 目标框高度 < T_LOWER（距离 > 2.0 m）→ 加速追赶（V_MAX）。
- T_LOWER ≤ 目标框高度 ≤ T_UPPER（1.0 m ~ 2.0 m）→ 比例调速。
- 目标框高度 > T_UPPER（距离 < 1.0 m）→ 停车，v = 0。
- T_LOWER 和 T_UPPER 需要在初始化时根据锁定目标的初始框高设定。

## 6. 目标丢失判定与处理

### 6.1 目标丢失判定

在当前帧，锁定目标的 `track_id` 未匹配到任何检测框时，记为一次丢失：

- 需要**连续多帧丢失**才判定目标丢失。单帧或偶发丢帧是正常的（遮挡、光照瞬变等）。
- **建议丢失阈值**：连续 5-10 帧未匹配（约 150-300 ms 在 ~30 FPS 下，约 500 ms 在 ~10 FPS 下）。
- 帧数阈值可根据实际模型帧率调整：帧率低 → 降低阈值，帧率高 → 可适当提高阈值以减少误判。

### 6.2 目标丢失后的行为

遵循 AGENTS.md 中确认的目标丢失处理策略：

1. **立即取消线速度**：`v = 0`，禁止继续向前跟随。这是最高优先级安全措施。
2. **进入 LOST 状态**，启动短时搜索计时（建议 5 s）。
3. **搜索行为（SEARCH）**：仅允许原地低速左右扫描。
   - 实现方式：发送低速左右交替的转向指令（如 `c<30, -30>` → `c<-30, 30>` 交替，每 1-2 秒切换一次方向）。
   - 转向速度应远低于正常跟随时的转向速度（例如角速度 ≤ 0.3 rad/s）。
   - **严格禁止**前向位移。
4. **重锁定判断**：
   - 如果在搜索计时内 SORT 重新匹配到锁定目标的 `track_id`，或检测到置信度足够高且在丢失位置附近的 person 框，则恢复 FOLLOW 状态。
   - 恢复前应保守确认 3-5 帧连续匹配，避免误恢复。
5. **超时停止**：搜索计时超时后，发送 `STOP`（`c<0, 0>`），进入 STOP 状态，等待用户重新初始化。

### 6.3 搜索参数

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| SEARCH_TIMEOUT | 5 秒 | 超时后进入 STOP |
| SEARCH_ω | ±0.2-0.3 rad/s | 搜索时原地旋转的角速度 |
| SEARCH_SWITCH_INTERVAL | 1-2 秒 | 左右方向切换间隔 |
| REACQUIRE_FRAMES | 3-5 帧 | 连续检测到目标的帧数确认阈值 |

## 7. 多人干扰与遮挡风险

### 7.1 风险场景

| 场景 | 风险 | 发生条件 | 缓解措施 |
| --- | --- | --- | --- |
| 非目标行人穿行 | 小车可能误跟随穿行者 | 穿行者位于目标用户和小车之间 | SORT track_id 锁定维持，不切换到其他检测框 |
| 同向行走的第三人 | track_id 可能漂移到第三人 | 第三人外形与目标相似或目标暂时丢失 | 目标初始化时记录初始位置；跟踪器维护 track_id |
| 多人聚集 | 匈牙利匹配难以区分 | 多人在画面中密集分布 | 首版不要求解决此场景，演示时控制人数 |
| 货架遮挡 | 目标部分或全部消失在货架后 | 转弯或经过货架端部 | SORT 卡尔曼预测可桥接 1-3 帧短时遮挡；连续丢失触发 LOST |
| 用户蹲下 | 人物形态变化大，检测置信度骤降 | 用户蹲下取底层商品 | 降低检测置信度阈值（如 0.3）；SORT 维护短暂预测 |

### 7.2 首版处理原则

- **首版演示场景控制在单目标 + 无干扰或仅偶尔穿行**。多人密集场景不作为首版验收要求。
- 首版不实现复杂的 ReID（重识别）；如果用户离开视野后返回，依赖于检测框的位置最近匹配来重新关联（属于目标丢失后 SEARCH 阶段的自然恢复逻辑）。
- 如果多人场景无法避免，首版降级方案：仅在初始化时保证目标用户是画面中唯一的人，跟随中若出现第三人则手动暂停跟随。

## 8. 参数标定与调优

### 8.1 标定流程

1. **参考距离标定**：让目标用户站在小车正前方 1.0 m、1.5 m、2.0 m 三个距离，记录对应的目标框高度（像素），拟合反比关系曲线的 `k` 值。
2. **转向系数标定**：让目标从画面中心向左/向右各偏移一定角度，记录横向像素偏差和对应的目标角速度，计算 `Kp_steering`。
3. **速度系数标定**：在直线通道中，记录小车以固定指令（如 `c<100, 100>`）行驶时的实际速度，建立 PWM 指令值到实际速度的映射。
4. **多帧丢失阈值标定**：在受控场景中模拟短暂遮挡，观察模型恢复检测的典型帧数，设置合理的 `LOST_FRAMES` 阈值。

### 8.2 调优策略

| 症状 | 可能原因 | 调整方向 |
| --- | --- | --- |
| 蛇形摆动 | Kp_steering 过大 | 降低转向系数 |
| 转弯迟钝 | Kp_steering 过小 | 增加转向系数 |
| 速度振荡（加速-减速交替） | 距离控制增益过大 | 降低 Kd，增加死区 |
| 频繁进入 LOST | 检测置信度阈值过高或模型帧率过低 | 降低置信度阈值，检查推理帧率 |
| 跟丢后无法重锁定 | 搜索范围不够或匹配阈值过严 | 降低重匹配 IoU 阈值，扩大搜索转向范围 |
| 跟随距离不稳定 | 单目距离估计误差大 | 增大距离控制的死区范围，允许 0.3-0.5 m 的距离容忍 |

## 9. 控制指令输出

### 9.1 从 P 控制到 `c<left>,<right>`

P 控制的输出是**目标线速度** v 和**目标角速度** ω，需要转换为差速底盘的左右轮指令：

```
v_left  = v - ω × L / 2
v_right = v + ω × L / 2
```

其中 L 为轮距（约 20-30 cm，取决于底盘）。再将左右轮速度映射到 OpenBot 的 PWM 范围 [-255, 255]：

```
left_pwm  = clamp(v_left  / V_MAX_PWM × 255, -255, 255)
right_pwm = clamp(v_right / V_MAX_PWM × 255, -255, 255)
```

发送指令：`c<left_pwm>,<right_pwm>`。

### 9.2 指令发送频率

- 跟随模式下，每帧检测完成后即发送一次控制指令（~10-34 Hz，取决于模型帧率）。
- 心跳包（`h<500>`）在首次连接或模式切换时发送一次，MCU 侧的心跳计数器在 `loop()` 中持续检查。
- 正常跟随过程中，周期性控制指令本身也在更新 MCU 的 `heartbeat_time`，因此不需要额外的心跳包（除非处于 STOP 等不发送运动指令的状态）。

## 10. 与安全状态机的衔接

人物检测和跟随控制模块需要与安全状态机紧密衔接：

| 人物检测侧事件 | 状态机转换 | 控制侧行为 |
| --- | --- | --- |
| 目标初始化成功 | → FOLLOW | 开始输出跟随控制指令 |
| 连续 N 帧丢失目标 | FOLLOW → LOST | v = 0，不输出前向指令 |
| LOST + 搜索启用 | LOST → SEARCH | 输出原地扫描转向指令 |
| 搜索期间重锁定目标 | SEARCH → FOLLOW | 恢复跟随控制 |
| 搜索超时 | SEARCH → STOP | 发送 c<0,0> |
| 置信度持续过低（> 阈值） | 可配置 | 保守策略：触发 LOST |

## 11. 阶段性结论

1. **检测模型**：优先使用 MobileNetV1-300 跑通基础链路；精度不足时切换到 YoloV5s-320。推理选择 GPU 优先，CPU 4 线程备用。

2. **目标锁定**：SORT 跟踪器维护 `track_id` 封装"检测+预测+匹配"管线，成本低但比纯检测中心匹配更稳定。首版不采用 DeepSORT 或复杂 ReID。

3. **跟随控制**：P 控制（横向偏差→转向，距离估计→线速度），参数少、易调优。初始参数需在联调阶段实测标定。

4. **目标丢失**：连续 5-10 帧丢失判定 → v=0 → 低速原地扫描（5 秒超时）→ 超时 STOP。搜索期间禁止前向位移。

5. **多人干扰**：首版通过 SORT track_id 锁定 + 演示场景控制（单目标/无干扰）来保证稳定性，不做复杂 ReID。

6. **安全优先**：任何控制输出必须在 `V_MAX ≤ 0.8 m/s` 的限幅内。目标丢失后的取消前进逻辑应优先于任何"尝试恢复跟随"的意图。
