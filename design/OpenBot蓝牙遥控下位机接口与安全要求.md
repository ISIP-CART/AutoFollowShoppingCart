# OpenBot 蓝牙遥控下位机接口与安全要求

> 面向对象：负责 ESP32、AT8236 和底盘联调的同学<br>
> 当前目标：在 USB OTG 线材到位前，先通过 OpenBot Android App 的 BLE 通道完成真实小车低速遥控验证<br>
> 范围边界：本阶段只验证可靠建连、手动遥控和安全停车，不要求自动跟随、完整麦轮横移、传感器融合或复杂状态回传

## 1. 当前软硬件链路

当前底盘控制链路为：

```text
Android 手机
  -> BLE GATT
ESP32 WROOM-32E
  -> UART（GPIO17 TX1 / GPIO16 RX1，115200）
AT8236 四路电机驱动板
  -> 4 个 520 编码器电机
```

主仓库已有测试固件：

```text
firmware/esp32_at8236_m1_test/esp32_at8236_m1_test.ino
```

这份固件已经实现：

- ESP32 SoftAP 网页遥控；
- ESP32 到 AT8236 的 UART 控制；
- 四轮速度映射、软启动和轮间 trim；
- OpenBot 风格的 `c<left>,<right>` 和 `h<timeout_ms>` 串口命令；
- 默认 500 ms 指令超时停车；
- `!C / !M / !S / !H / !Q` 调试协议。

但当前固件尚未实现 OpenBot BLE GATT 服务，且仍有若干接真车前必须处理的问题。

## 2. 必须先解决的问题

### 2.1 `f` 命令协议碰撞（P0）

OpenBot Android 把：

```text
f\n
```

定义为“查询车辆类型和功能”。

当前 ESP32 测试固件却把单字符 `f` 定义为：

```text
M1 持续正转，直到收到停止命令
```

这是严重安全冲突。Android 页面可能主动发送 `f\n`，因此接入手机前必须：

1. 删除或通过开发编译开关禁用“M1 持续正转”的 `f` 命令；
2. 将 `f\n` 改为标准车辆功能查询；
3. 返回一行以 `f` 开头并以 `\n` 结尾的车辆能力信息。

本项目首版建议返回：

```text
fCART_AT8236:\n
```

当前传感器尚未接入时，不要声明 `s:`、`v:`、`wf:` 等尚不可用能力。以后接入后再按需扩展，例如：

```text
fCART_AT8236:s:v:wf:\n
```

### 2.2 当前没有 BLE 服务（P0）

当前测试固件只支持：

- USB/串口命令；
- ESP32 WiFi SoftAP 网页命令。

要被现有 OpenBot Android App 扫描并连接，ESP32 必须新增第 3 节定义的 BLE GATT 服务。不能使用经典蓝牙 SPP 代替，因为当前 App 扫描的是指定 BLE Service UUID。

### 2.3 WiFi、BLE、USB 控制权尚未互斥（P0）

当前代码中的控制来源可能被后到达的命令直接覆盖。真车上不能允许网页遥控、BLE 和 USB 同时抢占电机。

建议增加控制权状态：

```text
OWNER_NONE
OWNER_WIFI
OWNER_BLE
OWNER_USB
```

首版规则：

1. 上电默认 `OWNER_NONE`，电机停止；
2. 第一个通过有效连接并显式发送运动命令的来源获得控制权；
3. 其他来源发送非零运动命令时拒绝执行；
4. 任意来源的停止命令都允许立即执行；
5. 当前来源断开或超时后，先停车并释放控制权；
6. 释放后必须重新收到有效命令才能运动，禁止恢复旧目标速度；
7. 联调期间建议在启用 BLE 时直接关闭 WiFi 运动接口，只保留 WiFi 状态查看。

### 2.4 测试命令不能留在正式遥控固件中（P0）

以下命令会触发定时动作、持续单轮转动或延迟演示：

```text
1..8, a, b, f, r, w, x, g, z, c, h, l, p
```

其中部分单字符还与 OpenBot 原生协议冲突。请将这些测试命令移入：

```cpp
#if ENABLE_UNSAFE_BENCH_COMMANDS
...
#endif
```

正式遥控构建必须默认：

```cpp
ENABLE_UNSAFE_BENCH_COMMANDS = 0
```

### 2.5 BLE 断开和命令过期必须独立停车（P0）

不能只依赖 Android 主动发送 `c0,0`。ESP32 必须独立处理：

- BLE `onDisconnect`：立即 `stopAllMotors()`；
- 当前控制来源超过超时时间没有收到有效保鲜消息：立即停车；
- AT8236 通信异常：立即停车；
- 输入格式错误或越界：拒绝执行，必要时停车；
- 上电、重启、重新广播 BLE：保持停车状态。

Android BLE 串口完成 Notify 订阅后启动 `h750` 心跳，并持续刷新运动命令。下位机分别检查“通信链路保鲜”和“非零运动命令保鲜”，避免 App 仍发心跳但推理或遥控线程已经卡死时继续沿用旧速度。

## 3. OpenBot BLE GATT 接口

ESP32 应复用 OpenBot 原生 BLE UUID，不设计新协议。

### 3.1 广播信息

建议设备名：

```text
OpenBot: CART_AT8236
```

广播包必须包含以下 Service UUID，否则当前 Android App 的扫描过滤器不会显示设备：

```text
61653dc3-4021-4d1e-ba83-8b4eec61d613
```

### 3.2 GATT Service 与 Characteristics

| 类型 | UUID | 属性 | 方向 |
|---|---|---|---|
| UART Service | `61653dc3-4021-4d1e-ba83-8b4eec61d613` | Primary Service | - |
| RX Characteristic | `06386c14-86ea-4d71-811c-48f97c58f8c9` | Write / Write Without Response | 手机 -> ESP32 |
| TX Characteristic | `9bf1103b-834c-47cf-b149-c9e4bcf778a7` | Notify | ESP32 -> 手机 |

TX Characteristic 需要添加标准 CCCD（`BLE2902`），使 Android 能启用通知。

Android 建连后会尝试请求 MTU 64。下位机可以接受该请求，但协议不能依赖 MTU 一定为 64；首版所有单条消息应控制在 60 字节以内。

### 3.3 数据编码与分帧

- 编码：ASCII/UTF-8；
- 一条命令一行；
- 行结束符：`\n`；
- 接收端允许忽略 `\r`；
- 必须支持一条命令被拆成多个 BLE 写包；
- 也必须支持一个 BLE 写包内包含多条完整命令；
- 缓冲区溢出时清空当前帧并保持停车，不能执行截断命令。

BLE RX 和 USB Serial 应进入同一个 OpenBot 文本协议解析器，不要分别维护两套运动语义。

## 4. 下位机需要实现的 OpenBot 命令

### 4.1 左右轮控制 `c`（必须）

格式：

```text
c<left>,<right>\n
```

示例：

```text
c0,0\n
c40,40\n
c-30,-30\n
c-25,25\n
```

接口约定：

- Android 输入范围按 OpenBot 原生协议接受 `[-255, 255]`；
- ESP32 内部必须再次限幅；
- 首轮真车联调建议将实际输出上限限制为 `25~40`，确认方向和停车可靠后再调整；
- `left` 映射左前、左后轮；
- `right` 映射右前、右后轮；
- 自动跟随首版不使用麦轮横移；
- `c0,0` 必须直接进入停止流程，不等待软减速完成；
- 非零命令可以使用软启动/斜坡限速。

### 4.2 心跳 `h`（必须）

格式：

```text
h<timeout_ms>\n
```

Android 目标行为：每 250 ms 发送：

```text
h750\n
```

下位机收到合法 `h` 后应：

1. 校验超时范围，建议允许 `100..3000 ms`；
2. 更新通信保鲜时间；
3. 将本次会话超时阈值更新为参数值；
4. 超时后立即停车并释放控制权。

在 Android BLE 心跳补齐前，合法 `c` 命令也应刷新当前控制来源的保鲜时间。不要因为只收到一次 `h750` 就允许旧的非零速度无限持续。

### 4.3 功能查询 `f`（必须）

手机发送：

```text
f\n
```

ESP32 返回：

```text
fCART_AT8236:\n
```

注意：`f` 绝不能再触发任何电机动作。

### 4.4 停车语义（必须）

OpenBot 原生运动停车命令是：

```text
c0,0\n
```

扩展急停协议：

```text
!S,<seq>\n
```

BLE 和 USB 均可发送该命令，`seq` 必须是递增正整数。收到后进入锁存 `EMERGENCY_STOP`，只能重启 ESP32 解除。无论从哪个来源收到明确停车请求，都应覆盖当前控制权并立即停车。

### 4.5 状态回传（首版最小要求）

BLE 建连并启用 Notify 后，建议至少回传：

```text
r\n
```

表示下位机已就绪。

收到 `f\n` 后返回车辆功能行。其他回传可后续增加：

```text
v12.1\n       电池电压
s35\n         前方距离，单位需与上位机统一
w120,118\n    左右轮速
```

未实际接入或未校准的数据不要伪造回传。

## 5. 建议的安全状态机

建议下位机至少维护：

```text
BOOT_STOP
READY_STOP
MANUAL_ACTIVE
COM_TIMEOUT
EMERGENCY_STOP
DRIVER_ERROR
```

基本转换：

```text
上电 -> BOOT_STOP -> 初始化成功 -> READY_STOP
READY_STOP + 合法非零命令 -> MANUAL_ACTIVE
MANUAL_ACTIVE + c0,0 -> READY_STOP
MANUAL_ACTIVE + BLE断开/命令超时 -> COM_TIMEOUT
任意状态 + 急停 -> EMERGENCY_STOP
任意状态 + AT8236错误 -> DRIVER_ERROR
```

以下状态必须持续向 AT8236 保证零输出：

```text
BOOT_STOP
READY_STOP
COM_TIMEOUT
EMERGENCY_STOP
DRIVER_ERROR
```

急停和驱动错误不能因 BLE 自动重连而自动解除。

## 6. Android 遥控模块预期使用方式

后续 Android 将新增独立的真实小车遥控页面，首版预计提供：

- BLE 扫描、连接和连接状态；
- 手动/自动模式选择，但第一轮只开放手动；
- 前进、后退、原地左转、原地右转；
- 按住移动、松开即 `c0,0`；
- 页面暂停、App 退后台、模式切换时发送 `c0,0`；
- 明显的急停按钮；
- 默认低速上限；
- BLE 心跳定时发送。

下位机只接收统一的 `c/f/h` 协议，不需要知道命令来自手动页面还是自动跟随算法。

## 7. 联调验收清单

### 7.1 无电机/车轮悬空测试

- 手机能扫描到 `OpenBot: CART_AT8236`；
- 广播包含指定 Service UUID；
- Android 能发现 RX、TX Characteristic；
- Android 能向 RX 写入 `f\n`；
- ESP32 通过 TX Notify 返回 `fCART_AT8236:\n`；
- `f\n` 不产生任何电机动作；
- `c0,0\n` 始终停车；
- `c20,20\n` 只产生低速前进方向；
- 左右轮映射和符号与实车一致。

### 7.2 失联安全测试

- 非零运动中关闭手机蓝牙，ESP32 立即停车；
- 非零运动中断开 BLE，ESP32 立即停车；
- 停止发送命令，最长不超过配置超时即停车；
- ESP32 重启后不恢复旧速度；
- BLE 自动重新广播时保持停车；
- BLE 重连后必须收到新的非零命令才允许运动。

### 7.3 控制权测试

- BLE 获得控制权后，WiFi 非零运动命令被拒绝；
- WiFi 获得控制权后，BLE 非零运动命令被拒绝；
- 任一来源的停车命令都能立即生效；
- 当前来源断开/超时后先停车，再释放控制权；
- 不出现 WiFi 与 BLE 指令交替导致底盘抖动。

### 7.4 低速落地测试

- 空载、开阔场地、旁边有人随时物理断电；
- 首次落地只测试短按前进和停车；
- 再测试后退、原地左转、原地右转；
- 测试松手停车、App 退后台停车、蓝牙断开停车；
- 上述全部通过前，不接入自动跟随输出。

## 9. 本阶段不要求

以下内容不要阻塞本次 BLE 手动遥控联调：

- 自动跟随算法；
- ReID、深度估计或通路检测；
- 麦轮横移自主控制；
- 自定义二进制协议；
- 完整编码器里程计；
- 手机与 ESP32 的配对加密体系；
- 超声波/ToF 尚未到货部分。

本阶段的验收目标只有一句话：

> 手机能够通过 OpenBot 原生 BLE 接口低速、可预测地控制真实底盘，并且在松手、断连、超时和异常情况下可靠停车。
