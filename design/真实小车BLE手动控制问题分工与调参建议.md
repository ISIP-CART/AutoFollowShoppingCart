# 真实小车 BLE 手动控制问题分工与调参建议

## 1. 目的与当前问题

本文将真实小车 BLE 手动控制联调任务明确拆分为“上位机”和“下位机”两部分，避免两边同时修改速度、斜坡、超时和方向映射，导致无法判断问题来源。

当前需要解决两类现象：

1. 按下新方向后，小车仍可能继续执行旧方向，或停车、新方向、周期重发的顺序与预期不一致。
2. 四个电机起步速度、起步时间、减速和换向响应不一致；换向时可能出现部分轮已经反转、部分轮仍在原方向转动。

第二类问题会放大第一类问题。必须先证明命令顺序，再分析 ESP32、AT8236、编码器、电机和机械负载。

## 2. 当前代码基线与控制链

截至 2026-07-12：

- 主仓库基线：`964e551`。
- 主仓库记录的 OpenBot 子模块：`106bf5f`。
- OpenBot 上位机核心修复：`072090f Serialize BLE manual control writes`。
- ESP32 USB 诊断固件：`09a4e88 Add ESP32 USB control diagnostics`，已合入主仓库。

当前控制链如下：

```text
Android 方向按钮
  -> 手动按键 pointer / generation 仲裁
  -> BLE 单在途写队列
  -> c<left>,<right>\n
  -> ESP32 设置四轮 target
  -> 每 40 ms、步长 6 更新四轮 current
  -> $spd:m1,m2,m3,m4#
  -> AT8236 四路电机控制与编码器反馈
```

正式运动命令不带动作持续时间。Android 按住时每 `100 ms` 重发当前方向；ESP32 在非零运动命令超过 `500 ms` 未刷新时停车。`500 ms` 是失效保护，不是正常换向延迟。

WiFi 版本看起来较稳定，主要因为网页端已有 `pendingCmd + inFlight + seq`，天然形成单请求串行与旧命令淘汰。BLE 本身不是必然更乱；此前的问题在于 Android 连续发起异步 GATT 写入时没有应用级顺序保证。最新 OpenBot 已为 BLE 补上对应机制，下一步是用真机日志验证，而不是仅凭源码宣布问题解决。

## 3. 上位机任务

### 3.1 已完成的软件职责

- BLE 串口使用单在途写队列，前一条完成后才发送下一条。
- 方向替换作为 `c0,0 -> 新方向` 的有序事务提交。
- 运动命令采用 latest-wins；停车与急停优先级高于普通运动和心跳。
- 手动按键绑定 pointer ID 与 generation，旧手指的迟到释放不得覆盖新方向。
- 页面显示构建版本、当前手动方向、左右输出和 BLE 写入状态。
- 手动速度首轮调整为前进 `14`、后退 `12`、原地转向 `5`；自动最大输出和搜索速度暂降为 `14/5`。
- 单元测试、格式检查与 Debug APK 构建已通过；这不等于真机验收完成。

### 3.2 当前必须完成的任务

1. 从 `106bf5f` 或更新提交构建并安装 APK，页面或日志必须能确认构建版本。
2. 记录每次触摸事件的时间、按钮、`actionMasked`、pointer ID、generation 和目标左右值。
3. 记录 BLE 队列的入队、实际写入、成功/失败回调、重试和耗时。
4. 分别验证以下手势，不能合并为一个用例：
   - 松开 A 后按 B；
   - 按住 A 时用第二根手指按 B；
   - 单指从 A 滑到 B；
   - 新方向生效后旧手指迟放；
   - 最后一个有效手指松开；
   - 页面退后台、BLE 断开和窗口取消触摸。
5. 输出一段完整证据，证明方向替换在 Android/BLE 层的顺序为：

```text
FORWARD down
-> c14,14 write success
LEFT down
-> c0,0 write success
-> c-5,5 write success
FORWARD late up/cancel
-> ignored
LEFT periodic repeat
-> c-5,5
LEFT up
-> c0,0 write success
```

### 3.3 上位机责任边界

- 上位机负责按钮语义、BLE 写入顺序、重试、旧命令淘汰和左右目标限幅。
- 上位机不负责四个电机的个体死区、编码器方向、AT8236 PID、单轮起步补偿或实际轮速同步。
- `14/12/5` 只是首轮低速目标，不代表四轮在这些值下都能稳定起转。若下位机测得某个值低于最低可靠轮速，应由双方重新确定安全可控速度，不能继续盲目降低。
- 在 BLE 写成功日志没有证明新命令已正确发出前，不要求下位机通过缩短 watchdog 掩盖上位机问题。

### 3.4 上位机交付物

- 新 APK 的构建提交与安装确认。
- 上述所有手势的触摸/仲裁/BLE 写入日志。
- 每个失败用例对应的最后成功写入命令和时间。
- 明确结论：旧命令是否可能在新 generation 之后写出。

## 4. 下位机任务

### 4.1 已完成的软件职责

- ESP32 支持 BLE 与 USB 控制权互斥、`c0,0` 立即停车、BLE 断连停车、运动保鲜超时和锁存急停。
- USB `!D,1/!D,0` 可开启/关闭诊断，默认关闭，不增加高频 BLE Notify。
- 诊断可记录合法 `c` 的时间、来源、递增计数、左右值、四轮 target/current 和实际 `$spd`。
- USB `!Q` 可查询最近运动命令、命令年龄、计数、target/current、owner 和状态机。
- 当前动态参数保持：`CONTROL_UPDATE_MS=40`、`CONTROL_RAMP_STEP=6`、`MOTION_REFRESH_TIMEOUT_MS=500`、四轮输出上限 `[-40,40]`。

### 4.2 第一优先级：证明命令执行链

1. 用 Arduino IDE 编译并烧录包含 `09a4e88` 或更新内容的固件。
2. 车轮悬空，通过 USB 发送 `!D,1`，然后只允许 Android BLE 发送运动命令。
3. 对照 Android 日志，确认 ESP32 按相同顺序收到 `c14,14`、`c0,0`、`c-5,5` 等命令。
4. 对每条命令核对：
   - `motion_rx` 的时间、来源、seq、left/right；
   - `motion_target` 的 M1-M4 target；
   - `drive_output` 的 current 与实际 `$spd`；
   - AT8236 `$MSPD` 的实际四轮速度和方向。
5. 若 Android 已写成功但 ESP32 没有 `motion_rx`，回到 BLE 链路排查；若 ESP32 的 `$spd` 正确但电机动作错误，再进入电机/驱动排查。

### 4.3 第二优先级：逐轮定位不同步来源

对 M1-M4 分别进行相同的低速阶跃测试，记录：

- 最低可靠起转命令；
- 从命令发出到开始转动的时间；
- 相同命令下的稳态 `$MSPD`；
- 从非零到实际零速的时间；
- 正转到反转的过零时间；
- 空载悬空与落地承载后的差异。

然后进行交叉替换：

- 问题随电机/减速箱移动：检查电机、编码器、减速箱阻力和机械负载。
- 问题固定在 AT8236 通道：检查通道参数、编码器接线、方向、供电和驱动输出。
- 问题只在落地出现：检查轮胎摩擦、车架受力、轮轴安装和电源压降。

必须核对四路编码器的方向与数值。trim 只能修正稳态速度比例，不能解决起步延迟、静摩擦、停止惯性和换向过零不同步。

### 4.4 第三优先级：实现安全换向

当前 `c0,0 -> 新方向` 只保证命令顺序，不保证四个真实电机已经停稳。ESP32 收到新方向后也不能仅依据内部 current 已归零就认为轮子停止。

若日志证明命令链正确但仍存在“部分轮正转、部分轮反转”，下位机应增加基于实际速度的换向过零联锁：

1. 检测新目标是否与当前实际轮速存在符号反转。
2. 立即向四轮发送统一零速，进入 `REVERSAL_WAIT`，禁止任何轮提前反转。
3. 根据连续 `$MSPD` 帧判断四轮均进入零速阈值，并稳定若干帧。
4. 四轮确认停稳后，从零开始执行最新方向；等待期间只保留最新目标。
5. 等待超时、遥测丢失、BLE 断开或急停时保持停车并进入安全状态，不得强制反转。

在 `$MSPD` 字段、单位、方向和刷新周期尚未通过实测确认前，不应直接实现闭环换向。临时固定延时只能用于悬空诊断，不能作为最终安全方案。

### 4.5 参数调整顺序

1. 首轮保持 `40 ms / 步长 6 / 500 ms` 不变，先采集证据。
2. 根据逐轮测试确定四轮共同的最低可靠速度；低于该值的命令可能让部分轮起转、部分轮静止。
3. 先修正编码器方向、接线、AT8236 参数和机械问题，再做每轮稳态 trim。
4. 若稳态一致但起步仍不同，再评估每轮死区补偿、前馈或 AT8236 PID；任何启动补偿都必须限幅并经过悬空测试。
5. 只有新 target 已及时到达而 current 响应确实过慢时，才单独调整斜坡。增大步长或缩短 40 ms 会增加机械冲击。
6. 上下位机链路稳定后，运动保鲜可从 `500 ms` 分阶段试到 `400 ms`、再试 `300 ms`；每次都要验证按住不误停、卡顿不误停和断连停车。

### 4.6 下位机责任边界

- 下位机负责命令接收证据、四轮映射、驱动输出、实际轮速、故障停车、换向过零和轮级标定。
- 下位机不负责 Android 触摸语义、BLE GATT 写入排序或旧 generation 淘汰。
- 不允许通过给每条命令增加固定动作持续时间、删除斜坡、删除 watchdog 或盲目缩短 500 ms 来掩盖问题。

### 4.7 下位机交付物

- Android 命令与 ESP32 `motion_rx/target/current/$spd/$MSPD` 的对齐日志。
- M1-M4 的最低起转、停止、正反转过零和稳态速度测试表。
- 电机/通道交叉替换结论。
- 是否需要换向过零联锁、每轮补偿或 AT8236 参数调整的证据结论。

## 5. 联调与验收顺序

### 阶段 A：版本确认

1. Android 使用 `106bf5f` 或更新版本，并显示构建提交。
2. ESP32 使用包含 `09a4e88` 或更新内容的固件。
3. 自动跟随保持关闭，测试人员可随时物理断电。

### 阶段 B：只验证上位机和 BLE

不接地面负载，完成全部触摸手势，证明新 generation 后不会再写出旧方向命令。

### 阶段 C：验证 ESP32 与 AT8236

开启 USB 诊断，对齐 Android、`motion_rx`、target/current、`$spd` 和 `$MSPD`；逐轮完成起转、停止和正反转测试。

### 阶段 D：悬空换向验收

依次验证前进、停车、后退、原地左右转、旧手指迟放、BLE 断开和 500 ms 运动保鲜停车。任何时刻不得出现已知条件下的异向轮输出。

### 阶段 E：低速落地

只有阶段 B-D 全部通过后才空载落地。速度必须采用逐轮测试确认的共同可靠区间；旁人保留物理断电，手动模式通过前不得恢复自动跟随。

## 6. 结论

任务边界应按证据链切分：上位机证明“最新方向按正确顺序成功写出”，下位机证明“最新方向被正确接收、映射、输出，并由四轮实际执行”。

BLE 命令排序修复和 ESP32 USB 诊断已经进入最新代码，但真机问题尚未验收。四轮不同步不能用 Android generation 解决，旧命令也不能靠调电机 trim 解决。两边必须用同一次换向事件的时间戳和数值对齐后，再分别处理各自层级的问题。
