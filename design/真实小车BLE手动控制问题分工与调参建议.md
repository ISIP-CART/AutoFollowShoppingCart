# 真实小车 BLE 手动控制问题分工与调参建议

## 1. 文档目的

本文用于拆分真实小车 BLE 手动控制联调中的上位机与下位机职责，处理以下问题：

1. 按住一个方向按钮后切换到另一个按钮，新方向没有正确执行，小车继续先前动作。
2. 当前直线和原地转向速度偏快。
3. 澄清 Android、ESP32 与 AT8236 控制链路中的时间参数，避免把安全超时误当成运动响应参数。

本轮先完成诊断与职责划分，不在同一次修改中同时调整 Android 和 ESP32，以免无法判断问题来源。

## 3. 当前控制链路

```text
Android 方向按钮
  -> RealCartFollowFragment
  -> Vehicle.setControl()
  -> BLE 文本命令 c<left>,<right>\n
  -> ESP32 解析并设置四轮目标值
  -> 每 40 ms 进行一次速度斜坡更新
  -> AT8236 命令 $spd:m1,m2,m3,m4#
  -> 电机持续执行最近一次速度目标，直到收到新速度或停车命令
```

### 3.1 命令是否带时间

当前两级运动命令都**不带动作持续时间**：

- 手机发给 ESP32：`c<left>,<right>\n`，只有左右轮目标值。
- ESP32 发给 AT8236：`$spd:m1,m2,m3,m4#`，只有四轮速度值。

AT8236 不会因为某条速度命令“到时”自动结束，而是保持最近目标。停车依靠以下机制：

- Android 松手时主动发送 `c0,0`；
- Android 按住时每 100 ms 重发当前非零 `c` 命令；
- ESP32 超过 500 ms 没有收到新的非零运动命令时停车；
- BLE 断开、通信超时、驱动异常或急停时停车。

### 3.2 哪些时间参数影响什么

| 参数 | 当前值 | 实际作用 | 是否建议为本问题直接缩短 |
|---|---:|---|---|
| Android 命令重发周期 | 100 ms | 按住期间刷新当前目标并满足运动保鲜 | 暂不调整，先确认切换命令是否成功发送 |
| ESP32 运动保鲜超时 | 500 ms | Android 卡死或不再产生非零命令时自动停车 | 可在稳定后试验 300-400 ms，但不能解决正常换向 |
| ESP32 通信保鲜超时 | `h750` | 判断控制连接是否仍存活 | 保持 750 ms |
| ESP32 控制更新周期 | 40 ms | 向目标速度推进并发送 AT8236 速度帧 | 属于下位机动态响应参数 |
| ESP32 每周期斜坡步长 | 6 | 决定非零速度变化快慢 | 属于下位机动态响应参数 |

缩短运动保鲜超时可以减少“最后一条命令丢失后继续运动”的最长时间，但不能让一条已经正确收到的新方向命令更快生效。设置过短还可能因 BLE 抖动产生误停车。

## 4. 为什么现有修复仍可能没有解决真机问题

`ManualControlArbiter` 已解决逻辑层的一类竞态：新按钮取得控制权后，旧按钮迟到的 `UP/CANCEL` 不再覆盖新命令。现象仍存在时，至少还要排查以下边界。

### 4.1 手机可能仍安装旧 APK

主仓库此前没有记录新的子模块指针。应在应用中增加可见的构建版本，或通过 Android 日志确认运行代码包含 `manual_replace`，不能只根据源码推断。

### 4.2 BLE 连续写入没有应用级串行队列

切换按钮时，当前 Android 实现会紧接着调用：

```text
c0,0
c<new_left>,<new_right>
```

底层 BLE 写入是异步操作。当前 `BluetoothManager.write()` 直接发起写请求，只记录成功或失败回调，没有显式的单写入队列、命令序号或确认后再发机制。需要确认设备和 BLE 库在前一次写完成前收到第二次调用时，是排队、拒绝还是覆盖。

此外，100 ms 调度器会继续重发 `latestOutput`。如果日志显示 UI 已切换但 BLE 成功回调仍是旧命令，问题在 Android 状态或写入链；如果 BLE 已成功发出新命令而小车仍执行旧动作，则继续查 ESP32/AT8236。

### 4.3 多点触控事件可能与单指顺序不同

需要分别复现并记录：

1. 松开旧按钮后再按新按钮；
2. 旧按钮仍按住时，用第二根手指按新按钮；
3. 手指从一个按钮滑到另一个按钮。

目前监听器主要处理 `ACTION_DOWN/UP/CANCEL/OUTSIDE`，没有围绕 pointer ID 建立完整多点触控模型。三种操作不能被视为同一个测试用例。首版安全交互可以明确只支持“单指、新按钮抢占”，或者补充经过验证的多点触控策略。

### 4.4 ESP32 当前没有上报每条运动命令的序号

ESP32 会解析 `c` 并立即更新目标，但普通运动命令没有序号和接收回执内容区分。`!OK,c` 只能证明收到过一条合法命令，不能证明具体是哪一个左右轮值。仅看小车动作难以区分“手机没发”“BLE 丢写”“ESP32 没解析”或“AT8236 没执行”。

## 5. 上位机同学负责的修改

### 5.1 第一优先级：建立可验证的命令链

- 在真实小车页面显示或记录 APK 构建提交，例如 `OpenBot 6ce5e00+`。
- 为每次手动事件记录：事件时间、按钮、`actionMasked`、pointer ID、仲裁 generation、目标左右值。
- 为每次 BLE 写入记录：待发送命令、写入开始、成功/失败回调和耗时。
- 不在默认状态保存相机画面；控制日志仍应由诊断开关控制。
- 增加 BLE 单写入队列，确保同一时刻只有一条 GATT 写操作在途。
- 将安全停车设为高优先级；方向切换命令不得被旧的周期重发插队。
- 对方向切换建立可测试顺序：旧方向停止确认或写完成后，再发送新方向；随后周期重发新方向。

建议不要依赖“连续调用两次 `write()` 大概率会按顺序送达”。

### 5.2 第二优先级：完善触摸抢占

- 明确支持的操作：至少保证“松开 A 后按 B”和“按住 A 时按 B”都以 B 为当前方向。
- 对每个按钮记录 pointer ID，避免旧 pointer 的释放影响新 pointer。
- `ACTION_CANCEL`、窗口失焦、页面暂停、模式切换和 BLE 断开必须立即停车。
- 增加 Fragment/Robolectric 或可替代的事件序列测试，不只测试纯 Java 仲裁器。
- 在页面上显示当前有效手动方向和最近成功发送的 `c` 命令，便于真机判断。

### 5.3 速度调整

手动模式建议先按用户反馈直接降低为：

| 动作 | 当前协议值 | 建议首轮值 |
|---|---:|---:|
| 前进 | 28 | 14 |
| 后退 | 24 | 12 |
| 原地左/右转 | 20 | 5 |

这是直线约减半、转向约降至四分之一的首轮标定值。调整位置应在 `RealCartSafetyController` 的手动速度常量，避免依赖全局 `speedMultiplier` 影响其他 OpenBot 页面。

自动模式的 `AUTO_MAX=32` 和 `SEARCH_SPEED=18` 也明显高于新的手动实验速度。自动跟随恢复真机测试前，应单独重新标定，建议初始不超过手动前进值，并将原地搜索速度限制到手动转向值附近。不要因本次手动调速顺手宣布自动模式安全可用。

### 5.4 上位机验收输出

上位机同学应提供一段不接地面电机负载或车轮悬空的日志，至少覆盖：

```text
FORWARD down -> c14,14
LEFT down -> c0,0 -> c-5,5
FORWARD late up/cancel -> ignored
LEFT periodic repeat -> c-5,5
LEFT up -> c0,0
```

每条命令必须能关联到 BLE 写成功或失败回调。

## 6. 下位机同学负责的修改

### 6.1 第一优先级：增加可观测性

- 在 USB 调试输出中记录每条合法 `c` 的接收时间、来源和解析后的 `left/right`。
- 记录 `setTankTargets()` 得到的四轮 target，以及控制循环实际发送的四轮 current。
- 为普通运动命令增加调试序号或扩展查询输出，使一次切换能从 Android 日志对应到 ESP32 日志。
- 保留 `!Q`，并建议增加“最近运动命令值、年龄和计数器”。
- 调试日志只走 USB，不要让高频 BLE Notify 反过来影响控制链路。

### 6.2 核对方向映射和执行链

- 用车轮悬空测试确认 `c14,14`、`c-5,5`、`c5,-5`、`c-12,-12` 的四轮方向符合机械安装。
- 确认收到新 `c` 后 target 在同一个主循环周期内更新。
- 确认 AT8236 接收 `$spd` 后不会因其内部模式、缓存或其他控制源继续采用旧目标。
- 保持 BLE 与 USB 控制权互斥，避免调试串口命令抢占手机控制。

### 6.3 时间参数建议

- 暂时保留 `CONTROL_UPDATE_MS=40`、`CONTROL_RAMP_STEP=6`，先用日志确认新命令是否到达。
- 暂时保留 `MOTION_REFRESH_TIMEOUT_MS=500`，它是失效安全参数，不是正常换向参数。
- 上位机连续写和命令顺序确认稳定后，可把运动保鲜超时分阶段试为 `400 ms`、再试 `300 ms`。
- 每次缩短后必须验证：按住不误停、手机短暂卡顿不误停、松手和断连停车时间满足预期。
- 如果新方向 target 已及时更新但 current 变化仍显得迟钝，再单独讨论斜坡：增大步长或缩短 40 ms 会提高响应，但也会增加机械冲击，不能和降速修改混为一项。

### 6.4 不建议的改法

- 不建议给每条普通速度命令增加固定“动作持续时间”，然后让 AT8236 开环执行一段时间。按住控制更适合目标值加 watchdog 模型。
- 不建议为了灵敏度直接删除斜坡和超时保护。
- 不建议仅把 500 ms 改得很短，却不确认 BLE 重发周期和手机调度抖动。
- 不建议在没有序号或日志证据时同时修改方向映射、斜坡、超时和 Android 触摸逻辑。

## 7. 联调顺序与责任边界

### 阶段 A：确认版本

1. 主仓库提交新的 OpenBot 子模块指针。
2. 上位机同学从 `6ce5e00` 或更新提交构建 APK。
3. 手机上的页面显示构建提交或版本号。

### 阶段 B：只看 Android 和 BLE

1. 不接电机或让车轮悬空。
2. 完成三种按钮切换手势测试。
3. 确认事件、仲裁、待发送命令和 BLE 回调一致。
4. 若新命令没有成功写出，由上位机同学修复。

### 阶段 C：看 ESP32 与 AT8236

1. Android 已证明新命令成功写出后，检查 ESP32 是否按顺序收到。
2. ESP32 收到后检查 target/current 和 `$spd` 输出。
3. 如果 ESP32 输出正确而电机仍保持旧动作，再检查 AT8236 和电机方向配置。

### 阶段 D：低速落地测试

1. 使用 `14/12/5` 首轮速度值。
2. 旁人准备物理断电，场地保持空旷。
3. 依次验证松手、换向、BLE 断开和 500 ms 运动保鲜停车。
4. 手动模式全部通过后，才重新标定自动跟随速度。

## 8. 本轮结论

按钮抢占的纯逻辑修复已经进入 OpenBot 最新主分支，但当前真机现象说明验证还停留在纯 Java 状态机层，尚未覆盖 Android 触摸事件、异步 BLE 写入和 ESP32/AT8236 执行的完整链路。

下一步不应先凭感觉缩短所有时间参数。应由上位机同学先证明“新按钮事件产生了正确且成功的 BLE 命令”，再由下位机同学证明“ESP32 收到并向 AT8236 输出了对应的新速度”。速度降低主要属于上位机动作限幅；运动保鲜、斜坡和电机帧诊断属于下位机。两边通过同一条带时间戳的命令证据对齐后，才能准确定位剩余问题。
